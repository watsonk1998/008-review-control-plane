# 解决审查报告网页端滚动卡顿及排查审查规则串区问题的执行计划

## 1. 现状与根因分析

### 问题一：网页端滑动卡顿、甚至白屏
通过排查代码与 `AGENTS.md` 的严格约定，该问题主要来源于**历史持久化脏数据中的恶意 CSS 污染**：
1. 后端此前在生成新报告时虽然已经修复了导致 `O(n)` 强制重排和渲染卡死的属性（如 `break-inside: avoid`、`content-visibility: auto` 以及 `transition: box-shadow`）。
2. 但**历史已经生成的审查结果**，其快照（`reportPrintCss` 字段）已被持久化。当前前端 `task-detail.tsx` 的清洗代码仅仅用正则过滤了 `break-inside` 相关属性，**漏掉了引发白屏和滚动卡顿的核心元凶**：`content-visibility: auto` 和 `transition: box-shadow`。导致历史任务在查看时仍会将这些毒性 CSS 注入到页面中，造成极度卡顿。
3. 同时发现 `apps/web/src/app/theme.css` 中关于报告容器 `.structured-report-host` 的 CSS，违反了 `AGENTS.md (HG-28)` 的约定，使用了 shorthand `overscroll-behavior: contain`，而规范明确要求必须是 `overscroll-behavior-x: contain`。

### 问题二：前端选择方案类型后，审查规则是否会串？
**结论是：在新版架构下绝对不会串。** 针对您的疑虑，系统已经在架构层面做了双重硬隔离：
1. **Agent 隔离（硬性门禁 Hard Gate）**：
   在之前的版本中，如果某个模块被启用（explicitly_enabled 得分 100 分），确实可能软覆盖文档类型的限制。但当前版本的 `HermesTemplateRegistry` 已将 `supported_document_types` 设置为**硬性门禁**。若用户前端选择“施工组织设计”，那么专属于“配网停电”的审查子 Agent（即便属于被开启的模块）也会直接被 `continue` 跳过，物理隔绝了跨类型的 Agent 执行。
2. **依据与规则包隔离（严格映射）**：
   审查规则并非随机组合，而是通过 `BasisPackResolver` 严格遵循 `profile_mapping.yaml` 获取。例如，选择“施工组织设计”，后端只会挂载专属的 `construction_org.core.v1` 规则包，根本不会吸入带有停电字眼的特殊依据。同时，前端（`structured-review-form.tsx`）在切换方案大类时，也会**自动清空**不匹配的专项标签（`disciplineTags`），截断了上游污染。
*您感觉“串区”，极大概率是因为查阅了修复前的历史审查任务，或者是误将“所有类型通用”的审查 Agent（如“文档完整性审查”）视为了特殊规则的越界。*

## 2. 修复计划与执行步骤

**步骤一：前端历史 CSS 深度防御清洗扩充（解决卡顿与白屏）**
- 修改文件：`apps/web/src/components/task-detail.tsx`
- 动作：在 `StructuredReportHtml` 组件的 `safeCss` 过滤逻辑中，追加对 `content-visibility`、`contain-intrinsic-size` 以及 `transition` (特指包含 `box-shadow`) 的全局正则表达式替换（置空）。确保无论历史数据中埋藏了多少有毒的样式，渲染前都会被彻底清理。

**步骤二：对齐 `HG-28` 滚动防穿透规范约束**
- 修改文件：`apps/web/src/app/theme.css`
- 动作：将 `.structured-report-host` 中的 `overscroll-behavior: contain;` 精确修正为 `overscroll-behavior-x: contain;`，防止对 Y 轴正常惯性滚动的误伤，严格遵守合同约定。

**步骤三：向用户作结论汇报与系统澄清**
- 动作：将上述“双重硬隔离”机制（文档类型 Hard Gate 与 profile 强映射）结论性地同步给用户，澄清疑虑，宣告修复完成。
