# 网页端审查报告滚动卡顿问题（HG-25）调查与修复计划

## 1. 问题根因分析

根据 AGENTS.md 中的 `HG-25` 规则，长列表卡片如果全局使用了 `break-inside: avoid` 和 `page-break-inside: avoid`，会在滚动时引发 `O(n)` 的强制重排，从而导致浏览器严重卡顿甚至白屏。

在后端的 `apps/api/src/review/report/final_report_view_model.py` 中，虽然近期的提交（`6cb1dff72ae62226b`）已经将全局范围的 `break-inside: avoid` 移除，并仅保留在了 `@media print { ... }` 块中，**但这只能解决“新生成的审查任务”的卡顿问题**。

**导致当前前端仍然卡顿的原因是：**
对于在修复提交**之前**生成的历史审查任务，其结果已经被固化并保存在了 SQLite 数据库或 `report.json` 制品中。这些历史制品中的 `reportPrintCss` 字段依然包含了全局的 `break-inside: avoid`。
当用户在前端浏览器查看这些历史任务时，`apps/web/src/components/task-detail.tsx` 中的 `StructuredReportHtml` 组件会直接注入旧的 CSS：
```tsx
{printCss ? <style>{printCss}</style> : null}
```
这导致带有破坏性重排属性的旧 CSS 依然被作为全局样式挂载，触发了浏览器的滚动性能问题。

## 2. 修复方案

由于 `printCss` 是一段纯文本，且历史数据已不可变，我们需要在前端进行防御性清洗（Sanitization），确保任何被注入的 CSS 都不包含会导致屏幕渲染卡死的全局 `break-inside: avoid` 属性。

### 具体修改

在 `apps/web/src/components/task-detail.tsx` 文件中修改 `StructuredReportHtml` 组件：

```tsx
function StructuredReportHtml({ htmlContent, printCss }: { htmlContent: string; printCss?: string }) {
  // 防御性剔除可能导致屏幕滚动卡死的强制重排属性（应对历史任务遗留的旧版全局 CSS）
  const safeCss = printCss 
    ? printCss
        .replace(/break-inside:\s*avoid;?/gi, "")
        .replace(/page-break-inside:\s*avoid;?/gi, "")
    : "";

  return (
    <div className="structured-report-host">
      {safeCss ? <style>{safeCss}</style> : null}
      <div dangerouslySetInnerHTML={{ __html: htmlContent }} />
    </div>
  );
}
```

### 为什么采用全局替换？
虽然这也会顺带移除了历史任务在 `@media print` 块中的分页控制，但：
1. 前端页面的首要职责是保证屏幕阅读的流畅性（避免白屏卡死）。
2. 用户如果需要完美的打印排版，系统已经提供了“导出正式排版 PDF”按钮（由后端 WeasyPrint 基于原始不可变的 `report.json` 渲染，后端渲染不受前端这段正则影响，分页依然完美）。
