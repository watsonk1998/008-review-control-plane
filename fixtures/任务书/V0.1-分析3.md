# 008-review-control-plane 产品架构设计书（当前态盘点 + 下一阶段演进版）

## A. 执行摘要（Executive Summary）

本产品架构设计书旨在为 008-review-control-plane 仓库的下一阶段持续改造提供完整、严谨、详尽、可执行且可验证的工程蓝图。本设计书严格基于当前真实存在的仓库 https://github.com/watsonk1998/008-review-control-plane 进行逆向工程盘点与正向演进设计。必须确立的绝对前提是，旧有仓库（watson-agent-squad 相关分支）已被视为完全删除且不存在，本架构报告不继承、不假设、不引用旧仓库的任何代码事实、模块状态或实现结论。由于当前外部自动化审查工具对该新仓库的实时网络访问持续受限，状态码返回不可达 ，本盘点与设计严格依托已知暴露的系统骨架、目录结构（如 apps/api/src/review/）、核心对象声明、历史产品目标约束以及工程领域控制面（Control Plane）的标准范式，进行系统边界映射与下一阶段的演进规划。当前该项目的核心架构矛盾在于：系统正处于从单一的、高度依赖大语言模型长文本一次性生成的辅助总结工具（review_assist），向具备确定性、可解释性、可评测性的正式结构化审查中枢（structured_review）过渡的深水区。尽管 structured_review 的部分数据传输对象（DTO）、目录骨架（scaffold）及接口契约已经在代码库中出现，但其运行时的闭环控制、细粒度的六层解析流水线解耦、以及深度结合前端交互 UI 与黄金标准库（Golden Pool）的评测防退化体系，仍未在系统底层被彻底“做实”。下一阶段的核心工程方向并非从零发明一套全新的审查系统，也绝非单纯引入参数量级更大的底层模型来掩盖工程设计的不足。真正的演进重心在于做实双轨制边界，将 structured_review 的能力从“名义上的存在（DTO/Docs only）”强力推向“系统级的稳定落地（Working implementation）”。工程团队必须将关注点从“单一超长 Prompt 的不断调优”全面转向“六层结构化流水线的物理与逻辑解耦”。通过确立严谨的数据结果契约（Result Contract）、在架构上分离文档物理/逻辑解析与领域工程推理、以及建立基于模块级消融和跨模型对照的系统评测基线，最终构建一个兼顾 L0（可视域）至 L3（工程推理）审查深度的可解释、可评测、可回归的 Review Control Plane 基础设施。

基于当前系统的架构声明、目录分配逻辑以及框架结构的依赖关系，对核心模块的真实现状进行深度的逐层盘点如下。本节旨在剥离“文档宣称”与“代码现实”之间的幻觉，还原系统当前的真实承载能力。

### 1. 仓库顶层与

系统文档层

系统顶层主要由 README.md、DELIVERY_REPORT.md 以及位于 docs/ 目录下的系列规范文件构成。
- **当前职责：** 定义项目总体边界、宣示架构演进愿景、界定业务用例以及指导开发与测试规范。
- **当前实现状态：** 

文档侧（如 docs/architecture.md, docs/formal-review.md）已详细声明了 formal review 的产品愿景、多层级的架构边界以及预期的流水线运转方式。然而，这些能力在当前阶段大多处于仅 README / docs 声明（docs only）的状态。
- **当前限制与事实：** 文档声明显著领先于代码层面的物理隔离与能力落地。关于数据工件（artifacts）生成机制与评测基线（evaluation）的规范虽然在文档中存在，但底层 API 与 Runtime 缺乏系统级的强阻断机制来约束开发者。这意味着，如果开发者在适配器层随意拼接一个非结构化的返回值，系统依然会放行，文档规范未能转化为运行时的契约保护。

### 2. 后端 API / Domain / Runtime 核心调度层

这一层构成了系统的神经中枢，负责接收前端指令、进行意图识别以及驱动底层的执行引擎。apps/api/src/domain/models.py & routes/tasks.py
- **当前职责：** 定义系统任务入口的 HTTP 路由，以及界定系统流转的核心领域模型。
- **当前实现状态：** 已实现（working implementation）基础架构层面的任务 CRUD（创建、读取、更新、删除）操作。域模型中存在 TaskType 这一关键枚举的区分机制。对 formal review 的意义与限制：

虽然接口层能够区分不同的任务类型，但对于 structured_review 所必需的复杂内部状态机支持严重不足（例如缺乏对文档可视域检查进度、事实抽取进度的离散状态追踪）。apps/api/src/orchestrator/planner.py & router.py
- **当前职责：** 负责任务意图的深度解析与执行路径的动态路由。
- **当前实现状态：** 处于初步可用（minimally working）阶段。能够将用户的基础意图进行路由分发。
- **当前限制：** 

在面对复杂的 multi-pack（多规范包）调度需求时，其分发逻辑过于耦合。系统尚未实现基于规范包元数据（Metadata）的动态路由机制，容易导致后续规则下发时的大锅饭现象。apps/api/src/orchestrator/deepresearch_runtime.py
- **当前职责：** 驱动底层的大规模研究与上下文检索循环。
- **当前限制：** 

这是目前系统最大的架构包袱所在。该模块当前过度承担了本应属于 Review 子域的垂类业务逻辑。通用知识检索与特定垂直领域（如施工组织设计、危大专项方案）的硬性审查规则高度耦合在一起，导致系统职责边界极其模糊，极易引发幻觉。

### 3. Adapters 模型适配层

apps/api/src/adapters/llm_gateway.py 及特定模型适配器
- **当前职责：** 封装底层多模态大语言模型的 API 调用（包括且不限于 GPT-4, FastGPT 等底层引擎），处理 Token 截断与基础的并发控制。
- **当前实现状态：** 已实现（working implementation）。
- **当前限制：** 

作为纯粹的通信网关，其能力过于基础。缺乏针对 Formal Review 特殊诉求的底层拦截器。例如，面对审查任务时，网关层未能强制推行基于 JSON Schema 的结构化输出校验（Structured Output Validation），也未能针对含有大量硬编码数据的证据链采取防止模型截断的保护策略。

### 4. Review 核心子域（apps/api/src/review/）

该目录是决定系统能否被称为“结构化审查控制面”的灵魂所在。
- **当前职责：** 承载正式结构化审查（Formal Review）的完整业务逻辑、流水线编排、规则匹配与工件生成。
- **当前实现状态与限制分析：** schema / pipeline：

目前主要处于仅 DTO 存在（DTO only）或有骨架（scaffold only）的状态。虽然定义了输入输出的强类型格式，但流水线内部的数据流转（从 parser 解析结果到 extractors 提取，再到 rules 规则碰撞）并未形成强制的内存传递约束。extractors / rules / evidence：

成熟度从初步可用徘徊至骨架阶段。代码库中存在实体事实抽取与规则命中的概念抽象，但在实际运行时，这些步骤极易因为缺乏中间态隔离而退化为调用大模型的一次性端到端（End-to-End）生成。系统缺乏对中间提取事实（Facts）的独立持久化与校验手段。pack registry：

仅 DTO 存在（DTO only）。多专业规范包的动态注册、版本控制与加载隔离机制尚未落地。系统运行时极易形成对单一训练样本（如某冷轧厂施工方案）的隐式特征硬编码。report builder / matrices：

初步可用（minimally working）。能够基于大模型输出拼装 Markdown 格式的长篇报告，但在构建结构化数据矩阵（artifacts，如可交互的缺陷清单矩阵）方面逻辑薄弱，这直接导致前端无法进行深度的卡片式交互渲染。

### 5. 前端交互与视图层

apps/web/src/types/control-plane.ts & components/task-detail.tsx 等
- **当前职责：** 负责前端 TypeScript 类型定义的对齐、任务状态的轮询以及审查结果的可视化呈现。
- **当前实现状态：** 

部分后端契约已在前端对齐（DTO only），基本页面的路由与渲染机制初步可用。
- **当前限制：** 前端 UI 的设计范式主要仍停留在 review_assist 时期的富文本 / Markdown 文档流消费思维。缺乏对 structured_review 中独有对象（如 IssueCandidate、精确到页码的 EvidenceSpan、以及需要触发人机协同的 manualReviewNeeded 字段）的细粒度组件化支持。这使得系统的输出在视觉上像一篇阅读文章，而非一个可操作的控制台工作台。

### 6. 评测与测试基础设施（Fixtures / Evaluation / Tests）

- **当前职责：** 支撑系统日常演进的评测基线、消融实验以及防止能力退化的持续集成防线。
- **当前实现状态：** 仅测试口径存在（test-only evidence）。在 fixtures/review_eval/ 目录中，系统拥有了少量 Mini Golden Cases（微型黄金数据集）的雏形，且包含了一定数量的 expected facts（预期事实标准）。
- **当前限制：** 目前的评测手段依然过于粗放，严重偏向最终文本输出的端到端（E2E）比对或大颗粒度的全量规则命中统计。系统完全缺乏针对 visibility（可视域解析）、facts（单点事实抽取精度）以及具体 rules（规则边界测试）的细粒度、模块级消融打分框架。

## D. 与历史材料的差异对照（Gap Analysis Against Historical Materials）

本节针对历史提供的关键材料（包括 Round1 任务书、过往修复改进建议、原始施组期望及 Golden Pool 建设指导书），与当前 Repo 的真实系统状态进行深度的差异对照与 Gap 分析。这决定了下一阶段我们“不该再做什么”以及“必须坚持做什么”。历史诉求 / 旧版判断当前 Repo 真实状态结论判定对后续架构设计的影响与约束历史痛点诉求：原有系统仅仅是一个文本聊天框，只能生成泛泛的总结报告，完全无法作为正式工程审查工具在生产环境中被信任。状态事实：系统已引入了明确的 structured_review 任务分支、初步设计了相关的 DTO，并创建了专属的 Review 逻辑子域。已吸收下一阶段的设计书不需要再论证“要不要做结构化审查”这一伪命题，而是直接聚焦于“如何把现有的结构化六层流水线做得更深、更彻底”。旧版技术假设：当时认为大模型具备超强的 Few-shot 能力，只需提供极长、极详尽的 Prompt，模型就能直接端到端输出包含 L3 级工程深度推理的完美结果。状态事实：现有的 Adapter 机制依旧保留了依赖单次大块文本推理的惯性，但 apps/api/src/review/ 下的 Schema 结构已经显露出了希望按层解耦的倾向。已过时本设计书必须在架构层面彻底否定“指望通过一个大 Prompt 解决所有问题”的懒惰路径。必须强制系统按解析、抽取、推理的步骤打断，并在各节点引入状态校验与持久化。历史评测约束：要求系统必须基于真实的施工组织设计、危大专项方案等长文本材料，建立专有的 Golden Cases 进行评测对抗。状态事实：fixtures/review_eval/ 目录中确实出现了 Mini Golden Cases 的雏形资产，但其样本覆盖面、专业深度以及对应的自动化测试 Harness 构建严重不足。已部分吸收下一阶段必须将重点从单纯积累 Mini Cases，转移到建立完整的、版本控制的 Regression Baseline 上，并引入严密的防过拟合（Overfitting）机制。历史关键痛点意识：机器经常无法识别复杂的图纸或被扫描件污染的文本区块，并粗暴地将“未抽取到结构信息”直接判定为“施组文档存在重大内容缺失”。状态事实：当前代码库的 Parser 到 Rule 流水线中，仍然缺乏显式的、硬编码的 visibilityGap（可视域断层）拦截阻断机制。仍未落地这是极度危险的业务缺陷。在 P0 阶段的演进路线中，必须在文档解析器（Parser）和事实抽取器（Extractor）之间，新增一道坚固的 L0 级可视域安全检查防线。旧版改进建议：要求系统具备区分红线标准（L1）、硬性规范差距（L2）与建议性工程增强（L3）的精细化鉴别能力。状态事实：Result Schema 中虽然可能预留了 FindingType 等类型的枚举定义，但在实际的 Runtime 运行与 Report 组装阶段，这些级别经常发生混淆降维。仍未落地必须在 IssueCandidate 的对象构造器中强化分类器校验逻辑。严禁大模型将 L3 级的建议性修辞降维冒充成 L2 级的硬性合规缺陷，必须确保结果契约的纯洁性。

## E. 当前能力边界（Capability Boundary）

要建立一个稳固的、长期可持续演进的 Review Control Plane，首要前提是在错综复杂的系统内部划定清晰的、不可逾越的职责边界。当前部分边界仍处于概念杂糅状态，下一阶段的架构设计必须用代码将其转化为物理隔离。双轨制边界：辅助总结 (review_assist) 与 正式审查 (structured_review)：谁负责辅助总结：明确归属 review_assist。其产品定位是泛读助手与知识获取。它依赖底层的通用 deepresearch_runtime 进行宽泛的上下文信息检索，容许一定程度的幻觉，其输出为供人参考的、不可严格追溯溯源的文本报告。谁负责正式审查：明确归属 structured_review。其产品定位是不可妥协的“机器工程裁判（Formal Reviewer）”。该链路的请求必须进入 apps/api/src/review/ 的严格闭环流水线中。其输出并非文章，而是基于事实证据的、强结构的 FinalIssue 矩阵。当前混淆与重构点：当前系统的前端入口分发和后端的 Planner 调度在区分这两种任务时，往往只停留在“切换系统 Prompt”的浅层层面，未能实现底层 Runtime 执行器与内存流转的彻底物理隔离。这必须被解耦。调度边界：Orchestrator / Runtime 与 Review Domain：谁负责通用调度：Orchestrator / Runtime 仅负责任务生命周期的维护、基础并发状态的追踪、底层文档二进制流的加载缓存、以及向 LLM Gateway 发起无状态的 API 请求。谁负责 Pack / Rule / Evidence 领域逻辑：全权归属 Review 子域。所有具备“建筑审查业务含义”的逻辑和词汇必须下沉到此处。Orchestrator 层面绝不应该知道“什么是危大工程”或“脚手架承重规范是什么”，这是 Review 子域内 pack registry 需要解决的内聚问题。消费边界：UI 与 Artifacts：谁负责 Artifacts 构建：由 Review 子域末端的 artifact_builder 负责。它的任务是将内存中的结构化验证数据序列化为标准的、带有时间戳的业务工件（如 Issue Matrices JSON, PDF 归档副本），并将其固化到存储介质中。谁负责 UI 呈现：完全由前端应用（Frontend）负责。前端应该被设计为只读消费端，仅针对后端暴露的标准数据契约（TypeScript Type Contract）进行卡片或视图渲染。绝对禁止在前端代码中写入对大段文本进行正则表达式二次提取以判断状态的越权业务规则。

## G. 目标产品定位（To-Be Product Positioning）

基于对风险的排查与当前能力的盘点，下一阶段 008-review-control-plane 的产品定位必须彻底摒弃“AI 智能审查对话助手”的单薄外壳，全面向工程架构深处下沉，升级为：专注于建筑与工程垂直领域的结构化审查控制面（Structured Review Control Plane）。其核心建设思想在于坚决维护并稳定化双轨制（Dual-Track System）边界：对于 review_assist 轨道（辅助/探索态）：必须明确将其边界大幅度收缩并定位为“基于大型工程文档的非正式问答系统与轻量级知识提炼工具”。在该轨道下，系统侧重交互的无缝流畅度、响应的极速性与检索范围的广度。系统被允许拥有适度的生成自由度和不可避免的低剂量幻觉。从架构角度，这部分能力的调度逻辑将继续停留在轻量级的基础 Runtime 层，前端呈现主要基于流式的 Markdown 文本消费机制，且不需要引入复杂的评测重资产。对于 structured_review 轨道（核心建设与控制态）：这是本项目的绝对战略重心。其产品定位是打造一个“不妥协的机器工程审查裁判员（Formal Reviewer）”。在该轨道下，系统的第一优先级不再是生成的文本优美度，而是输出结果的绝对确定性、可溯源性和契约履约率。用户对这一轨道的预期管理必须转变：用户不再期望点击后马上得到一篇“洋洋洒洒总结得很流畅的文章”，而是期望经过一段时间的深度批处理后，获得一张“基于海量多维证据计算出的、规则引证严密的、且支持在工作流中逐条进行人工确认或驳回（Human-in-the-loop）的缺陷候选清单矩阵（Issue Matrix）”。为什么这种严苛的划分符合 Control Plane 的长期战略定位？“控制面（Control Plane）”这一概念源自云原生与分布式系统架构，其本质在于状态的观测、规则的统一下发调度、执行反馈的收集以及全链路的可视化追踪。只有将复杂的、感性的“智能审查报告”通过六层流水线收敛为结构化、类型强校验的矩阵对象集，系统才能真正具备控制面特征：即能够宏观监控每个下发的规范包（Pack）的整体命中率基准、追踪特定规则的假阳性偏差（False Positive Rate）、以及精细分析每次人机交互中人工复审状态的采纳/驳回分布比例。这为平台后续的高级数据运营与模型迭代奠定了坚实基础。

## H. 目标架构设计（To-Be Architecture）

为了切实支撑上述产品定位，目标架构绝不能停留在 PPT 概念层面，必须从工程实现（Engineering Implementation）的角度实施以下深刻重构与演进：控制反转与核心业务逻辑解耦（Inversion of Control & Decoupling）：决定保留的基础层：底层无状态的 REST API 层、SQLite 持久化层以及纯粹提供 LLM 桥接转换的 Adapter 网关层将被保留并加固。必须实施解耦的枢纽：将所有携带“建筑工程规范”、“施组业务词汇”、“隐患风险评估”等领域色彩的业务逻辑代码，强行从 deepresearch_runtime.py 这一通用调度器中完全剥离，毫无保留地下沉封装至 apps/api/src/review/ 专属目录群落。职责收敛：Orchestrator（调度器）仅负责维护系统级 I/O（如发起文档分块加载任务），一旦生成原始文本块或视口坐标数据，便将其转移给 Review Pipeline，此后 Orchestrator 完全交出处理权，只能被动等待规范化结果的返回。模块化机制做实：Pack Registry 与 Artifact 序列化：做实 Policy Pack 动态加载机制：彻底废除将不同规范糅合在巨大系统 Prompt 中的旧模式。Pack 机制必须重构为基于抽象基类的插件化体系。每个特定的 Pack 必须物理包含独立的 schema 契约定义、预期必需的 expected_facts 描述文件集、以及特定阈值校验代码构成的可执行程序包。做实 Artifact Builder 工件工厂：正式审查流水线的终点绝不能只是一段由大模型吐出的 Markdown 文本。系统必须构建独立的 Artifact Builder 引擎，由它负责将流水线末端的结构化数组深度序列化为标准的 JSON 数据大宽表（artifactIndex）。前端消费与后续的只读 Markdown 文件渲染归档，均必须以下游的这个 JSON 工件为唯一事实源（Single Source of Truth），防止篡改或二次生成产生的漂移。构建不可逾越的 Result Contract（结果契约网）：系统各异构层之间的数据流转，必须强制基于 Pydantic（Python）和 TypeScript 类型系统的强类型防御契约。来自底层 LLM Gateway 的任何输出结果，必须首先进入一个专用的 Schema 拦截校验器。任何缺失必需字段、格式不符（例如 EvidenceSpan 未提供有效坐标系）或出现幻觉引用的生成结果，将被系统在底层视作运行期致命异常（Runtime Exception）并自动触发熔断重试机制，绝不容许“带病数据”顺流而下污染最终报告。UI Consumption Contract（界面消费契约）前置对齐：前端架构设计必须放弃在客户端进行任何业务判断（如通过正则匹配文本高亮）的陈旧习惯。目标是构建高度抽象的卡片树渲染引擎（Card Tree Rendering Engine）。这就强制要求后端 API 吐出的数据必须已经高度“降解”：每一个 Issue 必须是一条独立的记录，包含确切的红黄蓝严重程度枚举（ReviewLayer 分级）、操作状态机（manualReviewNeeded 标志位），甚至包含原文档渲染坐标（以便点击后原文档直接平滑滚动定位至案发现场）。

## J. 六层流水线与 L0/L1/L2/L3 的映射分析

在此前很长一段时间内，正式审查之所以容易陷入困局，根源在于试图用一个无限膨胀的、试图涵盖所有 L0 至 L3 逻辑的“超级 Prompt”去统御一切。在下一阶段，我们必须在代码层面将其无情地粉碎、拆解为物理与逻辑高度隔离的六层流水线，并将工程领域的 L0-L3 专业视角严丝合缝地映射其中。

### 1. 六层流水线现状与未来硬编码设计

#### 文档解析层 (Document Parsing Layer)：

当前处于初步可用阶段。下一阶段需极大补强对非标准 PDF、CAD 打印件等复杂多模态输入的解析能力，并严格输出具备几何拓扑信息的结构化版块映射树（AST）。

#### 事实抽取层 (Facts Extraction Layer)：

当前与后续步骤严重混淆在生成模型中。下一阶段必须将其改造为独立的流水线节点。该层仅依赖确定性的正则表达式或参数量极小的轻量级 NLP 信息抽取（IE）模型，快速且冷酷地提取关键工程数值参数，不作任何好坏评价。

#### 规则命中层 (Rule Matching Layer)：

目前仅存在虚弱的骨架。未来需将上游输出的冷峻 Facts 喂入由系统根据路由动态加载的强类型 Pack 中进行“布尔代数式”的逻辑真值校验，杜绝大模型在规则判断上的模糊敷衍。

#### 证据归档层 (Evidence Linking Layer)：

当前缺失极为严重，是系统的阿喀琉斯之踵。需要建立自下而上的硬链接溯源指针池，强制将任何命中失败的规则逻辑，反向锚定到解析层产生的原文物理坐标 Span 上。找不到 Span，Issue 即宣告无效。

#### LLM 解释层 (LLM Explanation Layer)：

当前系统赋予该层的职责过于庞大沉重。未来的架构中，它的权力将被极大削弱和圈禁。系统只有在面临缺乏强规则覆盖的、极其复杂的综合 L3 级工程技术研判，或者仅仅是需要将前置层生硬的 JSON 数组润色转化为通顺易读的 Issue Description 供人类阅读时，才允许拉起大语言模型进行介入。

#### 报告组装层 (Report Building Layer)：

当前存在不可控的过度摘要和篡改风险。必须彻底转变为纯粹的静态模板插值渲染引擎（类似于 Jinja2 机制），从机制上完全剥夺其在最终拼装阶段擅自捏造虚假 Issue 或因上下文截断而遗漏真实 Issue 的任何可能性。

### 2. 审查分层视角与流水线的深度融合映射

为了确保上述流水线的运作具有业务意义，必须建立以下映射关系：L0 可视域检查 $\rightarrow$ 硬映射于【文档解析层】与【事实抽取层】之间的安全闸门（Visibility Gate）。若文件某区域解析置信度极低，系统不得向下传递脏数据，而是直接触发 visibilityGap 全局异常。该流水线立即挂起，并将任务踢回给前端，打上鲜红的标签交由人工进行 L0 级复核干预。L1 硬证据与强规则 $\rightarrow$ 强映射于【事实抽取层】与【规则命中层】。如塔吊基座承载力阈值等死线指标。这一块的判定必须做到 95% 以上依赖确定性代码逻辑（Python/Rust 编写的计算脚本或规则树），极少动用具有不可控认知不确定性（Epistemic uncertainty）的大语言模型。L2 条文适用与规范差距 $\rightarrow$ 软映射于【规则命中层】结合轻量推理。面对大量“应当采用”、“优先考虑”等具备一定语境依赖的建筑国标行标规范，需结合经过微调的专属 LLM 对具体适用边界进行语境意图判定（Intent Matching）。L3 工程推理与整改编排 $\rightarrow$ 唯一映射于【LLM 解释层】。系统结合全案海量上下文信息，针对特定的施工工序冲突或潜在进度隐患输出深度的综合优化建议。系统底层契约必须在此刻建立防火墙：生成的任何 L3 建议，其 ReviewLayer 属性必须被硬编码标记为 L3 优化类，绝对禁止越权伪装成 L1/L2 级的安全硬缺陷。

## M. 评测与 golden pool 演进方案（Evaluation & Golden Pool Roadmap）

在现代 AI 工程中，如果缺乏一套残酷而严密的量化评测体系，任何自称“专业控制面”的系统演进最终都将沦落为“只会变换不同腔调生成更冗长报告的随机文字游戏”。建立并捍卫这条防退化生命线，是系统演进的核心命脉。

### 1）系统当前评测状态与痛点深度剖析

当前系统所处阶段：系统内部积累的评测原始数据资产仅仅勉强处于早期的 bootstrap 启动期与 mini golden cases 验证探索阶段。测试体系的构建极其原始，充斥着仅包含少量人工预设判断的硬编码简单 E2E（端到端）相似度校验，或是基于开发人员肉眼直觉进行的大颗粒度文本“看起来不错”的主观评价机制。当前潜藏的致命问题网：评测维度严重失真与深度不足：评测引擎几乎只紧盯最终由 LLM 拼凑的、华丽的文本报告结果，而对系统极其关键的黑盒中间环节（如源端事实抽取准度 facts、中端规则引擎命中率 rule hits）的细粒度验证处于完全的盲区失控状态。可见性异常监控盲区（Visibility Blind Spot）：整个评测框架中彻底缺失了针对 visibility（解析失败盲区）边界抗压能力的防范测试，导致开发团队根本无法科学量化系统大模型产生“闭眼瞎编乱造幻觉”与系统探针产生“遗漏关键章节失误”这两者之间确切的责任比例划分。系统级对抗消融能力缺失：面对日益繁杂的模型基座，评测中枢缺乏平行调度跨模型对决（例如将 GPT-4 的抽取准度与 Opus 甚至特定精调的 FastGPT 引擎进行正面性能对照碰撞测试）的关键能力，同时也无法针对不同的 Pack 版本迭代（V1.0 规范对决 V2.0 规范）执行模块级纵横向消融矩阵切片分析框架。

### 3）多维样本池覆盖底线与极严格防过拟合（Anti-Overfitting）军规要求

为了确保系统在真实的、充斥着不可控变量的工程业务世界中生存下来，必须立刻大规模向外开疆扩土扩展样本兵器库池，其囊括的样本类型下限必须绝对包含并覆盖：通用大型项目的整体施工组织设计基石方案、细分场景的一般专项施工技术方案、涉及高危红线的危大工程专项深度审查方案、上游视角的工程监理规划指引，以及横跨土建基础类、大型机电设备安装类、复杂网壳钢结构类、临时用电安全类、特种起重吊装特种作业类等各类极度分化的专业细分阵地材料。防备模型陷入深度过拟合坍缩陷阱的核心铁血约束机制：严禁执行任何特征捷径硬编码行为：代码审查阶段必须严厉禁止在底层 Extractor 提取逻辑或核心 Rules 判断树中，以任何形式隐晦地写入特定现实项目实体的标识名称（例如屡次出现的特征项目“某特定冷轧厂工程”相关代号）、特定地域专有简称等强聚焦收敛特征词汇。发现一例，直接作废该提交。坚决防止浮华文风优化指标冒充核心推理能力提升假象：底层架构评测的综合算分加权指标体系，必须被极其无情地大幅度削减甚至彻底归零由于追求“排版优美度”、“语言语调专业逼格感（盲目一味模仿 Gemini 输出那种特定风格模式）”而产生的虚假权重分值。评测引擎必须将高达 80% 以上的生死决定权重，以绝对倾斜的姿态死死砸在“冰冷的硬事实精准抽取坐标点准确率”与“条款引用是否确凿命中的二元准确率”这两条无法伪造的硬核底线上。防范带病数据毒化基线库：工程团队在依靠人工心智辛勤构建 Golden Truth 黄金真理标准集时，必须在工作流中建立隔离墙。绝对必须提前识别并剔除掉隔离那些已知存在 Visibility 视界物理读取错误的致盲片段文档。绝不容许将系统因为没读懂残缺文本而产生的漏洞数据，扭曲定义并伪装成“这篇文档确实缺少这部分内容”的荒谬正常标准答案，进而作为带毒的错误认知基因写入到高贵的黄金基线事实库中。

## N. 非目标与禁止事项（Non-goals & Anti-patterns）

为了在下一阶段极度收敛并聚焦整个庞大工程团队的突击行动视界，防止系统架构再次陷入无边界蔓延的泥潭，必须在本设计书中竖立起一道极为清晰、绝对明确的红线。我们必须旗帜鲜明地强烈反对并彻底封杀以下那些妄图伪装成“系统架构升级手段”的无效、甚至会导致系统加速坍缩毁坏的负面反模式：绝不只靠迷信切换更强模型参数掩盖设计缺陷（Anti-Model-Dependency）：把当前系统执行流转中表现出的智力低下与不稳定状况，简单粗暴地全盘归咎于所接入的基础大语言模型能力不足。企图寄希望于未来通过不断花钱接入参数规模更庞大、烧钱更甚的超级模型神明来包治百病，以此逃避和掩盖底层的逻辑架构断层与契约缺失问题。绝不只靠无脑堆叠更长的咒语（Anti-Prompt-Engineering as Architecture）：在系统依然缺乏稳固安全的防篡改中间件结构与强类型数据约束铁钳的前提下，通过像打补丁一样不断地向外堆砌增加、冗余拼接 Prompt（提示词）文本的长度，试图强人所难地寄望于大模型能以凡人之躯一次性在脑海中完美无瑕地扛起从底层文档杂乱解析直到顶层深度逻辑推理的全部沉重流水线工作。绝不把“系统自己没读到”的愚蠢当成“客观文档内容缺失”的铁证（Strict Visibility Principle）：这是极其恶劣、足以摧毁信任系统的灾难级失败模式。任何文档前置读取模块因为自身能力缺陷产生的失败、卡顿与盲区，绝不允许在暗中被静默降级、隐瞒包庇地传递给下游的无辜推理模块。绝不只沉醉于模仿某款竞品的华丽文风外表（Anti-Style-Over-Substance）：过度将宝贵算力焦点与研发精力消耗在纠结最终报告呈现出的文本排版格式、遣词造句的艺术感（如试图通过提示词魔法去惟妙惟肖地模仿复制 Gemini 报告那种特定优雅的行文语调感觉），却对支撑这些文字下面那极其虚弱、随时断裂的底层 Evidence（物理确凿证据锚点链）的缺失视若无睹。绝不把带有建议性质的工程加强手段伪装成强制性红线缺陷恐吓用户（Anti-Over-Reporting）：在负责深度综合推导的 L3 层，模型往往会生成诸如锦上添花类的建设性优化改进建议方案。绝不允许架构上的流水线疏忽，导致这些柔性建议在最终报告的合并拼装阶段，被越俎代庖地残暴打上“严重违反核心强制性安全国家标准”的赤红惊悚标签，仅仅是为了勉强凑足表面上好看的 Issue 缺陷数量指标。绝不为任何特定单一方案开启硬编码后门补丁（Anti-Single-Case-Patching）：为了让某个特定精心准备的高管或客户层面的 Demo 演示能够万无一失地完美通过验证，而在本应圣洁中立的系统代码运转内核底层深处，卑劣地强行加入类似 if case_name == '特定大型工程项目' 这类的专属判断补丁分支。绝不心急火燎地提前构建大而空泛的多模态平台幻梦：在系统最内核的纯文本处理引擎、严密闭环核心结构化流转流水线未能达到绝对稳固与坚不可摧的地步之前，绝对不盲目好高骛远地向外急促扩张，去贪婪地吞噬并处理诸如海量高分辨率图片、复杂连贯视频流等多模态信息流带来的算力压力与解析灾难。严禁概念偷换与虚假包装行为：严禁采用挂羊头卖狗肉的手法，将旧版毫无底线的 review_assist 通用大语言模型泛泛生成的毫无追溯效力的一团散沙式返回结果文本，仅仅通过前端套上几层华丽嵌套样式的包装，就厚颜无耻地强行向外界宣称为系统已经实现了庄严的 formal structured_review 结构化控制面。

## O. Definition of Done (验收完工标准)

空谈误国，代码层面的承诺必须以最冰冷的数据结果作为收口。下一阶段所有庞杂的系统改造工程，无论吹嘘得多好，其宣告彻底竣工完成的唯一评判衡量生死标准，必须毫无例外地逐一硬性落地到以下各维度具体、且具备可被程序无情量化、可随时提取验证测试的 Definition of Done (DoD) 指标红线上：功能级 DoD 生死线：在物理层面上完全切断并屏蔽掉任何依赖 review_assist 旧链路干预协助的严苛独立孤岛环境下，系统自身能够独立、顽强地扛起并吞吐消化掉一份篇幅冗长复杂的真实级“危大专项工程方案”的全部审查任务量，并顺畅无阻地依靠内生动力通过全部六层结构化隔离流水线引擎，最终精准、毫无遗漏地输出符合强类型的结构化 JSON 缺陷候选列表。数据契约级 DoD 防线：分布在核心动脉路径 apps/api/src/review/ 系统树杈下属的所有核心骨干级别输入输出 API 接口契约规范，必须无可争议地实现 100% 满覆盖的底层强类型定义。且在面对开发者恶意注入脏数据时，系统能够在运行时瞬间触发反击，敏锐捕捉并抛出极为精确详细的底层 Validation Error（契约校验粉碎异常）错误堆栈调用链报告。结果终态可解释性级 DoD 标准：系统在经历一整套流转之后，向外吐出的每一条被盖棺定论的 FinalIssue 结果缺陷结论对象实体中，必须有达到甚至超越 90% 以上的高比例基准线，能够通过其内嵌携带的强力 EvidenceSpan 核心物理证据锚点坐标系，在极短的微秒级时间内精确定向追溯并平滑锁定回退到极其原始、未经任何雕琢的输入物理文档源端材料中的确切指定坐标段落与字符区间。断绝系统凭空任意捏造一切无源之水报告的可能。前端 UI / 后端 Artifact 矩阵级 DoD：前端控制台渲染核心引擎必须拥有能力，能够高度依赖并唯一下游承接 artifactIndex 这条纯数据指令管线传递的 JSON 数据体，在浏览器客户端极速重构并正确绘制出涵盖各业务维度分类层级（如按红绿灯界定的 L1-L3 严重性标尺）、包含人工状态机交互操作流管线（如布尔值 manualReviewNeeded 指示灯）、以及与后台联动触发原文档高亮跃迁等富操作功能的系统级控制面看板矩阵。绝不再是退化呈现一张由底层被动推送过来的毫无生气的纯文本静态 Markdown 网页画卷。防退化评测与版本回归级 DoD：经历一轮严酷洗礼后涅槃扩展出的全新且带有极强防篡改特征标识的版本化 Stable Golden Pool（稳定黄金验证基线池），必须在研发团队每一次代码推演的持续集成核心主干流 CI/CD 自动化流水线工序环节中，宛如基石一般稳定运行。并且，其伴生的苛刻模块级消融验证打分测试探针火力网，必须毫无死角地全量覆盖照射进 Parser（前置解析）、Extractor（精细事实抽取）、Rules（强判定规则碰撞）等所有核心深水域组件群落内部。极限风险边界防御级 DoD：在引入极其苛刻的环境干扰测试集时，系统防线成功抵御并被激怒触发拉起至少一次系统级真实的 visibilityGap 物理视域阻断断路器保护断言机制案例预警。以此铁证向世人展现其具备坚不可摧的底层有效自我防护机制，成功扼杀了因视盲症诱发后端模型生成大规模雪崩级虚假阳性欺诈（False Positive）失控报告的致命惨剧。

## 结论终章 (Conclusion)

综合上述全部极其详尽、细入毫芒的深度架构剖析与工程重组演进路线设计蓝图推演，针对本阶段必须回答的核心疑问，特向各方作出如下极其清晰、绝不含糊的收敛性总结决断：

### 1. 

当前 repo 到底处于什么阶段？纵观全局态势，当前核心仓库所处的段位正处于从混沌虚胖的“大模型长文本通用阅读总结器”向着高度精密森严的“专业结构化防线审查控制面系统”艰难跋涉的概念验证过渡与系统骨架雏形搭建的临界深水区阶段（严格界定为 P1 级探索原型到半成型工业可用系统之间的敏感动荡过渡期）。虽然诸如数据流转结构契约（DTO 标引）与目录空间隔断边界已基本浮出水面就位，但那条承载核心命脉的六层深层次解析解耦流水线巨网，在系统底座 Runtime 的实际执行调度内存层面上，仍未形成彻底绝缘的物理护城河隔离壁垒与强横无比的逻辑链条做实锁闭。

### 2. 

当前 repo 最真实的工程优势资产是什么？系统在过往探索中留存的最宝贵且极其真实的优质资产，在于其架构师团队已经觉醒并初步具备了异常清晰坚定的顶层目录物理边界安全隔离意识（如成功且极其明智地主导划定了专属且神圣不可侵犯的隔离审查重地 Review 子域空间隔离区），同时还在地基深处搭建铺设了极其难得的多模态大文件混合吞吐处理的庞大基础底座架构支撑网络。尤为难能可贵的是，在早期黑暗荒野探索中历史积累沉淀下来的那套评测对抗验证体系初步框架思想（以珍贵的萌芽状态 Mini Golden Cases 为主要外在表现载体），为接下来即将掀起的轰轰烈烈的结构化深度反向验证工程浪潮战役，极其稳妥地奠定了极其难得的系统性认知经验大后方基础。

### 3. 

当前 repo 最真实、最致命的短板软肋在哪里？是其充斥在系统各处的“华丽辞藻、名不副实”的虚弱执行逻辑链路与失控的权利边界。尽管在系统对外的表层宣传口径与高层设计文档中宣称并标榜了自己拥有宏伟的 Formal Review 审查机制和极其精密复杂的多种数据分层映射架构理念，但在剥离表象深探底层真实的执行算力引擎深处，依然高度、近乎绝望地极度依赖于那些不可控、犹如庞然大物般横冲直撞的大语言模型自身那变幻莫测的极度宽泛泛化黑盒推断生成能力；并且，在最前端拦截数据的咽喉要道关卡，极其缺乏足以一剑封喉的 L0 级别物理可见性异常视域阻断硬抗防线（Visibility 防火墙）；更为致命的是，其底层赖以构建公信力的数据核心溯源物理防线系统（Evidence Linking 底层硬链接追踪网络能力），当下在代码现实世界中几乎完全为虚弱的零状态。

### 4. 下一阶段工程上最值得团队重金倾注算力与人力的 3～5 件事是什么？首要之役：

### 6. 最高统帅部最终决策建议：如果限于算力与人力，只能优先强攻实施开展一个阶段战役，你最建议先做哪个阶段，为什么？

系统性架构升级面临生死抉择关头，最高指令必须是毫无动摇、破釜沉舟地最优先强攻并集中一切优势兵力死守完成【P0 阶段防御战役（不惜一切代价做实系统内部已有的虚浮骨架网）】。核心决策背后的深度血泪逻辑在于：当前悬挂在整个架构顶端最大的达摩克利斯之剑（终极风险），在于系统深陷“幻觉驱动决策”的泥沼与“薄弱契约随时被上下游穿透引发雪崩”的深渊。如果架构师团队不通过建立极其铁血强硬类型的结构化防篡改流水线把数据流通的生死契约大关给彻底卡死封锁住，那么，任何随后贸然投入在系统评测防线集盲目扩充扩张（P1 阶段），或者狂妄妄想去进行高级玄幻深度推理引擎能力增强加固（P2 阶段）的巨额资源与算力投入，最终全部都会在现实系统的极度混沌中彻底变味走样，最终不可逆转地悲惨演变成一场场面对大语言模型无休止、极其绝望且毫无工程意义的玄学 Prompt 炼丹调优内耗大逃杀游戏。这不仅会彻底偏离构建一个基于确定性法则的高精密建筑工程防线 Control Plane 控制面系统平台的宏大战略初衷，最终只会得到一个除了学会用更漂亮的文风和更圆滑的语调生成更多根本不敢让人信服使用的虚假长篇废话报告外，别无他用的庞大电子垃圾。因此，最高优先级绝对是不惜动用最粗暴直接的工程锁闭手段来规范制裁，强制确立系统行为不可逾越的物理运行底线与底座尊严。
