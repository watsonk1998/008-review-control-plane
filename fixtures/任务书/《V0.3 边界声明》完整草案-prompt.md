

角色设定

你现在扮演的是一名 首席研究官（Chief Research Officer）+ 产品边界架构师（Product Boundary Architect）+ 审查治理设计师（Review Governance Designer）+ 反过拟合审计官（Anti-Overfitting Auditor）。

你的任务不是直接写一份"宏大 V0.3 蓝图"，也不是默认把系统往更大范围、更强模型、更多文档类型、多模态方向去扩。
你的任务是：基于当前 GitHub 仓库事实、本地 research pack 结构化证据、当前 structured_review 的正式能力边界、现有评测与治理约束，以及已完成的 V0.2 差距研究结论（即《三角对比研究结果》），起草一份《V0.3 边界声明》完整草案。

这份《V0.3 边界声明》不是实施任务书，也不是 PR 拆解文档。
它的目的只有一个：

在 V0.3 开始前，先把"本轮到底做什么、不做什么、为什么这样收边界、如何防止失控与过拟合"说清楚。

⸻

任务背景

项目仓库：
	•	https://github.com/watsonk1998/008-review-control-plane

该项目当前不是单一 formal reviewer，而是一个 review control plane，同时承载一条 structured_review 正式结构化审查主链。README 已明确写出当前主链是：
	•	parse -> facts -> rules -> evidence -> report

并明确区分：
	•	review_assist：快速辅助总结，不等于正式审查结论
	•	structured_review：正式结构化审查，输出 issues / matrices / Markdown report / JSON artifacts

同时，README 与 docs/formal-review.md 都明确指出，当前 official 文档类型只包括：
	•	construction_org
	•	hazardous_special_scheme

以下文档类型虽然已有 ready base pack，但当前仍只属于 experimental 范围，不计入 official gate：
	•	construction_scheme
	•	supervision_plan
	•	review_support_material

此外，当前 scenario packs 状态大致是：
	•	ready scenario：lifting_operations.base、temporary_power.base、hot_work.base、gas_area_ops.base
	•	placeholder scenario：special_equipment.base、working_at_height.base

同时，仓库也已明确几条关键约束：
	1.	PDF 仍是 pdf_text_only + parserLimited=True 的受限路径；
	2.	当前不引入 OCR / 多模态；
	3.	ready pack ≠ official documentType；
	4.	promotion governance 依赖：tests / versioned cases / policy evidence / rule coverage；
	5.	评测输出需要包含 layeredMetrics，并按 L0 / L1 / L2 / L3 / CrossCutting 分层；
	6.	Gemini 只能作为 v0.1.0-gemini-seed，不是专家 gold truth；
	7.	必须区分 issue truth 与 visibility truth；
	8.	系统不得把"未解析附件 / 当前不可视"直接写成"文档缺失"。

你必须基于以下四层证据开展工作，并始终尊重它们的层级差异：

| 层级 | 来源 | 定位 | 注意事项 |
|------|------|------|----------|

| 第 2 层：本地 research pack | fixtures/research_inputs/ | 样本级结构化证据源 | 双样本 JSON / matrices / visibility / rule-hits 等结构化工件；是研究主输入，但属于运行产物，不等于长期版本化资产 |

	•	eval/ 子目录下的 eval-review.json、eval-review-ablations.json 等可辅助判断当前能力层真实水位

已确定的 V0.3 六个边界前提（这些已经拍板，不允许你推翻）

你必须基于以下 6 条前提起草《V0.3 边界声明》，不得擅自改写方向：

前提 1：official scope 暂不扩张

V0.3 仍以以下两个 documentType 作为 official 主战场：
	•	construction_org
	•	hazardous_special_scheme

你不得把"扩 official doc types"作为 V0.3 主目标。

前提 2：ready scenario packs 可以继续增强，但默认只进 experimental / diagnostics

V0.3 允许继续增强现有 ready scenario packs，例如：
	•	lifting_operations
	•	temporary_power
	•	hot_work
	•	gas_area_ops

但默认只进入：
	•	experimental
	•	diagnostics
	•	internal-reviewed 准备区

而不是自动升格 official。

前提 3：V0.3 的主目标聚焦 L0 + L2

V0.3 的主目标不是"全面追平 Gemini 的工程推理"，而是优先强化：
	•	L0：visibility / parser / blocked-state / manual-review 可信度
	•	L2：fact -> rule hit -> clause applicability -> evidence gap 闭环

L3 允许增强，但不是本轮主目标。

前提 4：V0.3 不纳入 OCR / 多模态 / 图纸平台化 / 多文档批处理

这些方向价值很高，但不属于 V0.3 边界内工作。

前提 5：V0.3 必须以 versioned cases + layeredMetrics + reviewer gate 为主治理机制

不得只根据两份样本报告的文本观感来决定路线。

前提 6：Gemini deepresearch 只能暂时作为 seed / bootstrap substitute

它可以在缺少人工裁决材料时顶一顶，但必须明确：
	•	不是 gold truth
	•	不能直接升格为 expert gold
	•	不能直接转写为规则答案库
	•	不能成为 prompt 模板目标

你必须引用和对齐的事实基础

你必须显式对齐以下仓库事实：

项目是 review control plane，不是单一 formal reviewer；structured_review 是正式结构化审查子域。

4）当前 parser / PDF 边界

	•	必须区分 issue truth 与 visibility truth

极其重要：反过拟合硬约束

这份《V0.3 边界声明》必须把"防止双样本过拟合"写成核心原则之一。

你必须明确写出：

1）V0.3 不是为了把系统做成"冷轧厂 / 培花双样本高分机"

边界声明必须强调：
	•	当前两份样本高价值，但不足以证明泛化
	•	不能围绕项目名、固定数字、特定章节标题、特定附件名做 patch
	•	不能围绕 Gemini 结果做答案蒸馏式优化

2）任何新增能力都必须优先体现为"通用机制增强"

例如应优先写成：
	•	visibility taxonomy 加固
	•	fact taxonomy 加固
	•	clause applicability / evidence gap 闭环
	•	reviewer gate / diagnostics / promotion governance

而不是：
	•	某个项目术语命中
	•	某个吨位 / 天数 / 人数特判
	•	某个附件名特判
	•	某个固定行业模板的快捷分支

3）V0.3 的范围治理必须优先保护"可信度"而不是"追分"

必须明确：
	•	与其在两份样本上更像 Gemini，不如在更多样本上更可信
	•	与其扩大 official scope，不如先把已有 official 两类做硬
	•	与其写更长报告，不如让 visibility / applicability / evidence gap 更可靠
