# V0.3 前置工作流程指南——文档生成顺序与依赖关系

> **定位**：这不是治理文档本身，而是指导"治理文档应按什么顺序生产"的调度总纲。
> 任何研究型 AI / 执行者在开始 V0.3 前置工作时，应**先读本文件**，再按顺序执行。

---

## 核心原则

**"三角对比研究结果"不是三份治理文档之一，而是它们的共同上游研究输入。**

仓库当前的事实基础决定了这一点：项目先有 structured_review 主链、official/experimental 边界、artifact/result contract、PDF 可视域约束、评测与 versioned truth 语义；三份文档是在这些事实之上做"边界声明 → 差距裁决 → internal-reviewed 裁决"的治理化加工，不应反过来先写治理文档再补研究。

否则会出现 **"没有研究先写政策"** 的问题。

---

## 研究输入与证据层级

所有文档的研究与写作都必须基于以下四层证据，并始终尊重它们的层级差异：

| 层级 | 来源 | 定位 | GitHub 可读 | 注意事项 |
|------|------|------|:-----------:|----------|
| **第 1 层：GitHub 仓库事实** | README、docs/formal-review.md、fixtures/review_eval/README.md | **系统边界真相源** | ✅ | 所有判断的顶层约束 |
| **第 2 层：精选结构化证据** | `fixtures/research_inputs/` | **样本级结构化证据源** | ✅ | 从本地 `artifacts/research-pack/` 精选入仓；研究主输入 |
| **第 3 层：Markdown 报告** | `fixtures/supervision/` 中的 V0.0/V0.2/Gemini 结果 | **人类可读呈现层** | ✅ | 适合快速浏览，但不应作为 truth 分层的主要依据 |
| **第 4 层：Gemini** | Gemini deepresearch 结果 | **seed / candidate reference** | ✅（.md 在 supervision） | 仅可用于候选问题发现、差距探针 |

### 仍需人工手动上传的文件

以下内容仍需根据研究进展手动上传：

| 文件 | 何时需要上传 | 说明 |
|------|-------------|------|
| 三角对比研究结果正式文档 | 第 1 步完成后 | 当前仅有 prompt，尚无正式研究产出 |
| 新一轮 eval 结果 | 每次重新运行 eval 后 | 更新 `fixtures/research_inputs/latest-eval-summary.md` |
| 新增样本的结构化工件 | 新增监督样本时 | 按照 research_inputs/README.md 的精选原则复制入仓 |
| 本地临时 eval 日志 | 通常不需要 | 除非调试需要，原则上不上传 |
| 更新后的结构化 JSON | 系统版本升级后 | 重新运行 structured_review 后精选更新 |

### 关于 latest-eval-summary.md

- 它是"当前治理状态快照"，可作为边界声明和差距裁决的阶段性辅助证据
- 但它 **不是长期仓库真相**，更不能因为某次 eval 通过就自动认定差距已关闭
- 它不能替代 versioned stage gate

## 生成顺序（推荐严格遵守）

```
第 0 步：准备输入
│
└── 研究型 AI 从 GitHub 读取：
    ├── 仓库事实（第 1 层）
    ├── 精选结构化证据（第 2 层）— 优先输入
    ├── V0.2 研究设计与实施总文档.md
    ├── 使用说明（给研究型 AI）.md
    └── latest-eval-summary.md（当前治理快照）


第 1 步：先做"三角对比研究结果"  ← 上游研究，非治理文档
│
└── 产出：三角对比研究结果
    ├── 优先消费 research_inputs/ 中的结构化 JSON / matrices
    ├── V0.0 vs V0.2
    ├── V0.2 vs Gemini
    ├── V0.2 实际结果 vs V0.2 设计目标
    ├── L0 / L1 / L2 / L3 分层差距
    └── 反过拟合风险初判


第 2 步：基于"三角对比研究结果"做《V0.3 边界声明》
│
└── 产出：V0.3 边界声明
    ├── 可参考 latest-eval-summary.md 作为治理快照
    ├── 本轮做什么
    ├── 本轮不做什么
    ├── official scope 是否扩
    ├── scenario packs 如何治理
    ├── 为什么主攻 L0 + L2
    └── 为什么禁止 OCR / 多模态 / 多文档批处理进入本轮


第 3 步：基于"边界声明 + 三角研究结果"做《V0.2→V0.3 差距裁决与反过拟合约束清单》
│
└── 产出：差距裁决清单
    ├── 裁决优先依据结构化 JSON / matrices
    ├── 哪些差距 Confirmed
    ├── 哪些 Partially Confirmed
    ├── 哪些 Deferred
    ├── 哪些 Rejected
    ├── 哪些进 official / experimental / diagnostics
    └── 哪些因过拟合风险被否决


第 4 步：基于"边界声明 + 差距裁决清单 + 三角研究结果"做《双样本人工复核裁决稿》
│
└── 产出：internal reviewed adjudication notes
    ├── truth 分层优先依托结构化 JSON / matrices / visibility 对象
    ├── 样本 A 逐 issue 裁决
    ├── 样本 B 逐 issue 裁决
    ├── issue truth / visibility truth / evidence truth 分层
    ├── 可进入 v0.2.0-internal-reviewed 的条目
    └── 暂不进入的条目


第 5 步：三份文档交叉回收
│
└── 一致性检查
    ├── internal-reviewed 裁决稿里，是否有越过边界声明的条目
    ├── 差距裁决清单里，是否有被样本 truth 否决却仍推进的条目
    ├── 边界声明里，是否遗漏了裁决清单已经确认的主线项
    └── 是否仍符合 anti-overfit 原则
```

- **《差距裁决清单》** 再进一步约束《internal reviewed adjudication notes》

## 执行策略（串行/并行建议）

```
阶段 A：串行（不可跳过）
└── 先做"三角对比研究结果"
    └── 再做《V0.3 边界声明》

阶段 B：可并行（在《V0.3 边界声明》定稿后）
└── 同时起草
    ├── 《V0.2→V0.3 差距裁决与反过拟合约束清单》
    └── 《双样本人工复核裁决稿》

阶段 C：串行回收
└── 用《差距裁决清单》审核《人工复核裁决稿》
    └── 再统一修订三份文档
```
