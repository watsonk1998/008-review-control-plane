## B. 当前系统阶段定位

### B1. 已确认事实（GitHub 仓库证据）

当前仓库已经不是“只有 prompt 和 demo”的状态。GitHub 主干明确把项目定位为 **review control plane / orchestration 项目**，并把 `structured_review` 作为正式结构化审查子域；runtime 负责 orchestration，正式审查判断下沉到 `apps/api/src/review/`。`structured_review` 当前已支持 `fixtureId` 或 `sourceDocumentRef` 二选一输入，结果契约已包含 `summary`、`visibility`、`resolvedProfile`、`issues`、`matrices`、`artifactIndex`、`reportMarkdown`、`unresolvedFacts`，并且 `manualReviewNeeded` 是 canonical 字段，`whetherManualReviewNeeded` 仅保留 legacy 兼容。([GitHub][1])

后端与 review 子域已经形成真实执行链：`CreateTaskRequest` 强校验 `structured_review` 只能在 `fixtureId` 与 `sourceDocumentRef` 中二选一；`routes/tasks.py` 已暴露 create/list/get/result/stream/support-scope/reviewer-decision/artifacts 接口；`uploads` 路由支持 `.docx/.pdf/.md/.txt` 上传并返回 `SourceDocumentRef`；SQLite 任务存储已持久化 `document_type`、`discipline_tags_json`、`strict_mode`、`policy_pack_ids_json`、`source_document_ref_json`、`reviewer_decision_json`。([GitHub][3])

review 子域也不是骨架而已。`schema.py` 已定义 `StructuredReviewTask`、`VisibilityAssessment`、`DocumentParseResult`、`IssueCandidate`、`FinalIssue`、`StructuredReviewResult`；`pipeline.py` 会实际写出 `structured-review-parse`、`structured-review-facts`、`structured-review-rule-hits`、`structured-review-candidates`、`structured-review-result` 及多种矩阵 artifacts；`task-detail.tsx` 已消费 top-level `visibility` 与 `reviewerDecision`，前端表单与首页也已支持 structured_review 专用参数和上传文档路径。([GitHub][4])

当前正式支持范围也已经写进代码，而不是只写在文档里。`support_scope.py` 和路由测试都表明：**official document types 只有 `construction_org` 与 `hazardous_special_scheme`**；`construction_scheme`、`supervision_plan`、`review_support_material` 仍然在 document-type 维度标记为 `skeleton`。但这不等于它们没有实现：GitHub 当前 pack registry 中，这三类 base pack 以及 `lifting_operations`、`temporary_power`、`hot_work`、`gas_area_ops` 场景 pack 都已是 `ready` 且带有明确 `ruleIds`；只有 `special_equipment.base` 与 `working_at_height.base` 仍通过 placeholder 机制补位。([GitHub][5])

评测与测试同样已经成体系。`load_cases()` 已同时加载 legacy `metadata.json` 和 versioned `case_metadata.json`；`metrics.py` 已有 `facts_accuracy`、`rule_hit_accuracy`、`attachment_visibility_accuracy`、`manual_review_flag_accuracy`、`hard_evidence_accuracy`、`policy_ref_accuracy` 等指标；`harness.py` 已支持 main / ablations / cross-pack / cross-model；测试文件已覆盖 structured_review 输出、PDF 显式 parser、上传文档 runtime、reviewer decision 路由、support-scope 路由，以及 versioned stage gate。([GitHub][6])

### B2. 盘点注记（路径对齐）

你要求核查的若干路径，与 GitHub 当前主干存在两处重要差异，Codex 落地前必须先对齐这两个现实：

### B3. 合理推断

基于以上事实，当前仓库最准确的阶段不是“空白 formal review 系统”，也不是“已可稳定替代人工签发的正式审查器”。更贴切的表述是：

> **一个已定型的 review control plane，外加一条已经可跑、可回归、可上传、可复核记录的 `structured_review` 正式结构化审查主链；其中 official scope 在 P0 上只收敛到 `construction_org` 与 `hazardous_special_scheme`，其余 doc type / packs 处于 ready-but-experimental 或 skeleton-but-partially-landed 状态。**

## C. V0.2 目标与边界

### C1. 已确认事实（GitHub 仓库证据）

V0.2 不是因为仓库“还没有 formal review”才开始做，而是因为仓库已经有了 formal review 主链、上传入口、support-scope、reviewer decision、versioned eval 与 SSE 观察面，现在进入了一个更难也更值得投入的阶段：**要把深度、依据链与可回归性做上去**。同时，GitHub 当前也明确了若干约束：`strictMode` 仍是 `reserved / no-op`；当前只支持单文件 `fixtureId` 或单文件 `sourceDocumentRef`，不支持多文档批处理、多模态 OCR 平台化链路；official 支持范围仍仅两类文档。([GitHub][1])

### C2. V0.2 主目标

**主目标：**
在不重开平台新盘的前提下，把现有 `structured_review` 从“可跑的正式结构化审查主链”推进到“可稳定比较、可分层评测、可人工复核闭环、且在真实施组审查深度上更接近 Gemini deepresearch 风格”的 V0.2。

### C3. V0.2 不做什么

1. 不重新设计 control plane 顶层架构。
2. 不把 `review_assist` 包装成正式审查结论。
3. 不把多文档批处理、对象存储、OCR/多模态平台化当作 V0.2 主线。
4. 不把“更像 Gemini 的文风”当成功标准。
5. 不把 `strictMode` 扩展成一套未经评测支持的复杂语义开关。

### C4. V0.2 成功标准

| 维度       | 成功标准                                                                                                      |
| -------- | --------------------------------------------------------------------------------------------------------- |
| L0       | `visibility`、`parseMode`、`parserLimited`、manual review 语义在 fixture / upload / DOCX / PDF / MD/TXT 上一致、可回归 |
| L1       | official scope 的硬规则、红线、结构完整性与可视域命中更稳，不把“未解析”写成“缺失”                                                        |
| L2       | facts / rule hits / policy refs / unresolvedFacts 可进入 stage gate，不只看最终 issue                              |
| L3       | inferred-risk 与整改建议更可信，且与硬缺陷、visibility gap、建议性增强显式区分                                                     |
| Workflow | reviewer decision 能稳定写回、读取、展示，形成最小人工复核闭环                                                                  |
| Eval     | legacy baseline 不退化，official versioned cases 成为主 stage gate，experimental 只进 diagnostics                   |

### C5. V0.2 与当前仓库现状的关系

V0.2 不是“继续补平台壳子”，而是承接当前仓库已经存在的 `structured_review` contract、review 子域、pack registry、artifact API、upload/sourceDocumentRef、reviewer decision、support-scope、SSE/polling fallback 与 eval harness，把 formal review 的 **深度、证据链、可视域语义与评测可解释性** 做实。([GitHub][1])

## D. L0 / L1 / L2 / L3 差距研究

先给出总表，再逐层展开。

| 层  | 当前 GitHub 已做到什么                                                           | 与 Gemini deepresearch 比较的主要差距                              | V0.2 首要投入                                   | 主因类型                     |
| -- | ------------------------------------------------------------------------- | ---------------------------------------------------------- | ------------------------------------------- | ------------------------ |
| L0 | 已有 `VisibilityAssessment`、`parseMode`、附件状态、manual review、上传/fixture 双输入   | 输入可视域仍不均，PDF 仍是 `pdf_text_only`，L0 还未成为单独 stage gate 语言    | parser/visibility/manual-review 统一化         | parser + contract + eval |
| L1 | 已有 official 两类文档的硬规则，experimental/base/scenario packs 也已有真实 `ruleIds`     | hard rules 的法规分层、promotion 机制、official/experimental 边界还不够硬 | official L1 规则核深化 + pack promotion 标准       | pack/data/eval           |
| L2 | 已有 ClauseStore、EvidencePack、policy refs、facts/rule-hit metrics、若干场景 L2 规则 | applicable clause 链还偏静态映射，evidence gap 的解释链不够完整            | clause/applicability/unresolvedFacts 贯通     | evidence/data/eval       |
| L3 | 已有 inferred-risk 规则、矩阵、report、reviewer decision 最小闭环                      | 工程冲突分析、整改编排、建议与硬缺陷分离仍偏轻                                    | rule-grounded remediation & review workflow | report/workflow/eval     |

### D1. L0：可视域 / parser / 附件可见性 / manual review trigger

**已确认事实**
当前仓库已经把 L0 做成显式 contract，而不只是隐含状态：`VisibilityAssessment` 是 top-level canonical 字段；`DocumentParseResult` 显式区分 `docx_structured`、`pdf_text_only`、`markdown_text`、`plain_text`；`DocumentLoader` 已分别处理 `.docx/.pdf/.md/.txt`；`attachment_indexer` 会对附件给出 `parsed / attachment_unparsed / referenced_only / missing / unknown` 等状态，并用 `manualReviewNeeded` 标记非 fully visible 的条目；测试也直接断言 `attachment_unparsed` 会触发“附件处于可视域缺口，需人工复核原件”。前端详情页已把 `visibility` 独立渲染成可读面板。([GitHub][4])

**合理推断**
相对于 Gemini deepresearch 风格的“在真实文档上给出连续论证”，当前系统与其差距首先不在模型，而在 **输入可视域是否足够稳定、足够可解释**。因为只要 parser 可视域不稳，L1/L2/L3 全部会被污染。当前 DOCX 路径更强，PDF 仍显式标记为 `pdf_text_only` 且 `parserLimited=True`；这说明系统已经知道问题在哪里，但还没有把 L0 彻底做成第一优先级的 stage gate。([GitHub][4])

**建议改造项**
V0.2 在 L0 的第一优先级不是“换更大模型”，而是：

1. 让 `parseWarnings`、`parserLimited`、附件原因码和 manual review reason 在 API/result/report/eval/UI 完全同构。
2. 单独产出 L0 artifact，并在 eval 中把它当一层独立门槛。
3. 对 PDF 继续保持保守语义：**未解析 ≠ 缺失**，并显式提示“当前判断只在 text-only 可视域内成立”。
4. reviewer decision 要能回写附件级复核状态，闭合 L0。

**判断**
L0 的主要问题是 **parser / contract / eval 问题**，不是靠换模型就能解决的问题。

### D2. L1：硬证据 / 强约束规则 / 法定程序 / 红线类命中

**已确认事实**
当前仓库的 L1 已不是空白：`FindingType` 已区分 `hard_evidence`、`visibility_gap`、`engineering_inference`、`suggestion_enhancement`；rule engine 已有 `hazardous_special_scheme_core_sections`、`construction_scheme_structure_completeness`、`supervision_plan_structure_completeness`、`review_support_material_context_only` 等规则；`construction_org`、`hazardous_special_scheme` 是 official doc type，而 `construction_scheme`、`supervision_plan`、`review_support_material` 虽然在 support-scope 仍为 skeleton，但其 base pack 已是 `ready` 且带真实 `ruleIds`；`lifting_operations`、`temporary_power`、`hot_work`、`gas_area_ops` 场景 pack 也已是 `ready`，只有 `special_equipment` 与 `working_at_height` 仍 placeholder。([GitHub][3])

**合理推断**
与 Gemini deepresearch 相比，当前差距不在“有没有 L1”，而在 **L1 的法规层级、promotion 规则与 official/experimental 边界是否足够硬**。GitHub 当前主干已经出现一个非常关键的现实：部分 doc type / pack 在实现层已经 ready，但在 support-scope 层仍未升格为 official。这是健康的谨慎，但也说明 V0.2 需要补的是 **support-scope 真相管理 + 升格标准 + hard-rule regression**，而不是继续泛泛扩 doc type 名单。([GitHub][5])

**建议改造项**
V0.2 在 L1 的投入顺序应是：

1. 先加固 official 两类文档的红线与法定程序命中。
2. 再为 ready-but-non-official doc types 定义 promotion 条件：必须同时满足 rule coverage、policy evidence、tests、versioned cases 四条。
3. 把 `fixtures/construction` 中法条底线、重大隐患、危大管理规定系统化成 rule/evidence 资产，而不是直接把原文喂给模型。
4. 把 `visibility_gap` 与 hard defect 在 report、eval、UI 中彻底拆开。

**判断**
L1 的主要差距是 **pack / policy asset / support-scope / eval** 问题，模型只居次要位置。

### D3. L2：条文适用 / 依据链 / applicable clause / evidence gap

**已确认事实**
当前仓库已经有 L2 基础设施：`ClauseStore` 通过 `_RULE_TO_CLAUSE_IDS` 把 rule id 映射到 policy clause；`EvidencePack` 已收录来自《建筑施工组织设计规范》、危大专项方案编制指南、《建设工程监理规范》、《建设工程安全生产管理条例》等条文片段，并带 `severityMapping`、`ruleIds` 与 `applicability`；rule engine 已有 `lifting_operations_calculation_traceability`、`temporary_power_control_linkage`、`gas_area_ops_control_linkage`、`hazardous_special_scheme_emergency_targeted` 等 L2 规则；评测层也已经把 `facts_accuracy`、`rule_hit_accuracy` 纳入 aggregate 指标，并在测试里验证 versioned stage gate 会读取 `expectedFacts`、`expectedRuleHits`。([GitHub][8])

**合理推断**
当前与 Gemini deepresearch 的 L2 差距，核心不再是“有没有条文引用”，而是 **条文适用链是否足够完整**。现在的 ClauseStore 更像“规则到条文”的静态映射；而 Gemini 风格优势通常体现在“为什么适用这条、为什么不适用那条、当前证据缺在哪一段、缺口是否由 visibility 还是正文不足造成”的链式表达。当前仓库已经有 `unresolvedFacts`、`evidenceMissing`、`manualReviewReason` 这些拼图，但还没有把它们变成完整的 applicable clause / evidence gap 评价闭环。([GitHub][4])

**建议改造项**
V0.2 在 L2 的关键动作应是：

1. 为 `RuleHit` / `FinalIssue` 增加或派生 `applicabilityState`，至少区分 `applies / partial / blocked_by_visibility / blocked_by_missing_fact`。
2. 将 `unresolvedFacts` 直接接入 report 与 eval，而不是只挂在结果尾部。
3. 把 `fixtures/construction` 中规范原文拆成 **source → clause → evidence pack → rule → eval expectation** 的可追溯资产链。
4. 引入 L2 级诊断指标：applicable clause completeness、policy-evidence completeness、evidence-gap reason accuracy。

**判断**
L2 的差距主要是 **evidence/clause/data/eval** 问题，不是 prompt 长度问题。

### D4. L3：工程推理 / 冲突分析 / 整改编排 / 建议性增强与硬缺陷分离

**已确认事实**
当前仓库已经有部分 L3 元素：rule engine 存在 `hazardous_special_scheme_measure_linkage`、`temporary_power_control_linkage`、`gas_area_ops_control_linkage` 这类 `inferred_risk` 规则；`FindingType` 已把 `engineering_inference` 与 `suggestion_enhancement` 区分开；pipeline 会产出 conflict / rule-hit / attachment visibility 等矩阵；reviewer decision 能在 task / issue / attachment 级别写回确认、驳回或补件状态。([GitHub][3])

**合理推断**
当前仓库和 Gemini deepresearch 的 L3 差距，主要在于 **工程化编排能力仍偏轻**。现有 L3 更像“若干 grounded inference 的集合”，尚未形成“问题之间如何排序、哪些先补件、哪些先整改、哪些只是优化建议、哪些会阻断正式结论”的强流程表达。换言之，仓库已经能出工程推理句子，但还没完全把工程推理做成 **workflow language**。这部分不能靠自由生成硬堆，因为一旦脱离 rule/fact/evidence grounding，就会回到文风驱动而不是 formal review 驱动。

**建议改造项**
V0.2 在 L3 上不宜追求“大而全的智能整改编排平台”，而应做三件刚需：

1. 将 hard defect / visibility gap / evidence gap / suggestion enhancement 在 issue、matrix、UI 中强分层。
2. 基于现有 rule hits 产出 deterministic remediation buckets：`先补件`、`先补依据`、`先补正文结构`、`建议增强`。
3. 让 reviewer decision 反过来沉淀最小闭环：哪些 L3 建议经人工确认后真的有价值，哪些只是噪声。

**判断**
L3 目前首先是 **report/workflow/eval** 问题，模型只适合作为表述与摘要辅助。

## E. 数据与评测设计

### E1. `fixtures/construction` 的使用策略

**已确认事实**
`fixtures/construction` 已经包含法律法规、危大管理规定、严重缺陷清单、专项方案编制指南、施工组织设计规范、安全检查标准、安全生产/质量管理条例、消防规范、重大事故隐患判定标准、临时建筑规范等原文材料；而当前 `EvidencePack` 已经开始把其中一部分转成 `PolicyClause`，例如施工组织设计规范、危大专项方案编制指南、建设工程安全生产管理条例、建设工程监理规范。([GitHub][9])

**建议改造项**

### E2. Gemini deepresearch 结果的使用策略

**已确认事实**
仓库已经明确把 Gemini 相关结果放在 seed / bootstrap 语境里：`review_eval/README.md` 规定 `v0.1.0-gemini-seed` 是基于 Gemini deepresearch 结果构建的 seed labels，并明确写出“不得把 Gemini 结果直接当作专家 truth”；`fixtures/supervision` 中同时保留了两份 Gemini deepresearch 审查结果和两份 review-control-plane 审查结果，用作真实样本对照。([GitHub][10])

**建议改造项**
Gemini 结果在 V0.2 中只适合扮演以下角色：

| 安全用途                                             | 不安全用途             |
| ------------------------------------------------ | ----------------- |
| seed labels / bootstrap diagnostics              | 专家 truth          |
| 对比基线                                             | 目标文风模板            |
| candidate issues / candidate reasoning inventory | 训练或 prompt 中的隐式答案 |
| reviewer checklist 的候选来源                         | 直接覆盖规则命中结果        |

这意味着：

* 可以用 Gemini 结果帮助发现“当前仓库常漏掉什么”。
* 不可以用 Gemini 结果决定“正确答案是什么”。
* 可以用它做 case bootstrapping。
* 不可以把它当 final gold 或 UI 文风目标。

### E3. 两个施组及相关 deepresearch 结果的使用策略

**已确认事实**
`fixtures/supervision` 中同时保留了两份真实施组源文档（冷轧厂 DOCX、培花 PDF）及对应的 Gemini deepresearch 审查结果、review-control-plane 审查结果；`review_eval/README.md` 则要求 versioned cases 从 `v0.1.x seed` 逐步走向 `v0.2.0-internal-reviewed` 和 `v1.0.0-expert-golden`。([GitHub][11])

**合理推断**
这两个施组样本极其高价值，因为它们同时具备：

1. 真实源文档；
2. 不同系统输出；
3. 明确的 formal review 目标场景。

但它们也极易导致过拟合，因为：

1. 样本量太小；
2. 都是强语境、强命名、强工程背景样本；
3. 很容易诱导规则围绕 case id、项目名、章节习惯、行业特征打补丁。

**建议改造项**
V0.2 对这两个施组的使用方式应是：

| 样本位置                       | 用法                                |
| -------------------------- | --------------------------------- |
| `v0.1.x seed`              | bootstrapping、对照分析、回归 smoke       |
| `v0.2.0-internal-reviewed` | 经过人工复核后可进入 official stage gate 候选 |
| `v1.0.0-expert-golden`     | 只有专家确认后才能升格                       |

同时必须补至少三类新样本桶，避免形成“两例专用审查器”：

* 同类型但低问题密度样本；
* 同类型但 visibility gap 更重的样本；
* 非 `construction_org` 的 `hazardous_special_scheme / supervision_plan / construction_scheme / review_support_material` 样本。

### E4. V0.2 评测设计

**已确认事实**
当前代码已经具备做分层评测的技术基础：loader 同时支持 legacy 与 versioned cases；metrics 已包含 `facts_accuracy`、`rule_hit_accuracy`、`attachment_visibility_accuracy`、`manual_review_flag_accuracy`、`hard_evidence_accuracy` 等；testing 文档保留了 `make eval-review`、`make eval-review-ablations`、`make eval-review-cross-pack`、`make eval-review-cross-model`；测试还明确断言了 versioned stage gate、diagnostics、ablation variants、cross-pack 与 cross-model。([GitHub][6])

**建议改造项**
V0.2 的评测不应再只盯最终 issue。应把现有指标组织成真正的 L0/L1/L2/L3 分层门：

| 层  | 现有可复用指标                                                                   | V0.2 应补强                                                             |
| -- | ------------------------------------------------------------------------- | -------------------------------------------------------------------- |
| L0 | `attachment_visibility_accuracy`、`manual_review_flag_accuracy`            | `visibility_reason_accuracy`、parser mode / parserLimited diagnostics |
| L1 | `hard_evidence_accuracy`、`high_severity_issue_recall`、`severity_accuracy` | official/experimental 分开阈值                                           |
| L2 | `facts_accuracy`、`rule_hit_accuracy`、`policy_ref_accuracy`                | applicable clause completeness、unresolvedFacts accuracy              |
| L3 | `issue_recall` + inferred-risk 命中                                         | remediation bucket consistency、suggestion-vs-defect separation       |

同时把 case bucket 分成三层：

* **Stable legacy baseline**：防回归。
* **Official versioned stage gate**：V0.2 主门槛。
* **Experimental versioned diagnostics**：不阻断合并，但必须出报告。

### E5. V0.2 反过拟合原则

这部分必须写成硬约束，而不是建议：

1. **不得为冷轧厂、培花两个施组写 case-specific patch。**
2. **不得把 Gemini wording 当目标。**
3. **不得把 Gemini 结果当专家 truth。**
4. **不得把少量样本分数当泛化能力。**
5. **不得把 visibility truth 与 issue truth 混写。**
6. **不得让 pack promotion 只依赖单一项目通过。**
7. **不得在 `rules/*`、`extractors/*`、`report/*` 中按项目名/文件名/fixture id 分支。**

这些原则与 `review_eval/README.md` 中“不得把 Gemini 结果直接当作专家 truth”“必须区分 issue truth 与 visibility truth”“不得把未解析附件直接标成缺失”的非协商原则完全一致，且比 V0.1 更应在当前仓库里落成代码与评测门槛。([GitHub][10])

## F. V0.2 研究结论与优先级排序

### F1. 最值得优先做的 5 个方向

| 优先级  | 方向                                                     | 为什么现在必须做                                      |
| ---- | ------------------------------------------------------ | --------------------------------------------- |
| P0-1 | L0 parser / visibility / manual-review 统一化             | 不先稳住可视域，L1/L2/L3 结论都不可靠                       |
| P0-2 | official scope 的规则核与证据核加固                              | 当前已经有 official 两类文档，最值钱的是先把这两类做硬              |
| P0-3 | versioned + layered eval stage gate 制度化                | 代码已有 loader/metrics/harness，不尽快制度化就会“能跑但不可控”  |
| P1-1 | support-scope / pack promotion / profile resolution 收口 | 当前 ready packs 与 official scope 已出现差异，需要治理真相源 |
| P1-2 | reviewer cockpit 与最小人工复核闭环                             | reviewer decision 已有 API 与模型，但 UI/流程还不够强      |

### F2. 如果只能先做 3 个方向

如果资源有限，只保留这 3 个：

1. **L0 可视域与 manual review contract 加固**
2. **official doc types 的 L1/L2 规则与 evidence 深化**
3. **分层评测 + official versioned stage gate**

这三项不做，系统就算继续扩包、扩模型、扩 UI，也不会真正接近 Gemini deepresearch 在真实施组上的审查深度。

### F3. 哪些方向后做

后做而不是不做：

* 多文档批处理
* OCR / 多模态 / 图纸平台化
* 广泛扩充 official doc types
* cross-model 文风优化
* 高级整改编排代理化

### F4. 资源有限时应该先砍掉什么

最先砍掉的是：

1. 多模态平台化；
2. 大范围新增 doc types；
3. prompt / model tuning 驱动的“像 Gemini 文风”工作；
4. 与 official scope 无关的 UI 美化。

### G4. Definition of Done

V0.2 本轮工作完成，必须同时满足以下 DoD：

   * `make test`
   * `make eval-review`
   * `make eval-review-ablations`
   * `make eval-review-cross-pack`
   * `make eval-review-cross-model`

**不得把“文风更像 Gemini”作为 DoD。**

## H. 风险、边界与开放问题

### H1. 本轮必须解决的

1. source ingress 与 local path reality 对齐。
2. L0 visibility / manual review 语义稳定。
3. official scope 规则核与证据核加固。
4. versioned stage gate 制度化。
5. reviewer decision 最小闭环。

### H2. 本轮暂不解决，但必须显式挂起的

1. 多文档批处理。
2. 外部对象存储与大规模上传资产治理。
3. OCR / 图纸 / 图像多模态平台化。
4. `strictMode` 的复杂语义扩展。
5. 大范围提升所有 doc types 为 official。

这些挂起项与当前仓库已写明的限制一致：当前 structured_review 只支持最小单文件输入，不支持多文档批处理或多模态 OCR 平台化链路。([GitHub][13])

### H3. 再往后阶段才处理的

1. 跨文档冲突推理与整套整改编排。
2. 图纸/图片/扫描件的结构化 parser 家族。
3. reviewer feedback 反哺 pack release governance。
4. 更复杂的 cross-model / cross-agent benchmarking。

1. **事实来源**
   先读 GitHub 当前主干，再读 `V0.1-分析2.md`。GitHub 为第一事实源，V0.1 只是第二事实源。若冲突，以 GitHub 为准。([GitHub][1]) 

2. **执行路径**
   你将在本地目录 **`/Users/lucas/repos/review/008-review-control-plane`** 执行修改。但在动手前，先用 GitHub 当前主干核对本地路径与文件位置，特别是：

   * `apps/api/src/repositories/sqlite_store.py`
   * `apps/api/tests/*`
     不要把用户提示里的旧路径直接当作本地真相。([GitHub][7])

3. **如何理解这份任务书**
   这不是“从零设计 formal review”的蓝图，而是“基于现有主链做深、做稳、做可验收”的施工说明。优先做 P0：contract spine、source ingress、L0 visibility、official packs/evidence、layered eval。P1 再做 support-scope promotion、reviewer cockpit、更多 versioned cases。

4. **哪些项必须先跑 tests / eval 再提交**

   * 任何 contract、parser、runtime、routes、store 改动：先跑 `make test-review-unit` 与 `make test-review-integration`。
   * 任何 rules/evidence/evaluation/fixtures 改动：至少跑 `make eval-review`。
   * 任何评测框架改动：再加 `make eval-review-ablations`、`make eval-review-cross-pack`、`make eval-review-cross-model`。
   * merge 前跑全量 `make test`。([GitHub][14])

5. **哪些项适合按小步 PR / commit 推进**

   * contract spine
   * source ingress
   * L0 parser/visibility
   * official rules/evidence
   * support-scope / pack promotion
   * layered eval
   * reviewer cockpit
   * docs consistency

6. **执行时的硬约束**

   * 不按冷轧厂/培花项目名、fixture id、文件名写分支；
   * 不把 Gemini 输出写进专家 truth；
   * 不以文风相似度作为验收；
   * 不在未通过 tests / eval 的情况下提交。
