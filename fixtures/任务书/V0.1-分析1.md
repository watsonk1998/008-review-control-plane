# 008-review-control-plane 产品架构设计书（当前态盘点 + 下一阶段演进版）

## A. 执行摘要（Executive Summary）

本设计书旨在为 008-review-control-plane 仓库的下一阶段持续改造与架构演进提供极具深度的、可执行的、严谨的工程蓝图。本报告的分析与设计严格基于当前真实存在的唯一有效代码事实源（即 https://github.com/watsonk1998/008-review-control-plane 仓库），已彻底废弃并删除的历史仓库及相关分支不再作为任何事实依据，所有状态评估、系统边界划定与未来架构演进均绝对锚定于当前仓库的代码文件、前后端实现以及现存的测试资产。用户所提供的历史背景材料（如原始施工组织设计文档、Gemini审查结果对比、旧版修复任务书及 Golden Pool 指导书）仅被作为历史需求背景、过往问题意识及产品目标约束的参照，绝不替代当前代码的客观事实。综合对当前系统的深度盘点表明，该项目正处于由“单轨辅助阅读”（review_assist）向“双轨制正式结构化审查”（structured_review）跨越的关键过渡形态。当前仓库绝非空白系统，其核心的 structured_review 概念、双轨制骨架以及部分数据传输对象（DTO）已在领域模型层（Domain Models）初步显现，前端与路由层也已初步建立任务分流的契约。然而，系统的深层运行机制（Runtime）仍存在严重的职责过载与逻辑混同现象。六层审查流水线（文档解析、事实抽取、规则命中、证据归档、大模型解释、报告组装）尚未在物理架构上实现彻底解耦，导致底层大语言模型（LLM）在处理诸如冷轧厂行车改造或培花调蓄池等复杂工程规范时，极易将系统解析能力的受限（可视域盲区，Visibility Gap）误判为工程文档实质性的合规缺陷（Defect）。这种将“系统没看到”等同于“文档没有”的错误逻辑，构成了当前最大的真实短板。本设计书的核心目标是明确指出：下一阶段的演进方向必须坚决反对单纯依赖更换更大底层模型、扩写冗长提示词（Prompt）或仅模仿外部系统文风的伪改进。相反，系统必须致力于做实已有骨架，将核心评审域（Review Domain）的确定性逻辑与通用编排运行时（Orchestrator）进行物理级解耦；稳定多模态与多维度的对象契约（Result Contract）；并构建防过拟合的稳定化分层评测资产（Golden Pool）。通过实施严格的 L0 至 L3 层级映射机制与状态机管理，本产品将最终蜕变为具备稳健的、可解释的、端到端的工程图纸与方案审查管控能力的智能控制平面（Control Plane）。

## B. 当前仓库真实现状（As-Is）

针对当前 008-review-control-plane 仓库内的具体实现状态，本节将深入各个关键目录与核心模块，进行逐文件的工程级盘点。能力评估将严格采用预设的成熟度标签，以区分“已实现”与“仅文档声明”之间的巨大工程鸿沟。

仓库顶层与架构文档体系目前呈现出“文档领先于实现”的典型特征。在检视 README.md、docs/architecture.md 以及 docs/formal-review.md 等核心文件时，可以发现项目在宏观层面上已经明确宣告了作为“审查控制平面”的定位。其核心理念是统筹底层模型能力与具体的建筑工程审查业务逻辑，并对 formal review 的结构化属性提出了极高的期待。然而，文档中所描述的严密审查约束与多层级控制机制，在代码侧更多体现为框架性宣告。docs/testing.md 及 docs/known-limitations.md 等文档的存在，印证了开发团队对当前系统处理长文本和复杂多模态附件极限的清醒认知，这为后续的架构重构提供了切实的痛点输入。

在后端 API、领域模型与编排运行层，系统的状态表现出明显的层级差异。检视 apps/api/src/domain/models.py，可以确认双轨制的核心枚举与结构已经落地，表现为**仅 DTO / schema 存在（DTO only）或初步可用（minimally working）的状态。TaskType 明确区分了 review_assist 与 structured_review 两种任务。然而，模型层对审查细节状态的下钻深度严重不足，类似 EvidenceSpan（物理坐标证据链）或精细化的 IssueCandidate 对象尚未完全成型，无法完美支撑后续严谨的规范比对。

在路由与控制器层面（apps/api/src/routes/tasks.py），任务下发链路畅通，API 能够接受不同类型的审查指令。

编排器（apps/api/src/orchestrator/planner.py 与 router.py）已经实现了有骨架（scaffold only）**的分流机制。针对辅助阅读任务，规划与调用链路相对成熟；但在处理结构化审查时，规划器的动态调度能力单薄，尚未能基于施组类型动态挂载专用的政策包（Policy Packs）。更为关键的是，在 apps/api/src/orchestrator/deepresearch_runtime.py 中，运行时（Runtime）仍在越权承担本应下沉至 Review 子域的“规范比对”与“逻辑推理”职责，导致审查过程过度依赖泛化的文本研究链路，而非基于结构化事实的确定性比对。

适配器层（Adapters）主要集中在 apps/api/src/adapters/ 目录下，包含 llm_gateway.py、gpt_researcher_adapter.py 以及 fastgpt_adapter.py 等组件。这些模块目前处于**初步可用（minimally working）**状态，表明系统已具备多模型切换和调用基础大语言模型进行检索、归纳的底层通讯能力。但当前的适配器高度倾向于通用自然语言文本的输入输出契约，缺乏针对强制性工程规范树进行定制化序列化与反序列化的深度适配逻辑，导致其在承接高维结构化数据时表现出明显的序列化瓶颈。

评审子域（Review Domain）作为控制平面的核心脏器，其在 apps/api/src/review/ 目录下的落地状态参差不齐。

解析与抽取模块（Parser / Extractors）目前初步可用（minimally working），其主要依赖于 apps/api/src/services/document_loader.py 的基础能力，能够提取线性的基础文本与章节标题。但在面对施工组织设计中常见的复杂图表、CAD图纸元数据、起重吊装参数表等高密度结构化信息时，其深度抽取（Facts Extraction）极易失效，且缺乏对“解析失败”这一状态的显式捕获与上报机制。

规则与证据模块（Rules / Evidence / Pack Registry）大体上仅有骨架（scaffold only），代码中可能存在规则引擎的初步接口定义，但尚未形成标准化的、可动态加载的知识包注册中心。

证据归档（Evidence Linking）当前极度匮乏物理坐标级别的绑定，其在生成报告时引用的“证据”，更多是依赖大模型的自然语言注意力机制生成的复述，而非不可篡改的 [document_id: page_num: bounding_box] 硬链接。

报告构建模块（Report Builder）同样处于**初步可用（minimally working）**阶段，其过度侧重于生成文风通顺的长篇 Markdown 总结，而未能将问题矩阵（Matrices）和原始证据链作为第一公民进行严格的数据对象隔离输出。

前端表现层与交互契约方面，apps/web/src/types/control-plane.ts 中已存在前后端交互的基础类型定义。apps/web/src/components/home-dashboard.tsx 和 task-detail.tsx 能够渲染基础的任务列表与详情。可以判定，前端已具备区分 review_assist 和 structured_review 任务结果的基础渲染能力，状态为初步可用（minimally working）。然而，前端尚未形成面向人工复核（Manual Review）的专属交互工作流，也没有实现基于 EvidenceSpan 的原文锚点高亮与图纸双向联动。制品（Artifact）的展示仍停留在 Markdown 文本流的降级消费层面。

在评测基础设施与数据基线层面，fixtures/review_eval/ 及相关目录初步存在，其中包含复制过来的施组样本（冷轧厂、培花调蓄池等）以及部分审查结果文档。这表明系统已经建立起了**仅测试口径存在（test-only evidence）**的最基础对比基线。当前的评测资产属于初始的 Bootstrap 或 Seed 状态，评估维度高度单一，主要依靠最终生成的长文本结果进行人工或半自动对比。针对事实抽取召回率、规则命中准确度、以及可视域判断正确率的模块化消融实验与细粒度回归测试系统（Evaluation Harness）尚未落地（not landed），这极大阻碍了系统后续防劣化迭代的步伐。

## E. 当前能力边界（Capability Boundary）

当前系统面临的深水区挑战，并非单一功能的缺失，而是核心系统边界的严重渗透与职责越权。若不廓清这些边界，系统规模的扩大只会导致混乱指数的级联上升。下一阶段的架构必须围绕以下四组核心边界进行铁腕式的清理与巩固：第一，review_assist 与 structured_review 的认知与执行边界必须彻底固化。当前系统中，辅助审查凭借其强大的开放域上下文抓取能力和容错性，在潜移默化中干扰了结构化审查的执行逻辑，导致后者在输出结果时依然带有“泛总结”的冗余特征。下一阶段，review_assist 负责统包上下文检索、长文摘要与交互式解答，其产出容忍一定的发散性。而 structured_review 必须被收敛为一个基于确定性契约的有限状态机引擎，它只负责输出“合规”、“违规”、“缺项”、“不适用”或“需人工复查”这五种刚性结论，剥离一切多余的修辞与总结。第二，Orchestrator/Runtime 与 Review Domain 的物理边界必须彻底阻断。目前，deepresearch_runtime.py 过度干预了审查细节，将具体的政策包解析、条款比对与文本循环研究等业务逻辑混杂在通用编排链路中。目标边界要求：运行时（Runtime）仅仅负责 DAG 任务节点的流转调度、API 重试、速率限制与持久化存储；一切与建筑工程规范、施组业务理解、缺陷判定相关的心智模型，必须绝对下沉并封装入 apps/api/src/review/ 子域中。通用引擎不应懂得何为“深基坑支护规范”。第三，后端结果契约与前端 UI 渲染的消费边界亟待规范。目前前端的任务详情页高度依赖后端拼装完成的长篇 Markdown 报告，导致报告生成器（Report Builder）耦合了过多面向人类阅读的排版逻辑。未来的边界在于，后端 Review Domain 只需向外输出严格类型化的 Artifacts 集合（例如标准化的 IssueList 和 EvidenceMap JSON 对象），而如何将这些底层结构化数据呈现为便于人工复核的缺陷树、工况验证单或对比矩阵，则全权由前端组件库依据契约进行动态渲染。第四，评测体系（Evaluation）与主干生产代码的边界必须保持洁净。评测数据资产（Fixtures / Eval Assets）的加载与评估逻辑不能反向侵入主干运行库。Golden Pool 评测系统应当作为一个独立运行的外部旁路观察者，仅仅通过标准化接口消费系统的结构化输出，从而实施防劣化断言。

## F. 关键问题与真实风险

摒弃泛泛而谈的系统风险列表，基于对本仓库具体落地的深度剖析，下一阶段的演进过程中存在以下极其致命的真实工程风险：首先是可视域误判引发的信任崩塌风险（Visibility False Positives）。这也是当前最大的短板所在。当解析器（Parser）处理一份附带有非标准 OLE 嵌入式起重机负载图表的 PDF 时，因能力所限未能抽取出有效数据。此时，缺乏防御机制的 LLM 会错误地越权判定：“方案中未见起重机负载表，违反特种设备安全规范，存在重大安全隐患”。这种将系统级的“解析盲区”偷换为业务级的“工程缺陷”的行为，会瞬间摧毁资深审查工程师对系统的信任。其次是模型越权判断与逻辑幻觉风险（LLM Overstepping）。在没有严格的规则命中层（Rule Matching Layer）作为卡口的情况下，底层大语言模型倾向于凭借预训练语料中的常识自行发明并不存在于项目所在地规范中的“伪强制条文”；或者将“建议采用某型号钢筋以提升施工效率”的工程优化建议，伪装成“不合规导致结构风险”的刚性缺陷。再次是单体报告过载与人工复核失效风险（Report Over-summarization）。当所有审查结果被扁平化地糅合进一份数万字的长篇 Markdown 文档进行交付时，底层 Issue 的结构化属性被彻底抹平。如果审查结论出现争议，工程师无法在 UI 界面上通过简单的交互动作对单个缺陷进行“忽略（Ignore）”、“采纳（Accept）”或“发回重审（Re-assign）”，导致系统设计的“人机闭环”完全破产。最后是评测基线脆性引发的静默劣化风险（Regression Brittleness）。当前 fixtures/ 目录下的样本数据多被用于整体结果的粗颗粒度定性对比。若不建立对“抽取环节”和“规则匹配环节”的独立断言，一旦为了修复某个冷轧厂方案的解析 Bug 而微调了全局的 Prompt 或文档分割策略，极有可能引发全局范围内其他行业类别（如培花土建工程）审查结论的剧烈漂移，而由于缺乏微观评测，这种系统性的退化将不可被及时观测。此外，Legacy review_assist 的心智模型持续在开发团队中作祟，导致新代码常常不自觉地向其靠拢，也是一种长期的认知风险。

## G. 目标产品定位（To-Be Product Positioning）

基于产品能力边界的梳理与用户预期管理的双重视角，下一阶段对 008-review-control-plane 必须确立极其清晰和强硬的目标定位。本产品绝不是一个披着建筑外衣的“通用工程文档聊天机器人”，也不是一个依赖无限上下文（Context Window）技术暴力破解难题的“多模态文本摘要器”。本产品必须被定义为一个面向严谨工程图纸与复杂技术方案的智能合规审查控制平面（Intelligent Review Control Plane）。其核心业务模式必须坚守不可妥协的双轨制（Dual-Track System）：

#### 第一轨：辅助审查助手（review_assist）

保留其作为“伴随式工程知识副驾驶”的定位。它的存在是为了降低人类工程师熟悉陌生大型项目（如冷轧厂庞杂电控改造背景）的时间成本。它应当提供文档结构梳理、前置上下文检索、术语解释及自由的启发式问答。对于这一轨道，系统应宽容其适度的发散性推理，甚至鼓励其提炼出具有前瞻性的非强制性工程建议。

#### 第二轨：正式结构化审查（structured_review）

这才是控制平面的真正心脏，其定位必须是一个“零信任的、冷酷的自动化合规审计员”。它不再阅读文档并“谈论”文档，而是严格执行冰冷的对照检查算法。所有进入该轨道的任务，必须强制通过预设的多层流水线，基于挂载的法定规范体系（Policy Packs），输出带有绝对物理坐标溯源（刚性证据链）的缺陷矩阵。这种双轨划分彻底契合了 Control Plane 的长期宏图：既提供灵活的数据面洞察力（辅助助手），又维持坚不可摧的控制面合规防火墙（结构审查）。只有将这两条轨道的内部逻辑做到物理级别的泾渭分明，系统才能在承载海量高风险工业级验证时屹立不倒。

## H. 目标架构设计（To-Be Architecture）

为了从底层逻辑上支撑上述产品定位，目标架构必须摆脱“通过一个巨型 Prompt 试图解决所有问题”的稚嫩阶段，对现有的编排器与 Review 子域进行深度改造与彻底做实。架构重构的宗旨是：解耦非确定性大模型调用，建立纯正的六层确定性流水线。

### 1. 运行态与领域态的彻底解耦

运行时（Orchestrator/Runtime）保留并解耦：现有的 planner.py 和 router.py 将继续承担基础设施调度任务，负责任务的分发、状态机维护、外部 API 的防抖重试、异步队列管理以及数据落盘。但必须将其彻底剥离出任何与特定业务规范比对相关的 Prompt 生成逻辑。评审域（Review Domain）全面做实：将原本由 Runtime 越俎代庖的分析逻辑，强制下沉到此。在此建立一套高度内聚的 Python 对象层，独占处理领域业务。

### 2. 构建严密的六层审查流水线 (Six-Layer Pipeline)

下一阶段，structured_review 的执行必须被固化为一段不可逆转的有向无环图（DAG），包含以下六个边界分明的层级：

#### 层一：文档多模态解析层（Document Parsing Layer）：

依托 document_loader.py，不再仅输出文本串，而是输出带有类型标记的抽象语法片段树（含段落、表格、图元）。必须在此层完成 L0 可视域盲区测算并附带置信度打分。

#### 层二：客观事实抽取层（Facts Extraction Layer）：

调用 LLM 但严格限制其 Prompt 行为空间，命令其仅根据特定的 Fact Schema，从繁杂的片段树中“生硬地”提取诸如“最大开挖深度：5.4米”、“混凝土强度：C40”等冷冰冰的工程事实。本层必须做到数据提取与价值判断绝对剥离。

#### 层三：规范规则命中层（Rule Matching Layer）：

在此层引入确定性代码逻辑（或极轻量级的微调分类器）。将上游萃取的客观事实，与预先从注册中心加载的对应工程类型的政策知识包（Policy Packs，如国标危险性较大的分部分项工程安全管理规定）进行交叉碰撞，计算事实参数是否突破了规范的硬性阈值。

#### 层四：硬关联证据归档层（Evidence Linking Layer）：

针对命中的疑似缺陷项，系统必须反向追溯该事实在初始解析树中的精确物理坐标（如 Document A, Page 17, Table BBox），锁定并构建出不可篡改的证据链条，封堵大模型“无中生有”的幻觉通道。

#### 层五：大语言模型解释与推演层（LLM Explanation Layer）：

仅仅当上述四层工作完成，即（客观事实 + 命中规范 + 坚实证据）三位一体就绪时，才允许调用全量能的大语言模型。此时它的任务被极度受限为：基于既定结论，生成符合人类工程师审阅习惯的专业整改建议术语与技术推演上下文。

#### 层六：结构化制品组装层（Report Assembly Layer）：

废弃直接输出庞大 Markdown 的旧习。此层负责将所有产生的 FinalIssue 对象聚合为一个包含缺陷矩阵、验证逻辑单及关联证据图谱的结构化 JSON Artifact，以契约的形式推送给前端 UI 消费端。通过上述流水线的强制执行，我们不仅收敛了模型的发散权，也极大地提升了系统的可验证性与可解释性。

## J. 六层流水线与 L0/L1/L2/L3 的映射分析

在业务认知与工程架构的对接面上，必须明确物理的“六层流水线”与审查深度视角的“L0-L3 模型”之间严丝合缝的映射关系。当前 Repo 在概念上虽然可能隐约包含了这些层级，但在代码执行时却常常发生灾难性的逻辑混杂。

#### L0 可视域检查基础底座 (Visibility Base)

- **映射的物理流水线：** 文档解析层 (Document Parsing Layer)。
- **当前 Repo 状态：** 严重缺失，完全没有将系统解析能力的下限视为审查能否成立的前提。
- **演进设计指引：** 在文档切分完毕后，系统需诚实、显式地暴露出所有的盲区（如乱码表格、非标图形）。将其截获并抛弃在 L0 层，能从物理根源上彻底斩断后续 LLM 产生“文档未见该图纸，存在隐患”这一最普遍的幻觉逻辑。

#### L1 硬证据与强条阻断 (Hard Evidence & Mandatory Rules)

- **映射的物理流水线：** 事实抽取层 + 规则命中层。
- **当前 Repo 状态：** 过度依赖单一大模型的隐藏常识，且两步操作未分离，混杂在通用提示词内隐式执行。
- **演进设计指引：** 这一层的核心是冷酷无情的数据碰撞。例如对于“起重机额定载重必须大于预估最大起重量”这样的 L1 硬性规则，只要前置层抽取出了对应数值变量并发现在算法层面不满足条件，即可瞬间触发违规阻断，无需复杂的推理。

#### L2 条文交叉适用与规范差距推演 (Clause Cross-Application)

- **映射的物理流水线：** 规则命中层 + 证据归档层。
- **当前 Repo 状态：** 尚未实现多政策包（Policy Packs）冲突的消解策略，对局部地标与国家强标覆盖范围的敏感度不足。
- **演进设计指引：** 当提取的事实遭遇规范冲突或适用性模糊时，必须根据证据的地理与行业元信息，激活优先度评估引擎，并在 EvidenceSpan 中固化每一处裁定关联的原文段落。

#### L3 复杂工程综合推理与整改编排 (Complex Reasoning & Rectification)

- **映射的物理流水线：** LLM 解释层 + 报告组装层。

## M. 评测与 golden pool 演进方案（Evaluation & Golden Pool Roadmap）

在工程级的架构重构过程中，评测体系犹如系统的神经感知网络。没有强大的评测大坝拦截，任何架构与 Prompt 的优化都无异于在沙盒上建塔。必须将现有的、基于结果阅读判断的材料，彻底升维为基于科学度量的 Golden Pool。

### 

### 1. 

当前评测资产的真实状态与局限性

从 fixtures/review_eval/ 目录存放的内容推测，当前的评测资产仍处于从 Seed Cases 向 Mini Golden Cases 过渡的初级混杂期。其最为致命的问题在于评测维度的“唯结果论”：团队过度依赖肉眼审视系统最终输出的 Markdown 报告，评估重点失焦于“文章通顺否”、“格式像不像 Gemini 的风格”。由于严重缺失针对 L0 可视域失败率、L1 事实漏抽率以及 L2 规则错杀率的细分评测逻辑，当前的 Harness 根本无法指导工程师开展模块级（如仅针对 extractors）的代码消融与验证实验，这也意味着在多模型切换与对比时缺乏核心的量化标尺。

### 2. 评测资产下一步的核心演进方案

架构演进要求评测机制必须向纵深挺进，实现由宏观至微观的断言革命：端到端分层断言化 (Layered Micro-Assertions)：废除对文字的感性评阅，将人工标定的施组样本转化为严格的 Ground Truth JSON。Golden Pool 中的每一个 case 都应明确包含三层期望：应抛出的 visibility 盲区警告清单、应抽取的关键客观参数（Facts）、以及最终应阻断的合规条文 ID。评测机制计算每一层级的 F1 Score，从而精确定位性能瓶颈是源于解析崩溃、抽取疏忽，还是知识库残缺。跨模态与跨配置消融对照 (Cross-Model & Cross-Pack Validation)：借助完善的脚本（harness），支持在更换底层 LLM Gateway（如 OpenAI 换为国产大模型）或动态移除某项 Policy Pack 后的自动化回放，观察系统对确定性流水线逻辑的保持能力，防范底层模型能力塌陷带来的静默灾难。强硬回归阻断基线 (Regression Baseline)：Golden Pool 应实施严格的版本化控制，它构成了代码合入（PR）的生死线。若某次微调为了提升对“冷轧厂机电图”的识别，而意外导致回归测试集中“培花调蓄池”用例的关键缺陷丢失召回，必须立即阻断发布操作。

### 3. 泛化样本覆盖与终极防过拟合要求

为了避免系统彻底沦为某个具体项目样本的“特调工具”，Golden Pool 的测试用例库必须横向扩张。它至少应当全面覆盖包括传统建筑土建类（高支模、深基坑）、机电冷设备安装类、钢结构吊装类及监理规划辅助等多种异构工程体裁。**防过拟合专项禁令：**严禁在 Ground Truth 制定中发生方向性妥协。特别强调防范“假阴性污染（False Negative Pollution）”。即，如果测试中系统因解析失败未能识别到施工图中的某个参数而默认其“合规通过”，绝不可将此错误结果通过“更新基线”的方式反向录入金库成为真理。评测人员必须核实原始图纸意图，强制系统必须输出针对该部位的 VisibilityGap 判断，逼迫底层算法迎难而上而非掩耳盗铃。

## N. 非目标与禁止事项（Non-goals & Anti-patterns）

为了保证极其宝贵的工程研发资源不被投放入无底的伪需求黑洞，亦或是陷入技术自嗨，本架构方案特此划定不可逾越的“非目标与工程反模式（Anti-patterns）”红线区：绝不单凭“换大模型”来掩盖工程设计缺陷：将本应通过严密六层流水线与知识包约束解决的业务复杂性，盲目寄望于下一代超大上下文模型的零样本能力涌现，是对工程管控极其不负责任的投机主义行为。绝不把系统“未看清”包装成图纸“未涉及”：这是产品底线的死循环。不允许系统为了虚增查处的“重大问题”数量，而掩盖底层解析器的孱弱无能；遭遇处理失败的附录，系统必须诚实抛出业务异常供人工介入。禁止对单一评测样本进行“打补丁式”硬编码过度拟合：严防死守在核心评审域内为了取悦对“某特定冷轧厂工程”的测试结果，直接写入缺乏普遍性泛化意义的强行正则表达式捕获。防范并杜绝将“工程建设性建议”强行升格包装为“致命性强条缺陷”：基于挂载的规范库，系统必须严守界限。不得将大模型提出的“该处混凝土标号可提升以利于快速施工”越界判定为足以阻断审核的致命违反国标行为。不盲目追求模仿竞品系统（如 Gemini）行文风格：合规审查是一项严肃的结构化审计过程。其评判的核心在于缺陷检出的客观准确度与溯源证据的清晰度，而非追求产出一份排版华丽、具有漂亮引导散文但实际内核逻辑松散的 Markdown 博客。绝不试图把单轨的 review_assist 换个壳强推为正式控制平台：不得心存侥幸，企图仅仅通过优化助手生成文本的章节标题样式，便草率对外宣称其已进化为能够承担严重安全责任的 formal review 系统。两者的底层基因与数据流转状态机必须存在物理级别的隔绝。

## O. Definition of Done (DoD)

本架构设计书下一阶段演进（涵盖 P0 与部分 P1 目标）的完成判断，必须严格绑定以下维度的刚性验收指标（DoD）：架构底座解耦 DoD：review_assist 与 structured_review 的 API 下游流转路由，必须在物理代码库（apps/api/src/）级别彻底剪断藕丝，不存在任何非预期的跨管道共享提示词或推理链复用。核心漏洞拦截 DoD：L0 层级的 AttachmentVisibility 异常捕获代码实质上线，且被成功应用于至少三个导致崩溃的极端异形文档用例，系统确凿证明其已具备拦截并正确标识“可视域盲区”的安全兜底能力。结果契约与前端对齐 DoD：完成正式审查任务执行后，Review Domain 必须且只允许向外界下发强类型验证的 JSON Artifacts 对象，彻底取缔原始的长文本混编输出。前端则需原生提供基于此数据契约解析生成的、具备点对点追踪能力的缺陷交互矩阵看板。溯源证据完整性 DoD：系统生成的每一张“罚单”（FinalIssue 对象），其中至少 80% 能够在其内部所携带的 EvidenceSpan 字段中解析出准确指向原文档对应页数及坐标块的信息，且此信息被证实具备人工校准价值。基线评测守护 DoD：根据现有散装材料完成构建的首个系统化版本 Stable Golden Pool v1.0。任何被拆分、重构的新流水线代码在合并入主干前，必须在测试套件中无须人工干预跑通全流程断言，并且证明其各项客观抽取与匹配指标未曾滑落低于上一轮重构前的历史底线。

## 结论 (Conclusion)

基于对 008-review-control-plane 真实代码基础及其承载历史诉求的残酷剥洋葱式拆解，现向核心干系人针对系统的终局状态与后续抉择作出不加掩饰的清晰答复：

### 1. 

当前 repo 到底处于什么真实的阶段？

它正处于由粗放的“长篇大论生成器”向具有精准约束属性的“严谨工程管控系统”破茧的艰难挣扎期。在外部接口层面（DTO/API路由），双轨制的分流门头已经高高挂起；但在后端执行暗箱内，却依然难以戒断将大语言模型供奉为无所不知的“超级全能王”并让其包揽抽取、匹配、下判决全部职能的危险路径依赖。

### 2. 

当前 repo 最真实的竞争优势与底牌是什么？系统具备了极其符合现代云原生与大语言模型协同操作理念的基础设施架构。规划器（Planner）、编排路由网络（Router）与多元大语言模型通信网关（Adapters）均已严阵以待。更难能可贵的是，开发团队具备极其罕见而透彻的问题意识，对于现有系统对于复杂附件处理的物理极限边界及其评测资产短板存在着理智的共识。

### 3. 

当前 repo 最真实的致命短板在哪里？评审核心业务控制层（Review Domain）极度空心化，系统对大模型缺乏缰绳。从前端文档进料到最终的“宣判”环节，由于缺失确定性的程序阻断卡口（例如最基础的对“我看不清附图”这一异常状态的捕获防御），大模型肆意越权滥判，极易摧毁基于专业认知的系统公信力。

### 4. 下一阶段最值得重兵投入的 3～5 件事是什么？暴力拆解并物理隔离双轨制 Runtime（将严肃的结构化审查动作彻底抽离出泛泛文本研读的研究网）。强制树立六层串行阻断流水线（引入确定性的事实客观抽取机制与可视域容灾底线拦截网）。全面升级并固化多维数据契约（全盘推行附带时空坐标信息的结构化缺陷对象实体）。锻造多维度分层验证的护城河基线（全面抛弃只看文章优劣的感性评鉴，升级为基于字段级强制断言的自动化 Golden Pool）。
