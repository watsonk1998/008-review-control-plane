## 2. V0.3 实施目标与非目标

### 2.1 本轮实施目标

V0.3 的中心目标不变：

> **不是扩范围版本，而是收边界、硬内核、补证据、强评测的版本。**

本轮要做硬的能力，按 formal-review spine 对应如下：

- **可审前置**  
  把 L0 preflight / pre-review checklist / parser-limited gate 前置成正式入口纪律，而不是把 parser 受限影响留到报告末端。

- **可视域诚实**  
  把 `visibility`、`attachment state taxonomy`、`manualReviewNeeded`、`manualReviewReason` 做成一致、可回放、可评测的单一事实源。

- **证据可追溯**  
  把 `facts -> rule hits -> clause applicability -> evidence gap -> unresolvedFacts` 串成可追踪链路，避免 issue 只剩 prose。

- **规则可命中**  
  在 official 两类文档内继续做硬 rule/evidence core，但不扩 doc type。

- **结果可复核**  
  把 reviewer decision、issue/attachment/task 三层承接，以及 internal-reviewed preparation 接上治理链。

- **评测可闭环**  
  让 layered metrics、official stage gate、diagnostics / replay 各归其位，防止 snapshot 被误当长期 truth。

### 2.2 本轮非目标

以下明确不进入 V0.3 主线：

- 全文档类型覆盖
- 扩 official scope 到 `construction_scheme` / `supervision_plan` / `review_support_material`
- OCR / 多模态 / 图纸平台化
- 多文档联合审查
- ontology / graph-first 重构
- 企业级 workflow / SaaS 壳建设
- “更像 Gemini”的报告篇幅、文风、措辞追逐

原因很简单：这些方向会稀释本轮对 **L0 gate、L2 closure、reviewer/eval governance** 的硬化。

## 3. 实施总策略

### 3.1 总体方法

本轮不是外延扩张，而是 **边界内加固**。实施顺序必须围绕 **L0 + L2**：

1. **先固 L0**：先解决“系统到底看到了什么、没看到什么、是否必须人工复核”
2. **再固 L2**：再解决“为何命中、何处 blocked、缺什么证据、为何 unresolved”
3. **最后接治理**：把 reviewer gate、internal-reviewed preparation、eval gate 串起来

### 3.2 为什么顺序必须围绕 L0 + L2

- L0 不硬，L1/L2/L3 都会带着假的确定性往下游流
- L2 不硬，formal-review spine 只会留下命中结果，无法解释 **适用性 / 缺证 / 阻断**
- 当前仓库的主弱点也正是这两端：
  - sample B 仍显示 `pdf_text_only` 下 top-level visibility manual review 未完全前置
  - latest eval snapshot 虽过 gate，但 aggregate `attachment_visibility_accuracy` 和 `hard_evidence_accuracy` 仍偏软

### 3.3 official / experimental / diagnostics 的落位规则

- **official mainline**
  - 仅 `construction_org`
  - 仅 `hazardous_special_scheme`
  - 仅与其直接相关的 gate / contract / eval 强化

- **experimental / diagnostics**
  - `construction_scheme`
  - `supervision_plan`
  - `review_support_material`
  - cross-pack / ablation / cross-model
  - 非 blocking 的 scenario hardening

- **internal-reviewed preparation**
  - reviewer notes
  - truth-layering 承接
  - seed -> reviewed 的过渡资产
  - 不得冒充 reviewed truth

- 不做完整审批流

**M4：Eval / Diagnostics 收口**
- 完成 layered metrics 对齐
- 完成 official stage gate / diagnostics 分层稳定
- 完成 regression / replay 命令验收

### 8.2 PR 拆分原则

- **PR1 只做 L0**
- **PR2 只做 facts/rules/evidence closure**
- **PR3 只做 reviewer gate / reviewed-preparation**
- **PR4 只做 eval / diagnostics / docs 同步**

禁止出现的大 PR 混锅：

- 边界声明 + 实施 + 代码 + 样本裁决混在一起
- 扩 doc type 与收口 L0/L2 混在一起
- UI 壳改造与 formal-review core 混在一起

## 9. 验收标准与退出条件

### 9.1 本轮完成标准

#### 产品上完成
- V0.3 仍严格停留在 official scope

### 9.2 本轮退出条件

出现以下任一情况，应立即降级、延后或退出本轮：

- 开始扩 official doc type
- 开始引入 OCR / 多模态 / 图纸 / 多文档

- 开始把报告 prose 优化当主线
- 开始把 Gemini seed 当裁决 authority
- 某项增强无法进入 official mainline，只能放 diagnostics / internal-reviewed preparation，却仍被包装成 P0 成果

### 9.3 关键验收场景

- **场景 A：DOCX + 附件标题可见正文未见**  
  top-level visibility 与 attachment matrix 同步触发 manual review；issue 落 `visibility_gap`，不写 missing
- **场景 B：PDF text-only + attachment matrix 为空**  
  L0 仍能前置表达 parser-limited / visibility-limited；后续 blocked-state 与 unresolvedFacts 一致
- **场景 C：高风险场景已识别但专项方案仅 generic mention**  
  结果应落 `partial` / manual confirmation，而非强行 hard defect
- **场景 D：应急标题缺失但 parser 受限**  
  应落 `evidence_gap` / `blocked_by_missing_fact`，不伪装为正文绝对缺失
- **场景 E：reviewer decision**  
  task / issue / attachment 三层状态与 artifact/UI 同步
- **场景 F：eval**  
  official gate 通过；diagnostics 不越权

## 10. 风险与反过拟合控制

### 10.1 主要风险

- **样本级 patch 风险**  
  围绕冷轧厂/培花词面、特定吨位、特定页数、特定章节标题做特判

- **prose chase 风险**  
  用报告更像 Gemini 掩盖 evidence closure 没变硬

- **ready pack 误升格风险**  
  把 ready 场景 pack 或 experimental base pack 偷渡成 official support

- **latest eval snapshot 误读风险**  
  把 `/Users/lucas/repos/review/008-review-control-plane/fixtures/research_inputs/latest-eval-summary.md` 当长期 truth

- **Gemini seed 污染治理链风险**  
  把 seed labels、Gemini 候选、prompt wording 反灌为 adjudication 或规则答案库

- **外部项目借鉴过度风险**  
  把 OpenContracts / AEC-Bench / RAPID 借成平台壳、benchmark 壳、workflow 壳

### 10.2 控制手段

- 所有 P0 变更必须能用“机制增强”描述，不能用“样本修复”描述
- 所有新字段优先进入现有 canonical artifact，不造平行 truth source
- 所有 experimental / diagnostics 输出不得进入 blocking success claim
- 所有 reviewer / reviewed-preparation 输出必须显式 provenance
- 所有外部借鉴都必须落成 008 的具体改造项，而不是架构迁移冲动

## 11. 最终执行结论

V0.3 这一轮，**最该做的** 是：

- 把 L0 preflight / visibility / manual-review 前置做硬
- 把 fact -> rule -> applicability -> evidence gap -> unresolvedFacts 做成可回放闭环
- 把 reviewer gate / reviewed-preparation / eval gate 串成最小治理承接层
- 把 official 两类文档内的 formal-review spine 做成可信主链

V0.3 这一轮，**最不该做的** 是：

- 扩 doc type
- 追 Gemini prose
- 上 OCR / 多模态 / 图纸 / 多文档
- 做 ontology / graph-first
- 做 workflow 壳

这份任务书对上游的承接关系是清楚的：

- 它不重新争论产品身份
- 不重新争论 official scope
- 不重新争论 Gemini 地位
- 不重新发明边界
- 只把 `/Users/lucas/repos/review/008-review-control-plane/docs/product-strategy.md` 与两份治理文档已确认的结论，翻译成工程主线

它对后续 PR / 实施阶段的约束也清楚：

- 任何实现都必须先问：是不是在加固 L0/L2/gate，而不是在做样本 patch 或 prose chase
- 任何新增能力都必须先落 official mainline / diagnostics / internal-reviewed-preparation 之一
- 任何超出本文边界的方向，都应另立版本，不得偷渡进 V0.3
