# 《三角对比研究结果》

## 0. 文档定位

本文不是治理裁决文档，不是实施任务书，不是 internal-reviewed truth，也不是 patch list。按仓库中的流程指南，它应作为后续《V0.3 边界声明》《V0.2→V0.3 差距裁决与反过拟合约束清单》《双样本人工复核裁决稿》共同依赖的第一份上游研究底稿；任务书也明确要求先做三角对比研究，再进入后续治理文档。([GitHub][1])

更具体地说，本文的目标不是判断哪份报告“更像专家”，而是回答四件事：V0.2 相对 V0.0 是否是实质升级；V0.2 相对 Gemini 的差距主要在哪一层；当前 V0.2 对设计目标兑现到什么程度；这些发现如何转译为后续治理输入，而不是转译为双样本专用补丁。([GitHub][2])

## 1. 证据范围与研究边界

本文按四层证据执行：第 1 层使用仓库边界真相源，即 README、`docs/formal-review.md`、`fixtures/review_eval/README.md`；第 2 层使用 `fixtures/research_inputs/` 的样本级结构化工件；第 3 层把 `fixtures/supervision/` 中六份 V0.0/V0.2/Gemini Markdown 结果作为人类可读呈现层；第 4 层把 Gemini 仅当作 seed / candidate reference，而不是 truth。除这四层外，`fixtures/任务书/使用说明（给研究型 AI）.md` 与 `V0.3 前置工作流程指南——文档生成顺序与依赖关系.md` 在本文中被当作“研究方法约束”，不是样本真相源。([GitHub][3])

仓库边界层面，README 已把项目定位为 review control plane，而不是单一 formal reviewer；`structured_review` 主链是 `parse -> facts -> rules -> evidence -> report`，并与 `review_assist` 双轨并存。README 还把稳定 DTO 明确到 `summary / visibility / resolvedProfile / issues / matrices / artifactIndex / reportMarkdown / unresolvedFacts`，并把 `manualReviewNeeded`、top-level `visibility`、`issueKind`、`applicabilityState` 设为 canonical 语义。同时，P0 official documentType 仍只包括 `construction_org` 与 `hazardous_special_scheme`；`construction_scheme`、`supervision_plan`、`review_support_material` 仍是 experimental。([GitHub][3])

`fixtures/review_eval/README.md` 则提供了 truth 语义的硬边界：Gemini 不能直接当专家 truth；必须区分 issue truth 与 visibility truth；必须记录 provenance；“未解析附件 / 当前不可视”不能直接写成“缺失”。这决定了本文可以把 Gemini 当差距探针，但不能把 Gemini 当仲裁者。([GitHub][4])

research pack 层面，任务书把 `fixtures/research_inputs/` 描述为主输入，并说明其包含 manifest、双样本目录、eval 子目录与 `latest-eval-summary.md`。同时，manifest 指向了更完整的 `artifacts/research-pack/...` 运行产物。当前 GitHub 目录页中，`sample-a-cold-rolling/` 与 `sample-b-puhua-rainwater/` 直接暴露的是 9 个精选结构化文件；manifest 则说明完整 pack 还包含 report buckets / report markdown 等其他工件。因此，本文的“主证据”是 GitHub 中可直接读取的这 9 个 JSON / matrix 文件，manifest 与 `latest-eval-summary.md` 只作为辅助说明。([GitHub][2])

第 3 层 supervision 中，仓库同时保留了两份真实源文档（冷轧厂 DOCX、培花 PDF）以及各自对应的 V0.0、V0.2、Gemini Markdown 结果。这使得本文可以做真正的三角对比，但 supervision 结果本身仍只是呈现层，不能覆盖第 2 层结构化工件的证据优先级。([GitHub][5])

`latest-eval-summary.md` 只能当阶段性快照。它自己就写明：这是本地某次 `make eval-review` 等命令生成的阶段性评测快照，不等于长期仓库 truth，不替代 `fixtures/review_eval/README.md` 的版本语义与 stage gate 定义，而且不能因为本次 eval 通过就自动认定差距已关闭或边界可以放宽；它还记录了本地 dirty 状态。本文会参考它，但不会把它当永久结论。([GitHub][6])

本文中的“直接证据”，是样本 JSON / matrix 的字段值、V0.0/V0.2/Gemini 三类结果中的明确条目、以及 README / review_eval / 任务书写明的 contract 与边界。本文中的“合理推断”，主要是把样本事实对照 V0.2 设计目标后的兑现度判断，例如“sample B 的 parser-limited PDF 还没有完全升格为任务级 L0 gate”这类判断。本文不假装裁决 expert truth，不把 Gemini 的候选问题自动升格为 hard defect，也不把 PDF text-only 盲区直接判成文档缺失。([GitHub][7])

## 3. 样本 A：冷轧厂三角对比研究

### 3.1 源样本关键特征

样本 A 的源文档是 DOCX，`structured-review-l0-visibility.json` 显示其 `parseMode` 为 `docx_structured`、`parserLimited=false`。样本中共有 2 个附件，其中 1 个附件正文可见、1 个附件只检测到标题而正文不可见；`missing=0`，并且顶层 `manualReviewNeeded=true`，理由是 `title_detected_without_attachment_body`。这说明样本 A 不是 parser 盲区主导，而是“局部附件可视域缺口 + 正文结构问题”主导。([GitHub][5])

从事实抽取看，样本 A 已识别出起重吊装、煤气区域作业、动火、临时用电等高风险类别；吊车能力 50t、计算起重量 2.86t、停机窗口 7 天、劳动力总数 37 人，都已被结构化提取。更重要的是，应急层并非完全空白：系统抽到了 4 个针对性预案标题，包括高处作业、触电、起重伤害、高温中暑。这意味着 A 样本里很多判断不是“无中生有”，而是在已有事实与已有措施基础上判断“闭环是否完整”。([GitHub][8])

### 3.2 三方总体差异

V0.0 在样本 A 上已经不是空白：Markdown 中有 4 个问题，分别是重复标题、附件可视域缺口、高风险作业专项方案挂接不清、以及停机窗口/资源/高风险并行的组织压力；并且它已经附带危大识别矩阵、规则命中矩阵和冲突矩阵。也就是说，V0.2 相对 V0.0 的升级不是“从无到有”，而是从“早期结构化报告”升级到“更正式的问题对象语义和边界纪律”。([GitHub][9])

V0.2 在样本 A 上把问题扩展到 8 条，层级分布为 L1=5、L2=2、L3=1；问题类型分布为 `hard_defect=6`、`evidence_gap=1`、`visibility_gap=1`，且 `unresolvedFactCount=0`。这不是单纯把 V0.0 的 4 条换个标题，而是把“缺证据”“可视域阻断”“适用性部分成立”“需要人工确认的专项方案挂接”这些 formal-review 语义真正写进了结果对象。([GitHub][10])

Gemini 在样本 A 上的长文报告则明显更重叙事与强结论：它直接把危大工程判定称为“根本性失误”，把煤气区域防范称为“系统性缺失”，并额外提出吊装站位地基承载力验算、七天停机窗口下资源极限调配逻辑等更丰富的 L3 候选问题。它在“候选问题 inventory”上更宽，但这不等于它已经完成了更稳的 truth 分层。([GitHub][11])

### 3.3 L0：可视域 / parser / 附件 / manual review

样本 A 最能证明 V0.2 相对 V0.0 的 L0 进步。V0.0 已经知道“附件处于可视域缺口”，但主要是 issue 文案层。V0.2 则把这件事做成了 top-level visibility contract：附件 1 明确是 `parsed`，附件 2 明确是 `attachment_unparsed`，`missing` 保持为 0，顶层和附件级都带 `manualReviewNeeded` 与 reason code；对应 issue 也被明确标成 `issue_kind=visibility_gap`、`applicability_state=blocked_by_visibility`。这正是“未解析 ≠ 缺失”从理念进入结果层的证据。([GitHub][9])

因此，样本 A 上 V0.2 相对 V0.0 的 L0 不是包装升级，而是 contract 升级。V0.0 的优势是已经有“看见问题”的雏形；V0.2 的升级是把“为什么只能看到这里、为什么需要人工复核、为什么不能直接叫缺失”写成了可被 API / eval / UI 共用的对象语义。([GitHub][3])

在这一层，Gemini 不是更强的 truth surface。仓库里 Gemini 对应的只是 supervision Markdown，而不是 visibility DTO 或 attachment matrix；研究说明和 review_eval 也都明确要求 Gemini 不能直接承担 visibility truth。对样本 A 而言，Gemini 可用来发现候选问题，但不能决定“附件 2 究竟是缺失、不可视，还是已可见”。([GitHub][5])

### 3.4 L1：硬规则 / 红线 / 结构完整性

样本 A 上，V0.0 的 L1 已能识别重复标题、附件可视域缺口和高风险作业专项方案挂接不清，但它的 issue 对象只有 `severity / finding_type / summary / manual_review_needed` 这些较薄字段。V0.2 则把同类问题分拆得更清楚：核心章节不完整被归为 `evidence_gap + blocked_by_missing_fact`；重复标题被归为 `hard_defect + applies`；附件问题被归为 `visibility_gap + blocked_by_visibility`；专项方案挂接不清被归为 `hard_defect + partial + manual_review_reason=special_scheme_reference_requires_manual_confirmation`。这说明 V0.2 已经把“硬命中”与“证据阻断”拆开，而不是只会统一报问题。([GitHub][9])

另一个不是文风而是机制的进步，是 pack 纪律。样本 A 的 `resolvedProfile` 里，`disciplineTags` 已经能识别出 `special_equipment` 与 `working_at_height`，但 `policyPackIds` 仍只选择 ready 的 `construction_org / lifting_operations / temporary_power / hot_work / gas_area_ops`；而 V0.0 的 Markdown 顶部仍把 `special_equipment.base` 与 `working_at_height.base` 直接列为选用 pack。结合 README 中这两个 scenario pack 仍是 placeholder，可以判断 V0.2 在“识别到标签”与“真的纳入正式 pack”之间，已经开始守住边界。([GitHub][9])

### 3.5 L2：条文适用 / 依据链 / evidence gap

样本 A 上最实的进步发生在 L2。V0.0 明确写出“L2 问题：无”；但 V0.2 的 rule hits 已经形成了清晰的二阶链条：`lifting_operations_calculation_traceability=pass`，说明起重参数和验算依据可追溯；`hot_work_emergency_targeted=hit`，说明动火场景的针对性应急安排仍有缺口；`gas_area_ops_control_linkage=hit`，说明煤气区域作业的控制措施 / 监测监控 / 应急链路未形成完整闭环。换言之，V0.2 已能区分“吊装场景存在”与“吊装验算已经有证据”，这已经是 formal review 语义，而不是笼统提示。([GitHub][9])

这一点还可以从事实层反证。样本 A 并不是完全没有措施：`measureSectionPresent=true`、`monitoringSectionPresent=true`，并且已有 4 个针对性应急预案标题。因此，V0.2 把煤气区域问题表述为“控制与应急链路不完整”，比“完全缺失”更贴近当前结构化证据。Gemini 把同一问题上升到“系统性缺失”甚至“致命合规盲区”，更像强结论式 candidate judgment，而不是更稳的 visibility-disciplined truth。([GitHub][8])

这里的差距归因，主要在 facts→rules→evidence 的闭环，而不在“会不会写规范条文”。样本 A 证明 V0.2 已经不再是简单“挂规范”，但它离成熟 L2 仍有差距：它能形成 partial / blocked / pass / hit 的语义，却还没有把“为什么适用、为什么只部分适用、缺口更像 visibility 还是正文不足”完整铺成可复核的 clause-applicability narrative。([GitHub][7])

### 3.6 L3：工程推理 / 冲突分析 / 可操作整改

样本 A 也是 V0.2 当前 L3 最像样的样本。V0.0 已经有一条 L3：停机窗口、投入人力与高风险工序并行的组织压力。V0.2 保留了这条 grounded inference，并通过 conflict matrix 把 `scheduleVsResources.issueTriggered=true` 明确写出；同时新增了“煤气区域作业控制与应急链路不完整”这一条更工程化的问题。尤其值得注意的是，`construction_org_shutdown_resource_conflict` 在样本 A 上是真的命中，这不是文风差异，而是规则基于 `shutdownWindowDays=7`、`laborTotal=37` 与高风险类别并行触发出来的。([GitHub][9])

但样本 A 同时也暴露出 V0.2 的 L3 未成熟：`ISSUE-005` 与 `ISSUE-008` 的 `finding_type` 都是 `engineering_inference`，却仍被 `issue_kind=hard_defect` 吸收。这与设计文档希望把 hard defect / visibility gap / evidence gap / suggestion enhancement 强分层的目标相比，只能算半兑现。也就是说，V0.2 已经能“说出 grounded engineering inference”，但还没有把这些 inference 全部放在一个成熟的 workflow language 里。([GitHub][12])

Gemini 在样本 A 的主要领先，也基本发生在这一层：它能提出更丰富的工程层候选问题，比如吊装站位地基承载力验算、资源极限调配逻辑重构等。但这类领先当前更像“candidate issue inventory 更宽”，而不是“L0-L2 更稳”。因此，这些点更适合进入 future adjudication，而不是被直接当作 patch 目标。([GitHub][11])

### 3.7 主要差距归因

样本 A 的主要差距，不在 parser。A 是 `docx_structured` 且 `parserLimited=false`，所以 V0.2 和 Gemini 的差距主要暴露在 facts 归并、rule coverage、evidence closure 与 report/workflow language。尤其是“gasArea 已识别，但 V0.0 的高风险类别未把 `gas_area_ops` 显式纳入；V0.2 则纳入并生成相应 issue”这一点，直接说明 V0.2 对 V0.0 的升级带有真实的 facts/rules 增量。([GitHub][13])

报告层差异也存在，但不是主因。V0.2 相比 V0.0 的核心，不是多了更长的句子，而是多了 `issue_kind / applicability_state / manual_review_reason / evidence_missing`；相对 Gemini 的核心，也不是“写得不够像专家”，而是缺少更完整的 L3 编排语言与更宽的 candidate inventory。([GitHub][12])

### 3.8 anti-overfit 初判

样本 A 可以抽象出的通用问题模式有四个：其一，附件可视域缺口必须与文档缺失严格分开；其二，高风险事实识别不等于专项方案挂接闭环；其三，已有措施/应急标题不等于控制链路完整；其四，工期-资源-高风险并行的冲突需要 grounded inference，而不是项目名驱动。([GitHub][13])

高过拟合风险的写法则非常清楚：不要围绕“煤气区域”“50t”“2.86t”“7 天”“37 人”“附件 2：施工总平面布置图”“防火安全”这些表面词或数字写规则；设计文档已经把“不得按项目名/文件名/fixture id 分支”“不得把 Gemini wording 当目标”写成硬约束。样本 A 能支持的是“通用问题模式”，不支持“冷轧厂专用补丁”。([GitHub][7])

## 4. 样本 B：培花三角对比研究

### 4.1 源样本关键特征

样本 B 的源文档是 PDF，且是本文最重要的 L0 压力样本。`structured-review-l0-visibility.json` 明确给出：`parseMode=pdf_text_only`、`parserLimited=true`；并记录了 `pdf_tables_not_preserved`、`pdf_figures_images_not_parsed`、`pdf_attachment_visibility_may_be_unknown`、`pdf_source_pages:276`、`pdf_extracted_pages:261` 等警告。附件计数为 0，attachment matrix 为空。也就是说，样本 B 的基础前提不是“文档差”，而是“当前只在 text-only 可视域内成立”。([GitHub][5])

在事实层，系统识别出了起重吊装、临时用电、动火等风险，也识别到了 50t 吊车与 `specialSchemePlanStatus=generic_mention_only`；但 `calculatedLiftWeightTon` 为 null，`monitoringSectionPresent=false`，`emergency.planTitles=[]`，`targetedPlanCount=0`，`shutdownWindowDays` 与 `laborTotal` 也都是 null。换言之，样本 B 的真实问题不是“完全没抽到风险”，而是“关键调度、监测、应急、参数事实存在大片空白”。([GitHub][14])

### 4.2 三方总体差异

V0.0 在样本 B 上只有 2 个问题：高风险作业专项方案挂接不清、应急预案针对性不足，且 L3 为无；其顶部还把 `special_equipment.base` 与 `working_at_height.base` 一并列为选用 packs。与样本 A 一样，V0.0 不是没有判断能力，但问题对象语义薄、pack 边界也更松。([GitHub][15])

V0.2 在样本 B 上的输出明显更“结构化审查器化”：结果对象给出 `issueCount=6`、`unresolvedFactCount=6`、`L1=3/L2=3/L3=0`，并把问题拆成核心章节不完整、总的专项方案挂接不清、起重吊装专项方案挂接不清、应急预案针对性不足、临时用电/停送电链路不完整、动火应急安排不足等更细颗粒度问题；同时，它把 `policyPackIds` 收敛在 ready packs 上，没有把 placeholder packs 当正式审查依据。([GitHub][16])

Gemini 在样本 B 上的“领先感”非常强，但它主要来自域覆盖与工程叙事：它展开了深基坑、软土、承压水、周边生命线工程、微顶管、多工艺耦合等复杂风险图谱。关键是，这种领先并不纯粹是“模型更聪明”，而很大程度上是“它在讲当前 ready pack 之外的域”。更值得注意的是，这份 Gemini 报告自己也在许多关键尺寸、埋深、距离处保留了空白占位，这说明它同样不能被当作稳定 truth。([GitHub][17])

### 4.3 L0：PDF / visibility / manual review

样本 B 是 V0.2 当前 L0 的优点和短板同时暴露的样本。优点是：它非常诚实地保留了 `pdf_text_only`、`parserLimited=true`、表格/图示未保留、附件可视域可能未知等警告；附件矩阵为空，并没有因为“PDF 有很多图表”就虚构附件缺失。这一点与仓库“未解析 ≠ 缺失”的边界是一致的。([GitHub][18])

短板是：尽管 visibility 明确承认 parser-limited，样本 B 的 top-level `visibility.manualReviewNeeded` 仍是 false，顶层 visibility 也没有生成 visibility-gap issue；真正把任务推到“需人工复核”的，是结果对象中的 issue-level partial / manual review。严格说，这不是直接的 contract 违背，但它说明 V0.2 设计文档里“让 L0 成为独立 stage gate 语言”的目标，在 parser-limited PDF 上还没有完全兑现。([GitHub][18])

Gemini 在这一样本上更不能被当 visibility truth。因为它一边做了高度具体的工程叙事，一边在多个核心尺寸、距离、开挖深度上留下了空白数字位。也就是说，它能放大“候选风险画像”，但它自己并没有消除 visibility 限制。样本 B 是 Gemini 只能做差距探针、不能做真相仲裁者的最好证据。([GitHub][17])

### 4.4 L1：硬规则 / 结构完整性 / 专项方案挂接

样本 B 上，V0.2 相对 V0.0 的升级非常具体。V0.0 只有“高风险作业专项方案挂接不清”这一条总括性 L1 问题；V0.2 则先把“核心章节不完整”单列为 `evidence_gap + blocked_by_missing_fact`，再把“高风险作业已识别，但专项方案挂接不清”与“起重吊装场景专项方案挂接不清”拆开，且都带上 `applicability_state=partial` 与 `manual_review_reason=special_scheme_reference_requires_manual_confirmation`。这不是换标题，而是把“总括性问题”分解成可复核的子问题。([GitHub][15])

更关键的是，样本 B 的 `resolvedProfile` 再次证明了 V0.2 的 pack discipline：它的 `disciplineTags` 里已经有 `special_equipment` 与 `working_at_height`，但 `policyPackIds` 仍只收敛到 `construction_org / lifting_operations / temporary_power / hot_work`。结合 README 中这两个场景 pack 仍是 placeholder，可以把样本 B 的一部分差距直接归因为“当前 ready rule/policy asset 尚未覆盖该类风险”，而不是“系统没想明白”。([GitHub][16])

### 4.5 L2：条文适用 / evidence gap / unresolved facts

样本 B 的 L2 特别能说明 V0.2 的真实上限。rule hits 显示：`lifting_operations_special_scheme_linkage=manual_review_needed`，`lifting_operations_calculation_traceability=pass`，`temporary_power_control_linkage=hit`，`hot_work_emergency_targeted=hit`，而 `construction_org_shutdown_resource_conflict=pass`。这意味着系统不是“什么都没做”，而是已经能区分：吊装场景存在，但专项方案挂接仍需人工确认；某些起重参数/验算痕迹存在，但临时用电和动火的控制链路仍不闭环。([GitHub][19])

同时，`unresolvedFactCount=6` 不是装饰性字段，它解释了为什么样本 B 没有进一步长出更强的 L3 冲突判断：`shutdownWindowDays`、`laborTotal` 等调度事实是 null，`monitoringSectionPresent=false`，应急 plan titles 为空，所以很多问题会停在 `evidence_gap` 或 `partial`，而不能稳健升级成更硬的结论。这里的差距性质首先是 facts / evidence / parser 的联合缺口。([GitHub][16])

样本 B 与 Gemini 的差距，尤其不能被理解成“V0.2 工程脑子不如 Gemini”。Gemini 长篇展开了深基坑、承压水、周边 DN3600 原水管、微顶管、多工艺耦合等复杂风险；而 V0.2 当前 ready pack 只覆盖 construction_org、起重、临电、动火，没有“深基坑 / 周边环境 / 降水 / 顶管”这一组 ready scenario pack。也就是说，这一差距很大一部分是 pack/policy asset coverage gap，而不是纯模型 gap。([GitHub][17])

### 4.6 L3：工程推理 / 冲突分析 / workflow

样本 B 上，V0.2 并没有形成真正的 L3 conflict issue。`construction_org_shutdown_resource_conflict` 在 rule hits 里是 `pass`，冲突矩阵里 `scheduleVsResources.issueTriggered=false`，直接原因就是 `shutdownWindowDays` 与 `laborTotal` 都是 null。换句话说，这不是“V0.2 不会写 L3 句子”，而是“同一条 L3 规则在没有关键事实时不应硬触发”。这一点反而说明它比自由发挥更保守。([GitHub][19])

但这也暴露了 V0.2 的真实短板：当 PDF text-only 造成关键调度与工程参数空白时，V0.2 的 L3 会显著收缩；Gemini 则会继续以长篇 narrative 展开工程逻辑。这里的差距不是文风，而是“是否有足够事实支撑 grounded L3”。因此，样本 B 上 Gemini 的价值主要是 future adjudication 的 candidate pool，而不是直接作为 L3 gold。([GitHub][17])

### 4.7 主要差距归因

样本 B 的第一归因是 parser / visibility。`pdf_text_only + parserLimited=true + tables/figures not preserved` 直接压缩了后续 facts、rules、evidence 的可用空间。第二归因是 facts：关键 schedule/resource/monitoring/emergency facts 未稳住。第三归因是 rule/policy asset coverage：当前 ready pack 并不覆盖 Gemini 在 B 样本中大量展开的深基坑、周边生命线与多工艺耦合域。第四归因才是 report/workflow。([GitHub][18])

因此，样本 B 不能被错误解读成“Gemini 更像专家，所以 V0.2 只差文风”；它更接近这样一个事实：当 L0 与 ready pack coverage 受限时，Gemini 看起来像“更懂工程”，但这种领先里混入了大量不受 canonical visibility/provenance 约束的叙事扩展。([GitHub][18])

### 4.8 anti-overfit 初判

样本 B 可以抽象出的通用问题模式是：PDF text-only 条件下，表格/图示/附图不可见会直接削弱事实完整性；专项方案挂接常常只能做到 partial + manual review，而不能直接升格为“已缺失”；L3 调度冲突高度依赖 schedule/resource 事实，缺关键事实时应保守收缩。([GitHub][18])

高过拟合风险的写法则是：围绕“10400 立方米”“DN3600 原水管”“高科西路”“北蔡镇”“微顶管”“承压水”等表面词写专用规则，或者把 Gemini 对深基坑叙事里的空白数字位置反向编码成 prompt 模板。这些都只能让系统变成“培花专用解释器”，而不会变成更稳的 formal reviewer。([GitHub][17])

## 5. V0.2 vs V0.2 设计目标：兑现度研究

### 5.1 已兑现的部分

第一，V0.2 的 contract spine 基本已经兑现。README 和样本结果都表明：`visibility` 已是 top-level canonical contract；`manualReviewNeeded` 是 canonical 布尔语义；`issues` 已显式带 `issueKind / applicabilityState / evidenceMissing / manualReviewReason`；`artifactIndex` 和 `structured-review-l0-visibility` 已进入结果体系。样本 A/B 的结果对象都清楚体现了这一点。([GitHub][3])

第二，V0.2 已把“未解析 ≠ 缺失”从设计原则推进到真实结果。样本 A 的附件 2 被明确保持为 `attachment_unparsed`，`missing=0`；样本 B 在 PDF text-only 下则宁可给空 attachment matrix，也不虚构附件缺失。对照 review_eval 的非协商原则，这说明 V0.2 至少在 L0 诚实度上已经超过了“泛泛报告器”。([GitHub][13])

第三，V0.2 已经兑现了“早期 structured reviewer”而不是“更成熟的辅助审查器”。样本 A/B 都有真实的 facts、rule hits、visibility、matrices、resolvedProfile 与 issue semantics；这已经明显超出 `review_assist` 那种快速辅助总结。([GitHub][3])

### 5.2 半兑现的部分

L0 是半兑现。设计文档明确把 L0 设为第一优先级，并希望它成为单独的 stage gate 语言；样本 A 上这一目标完成得较好，但样本 B 仍呈现“parseWarnings 很重、parserLimited=true、visibility.manualReviewNeeded=false、结果整体 manual review 却由 issue-level 触发”的状态。它说明 L0 contract 已存在，但 L0 gate 还没有彻底前置。([GitHub][7])

L2 也是半兑现。设计文档希望把 applicability / unresolvedFacts / evidence gap 做成完整闭环；样本 A/B 确实已经把 `applicabilityState`、`evidenceMissing`、`manualReviewReason` 写进结果，并出现了 `blocked_by_missing_fact`、`blocked_by_visibility`、`partial` 等真实状态。但样本 B 的 `unresolvedFactCount=6`，以及样本 A/B 都反复出现的 `special_scheme_reference_requires_manual_confirmation`，说明当前闭环仍经常停在“知道卡住了”，还没走到“为什么适用 / 为什么不能升级 / 下一步该先补什么”。([GitHub][7])

L3 仍是半落地。设计文档希望把 hard defect / visibility gap / evidence gap / suggestion enhancement 强分层，并进一步做 deterministic remediation buckets。现实里，样本 A 的 grounded inference 已经能命中 schedule-resource conflict，样本 B 也能形成临电/动火链路问题；但 issueKind 仍倾向把部分 engineering inference 吸进 `hard_defect`，而输出里也还没有“先补件 / 先补依据 / 先补正文结构 / 建议增强”这类稳定整改桶。([GitHub][7])

### 5.3 仍未兑现的部分

最没有兑现的是“把 parser / visibility / eval 完整制度化为第一真相源”。设计文档强调 L0 应独立入 gate，且 PDF 仍要保持保守语义；但样本 B 说明 parser-limited PDF 仍然会把部分问题推向 evidence gap / hard defect 侧，而不是先在任务层形成更强的 visibility truth。([GitHub][7])

另一个未兑现点，是 L3 workflow language。当前 V0.2 更像“若干 grounded inference 的集合”，还不是“哪些先补件、哪些先补依据、哪些阻断正式结论、哪些只是增强建议”的工作流语言。设计文档对此写得很清楚，而样本输出来看，这块还停在中途。([GitHub][7])

### 5.4 latest-eval-summary 的辅助判断

作为阶段性辅助证据，`latest-eval-summary.md` 既能说明进展，也能提醒不要过度乐观。它一方面记录了 `official versioned stage gate` 通过；另一方面又明确声明这只是本地阶段性快照，不等于长期 truth，且不能因为这次通过就认定边界可放宽。同时，同一快照里 L0 的 `attachment_visibility_accuracy` 仍是 0.5833，L1 的 `hard_evidence_accuracy` 仍是 0.6945，L3 的 `suggestion_defect_separation` 仍是 0.7542，且 L3 还被标成 `diagnosticOnly=true`。所以，这份快照能证明“V0.2 不是没进展”，但不能证明“V0.2 已经兑现完成”。([GitHub][6])

### 5.5 当前最准确的定位

综合双样本与设计目标，当前 V0.2 最准确的定位是：**已经明显超过“更成熟的辅助审查器”，更接近“早期结构化审查器”，但还不是成熟的 formal reviewer**。它最强的部分是 contract 与可视域诚实度，次强的是 rule-hit 到 issue 的 formal semantics；最弱的是 parser-limited PDF 下的 L0 gate、未覆盖场景 pack、以及 L3 workflow language。([GitHub][3])

## 6. 分层归因：L0 / L1 / L2 / L3

### L0

V0.2 相比 V0.0 的真实进步，是把可视域从“问题文案”升级成了 canonical result contract。样本 A 的 `attachment_unparsed != missing`、样本 B 的 `pdf_text_only + parserLimited + parseWarnings` 都是机制级进步，不是措辞升级。([GitHub][13])

V0.2 相比 Gemini 的核心差距，主要不是 L0 意识，而是 parser coverage。V0.2 在 L0 诚实度上其实更安全；它的短板是 DOCX / PDF 可视域能力不均，尤其是 PDF 仍停在 text-only。Gemini 的文本叙事能“看起来更完整”，但不是 canonical visibility truth。([GitHub][7])

这一层的主因是 parser / contract / eval，而不是 report 文风。它是系统机制问题。([GitHub][7])

### L1

V0.2 相比 V0.0 的真实进步，是把 L1 从“命中某条问题”推进到“这个问题到底是 hard defect、visibility gap 还是 evidence gap；是 applies、partial 还是 blocked”这一层。样本 A/B 的 issue objects 都给出直接证据。([GitHub][12])

V0.2 相比 Gemini 的核心差距，是 ready pack / policy asset coverage 与 severity calibration，而不是“不会写红线”。Gemini 往往更敢把问题升级成强结论；V0.2 则更受 pack 与 visibility 约束。([GitHub][11])

这一层的主因是 pack / policy asset / support-scope / eval。表达层有差异，但不是主矛盾。([GitHub][7])

### L2

V0.2 相比 V0.0 的真实进步，是 facts / rule hits / policy refs / applicability semantics 已经可以进入 stage gate 语言。双样本里都能看到同一组 generic 模式：`lifting_operations_special_scheme_linkage` 常常是 `manual_review_needed`，而 `lifting_operations_calculation_traceability` 却可以是 `pass`。这说明系统已能把“场景存在”与“依据链是否完整”分开。([GitHub][20])

V0.2 相比 Gemini 的核心差距，是 applicability closure 与 domain coverage。Gemini 常常在更宽的域里展开“为什么适用这条、为什么现场风险更大”，而 V0.2 当前更多依赖 ready pack 已覆盖的规则族。样本 B 尤其说明：很多“Gemini 看见、V0.2 没看见”的内容，实质是 current pack set 还没覆盖。([GitHub][7])

这一层的主因是 facts / evidence / policy asset / unresolved facts。不是纯表达问题。([GitHub][16])

### L3

V0.2 相比 V0.0 的真实进步，是已经能产出 grounded engineering inference，并能在有事实时触发冲突矩阵；样本 A 就是实证。([GitHub][9])

V0.2 相比 Gemini 的核心差距，是 workflow language 与 candidate breadth。Gemini 更擅长把若干风险串成连续工程论证；V0.2 目前更像 grounded inference 的集合。样本 B 则反向证明，这个差距不是纯文风——当 schedule/resource facts 为空时，V0.2 的 L3 就会明显收缩。([GitHub][7])

这一层的主因是 workflow / report / eval，其次才是 model expression。([GitHub][7])

## 7. 反过拟合研究观察

双样本共同出现、且具备通用研究价值的问题模式，至少有五类。第一，visibility gap 与 missing 必须永久分离；第二，高风险事实存在，不等于专项方案挂接与适用边界已经闭环；第三，L2 的真难点是 applicability / evidence gap，而不是多挂几条规范；第四，L3 grounded inference 强依赖关键 schedule/resource/monitoring/emergency facts；第五，ready pack coverage ceiling 会伪装成“模型不够聪明”。这些都是通用模式，不是样本表面词。([GitHub][13])

样本特有现象也很明显。样本 A 的“煤气区域作业控制链”与“7 天停机窗口 + 37 人 + 多高风险工序并行”是 A 特有高价值现象；样本 B 的“深基坑 / 周边生命线 / PDF 表格图示丢失 / 大量关键数值缺口”是 B 特有现象。这些都能启发研究，但不能直接上升为通用规则模板。([GitHub][8])

若把以下内容直接写成规则，几乎必然过拟合：项目名、文件名、fixture id、具体数字阈值、固定章节标题、具体附件名、Gemini wording。设计文档已把“不得在 `rules/*`、`extractors/*`、`report/*` 中按项目名/文件名/fixture id 分支”“不得把 Gemini wording 当目标”“不得把 Gemini 结果当专家 truth”写成硬约束。([GitHub][7])

Gemini 只能做差距探针，而不能做 gold truth，原因在双样本里都能看见：样本 A 中，它倾向用更强 severity 语言概括一个实际上“已有措施但闭环不完整”的场景；样本 B 中，它能展开很深的工程叙事，但同样带着多个空白数值位。它的价值是“候选问题发现”和“差距探针”，不是“最终仲裁答案库”。([GitHub][11])

因此，这份研究不能被直接跳译成 patch list。因为双样本极高价值，但样本量极小；任务书也已经明确警告，两例真实施组很容易诱导系统朝 case id、项目名、章节习惯和行业特征打补丁。本文能给的是“问题模式”和“分层归因”，不能直接给“样本特化改造脚本”。([GitHub][7])

## 8. 给后续治理文档的输入清单

### 输入给《V0.3 边界声明》

最应该收敛的边界有三条。第一，official scope 不应扩。README 仍明确只有 `construction_org` 与 `hazardous_special_scheme` 是 P0 official；双样本也已经说明，哪怕在 `construction_org` 内部，PDF L0 与场景 pack coverage 都还没稳。第二，本轮最值得优先聚焦的是 L0 parser/visibility gate、official L1 rule/evidence 核、L2 applicability closure，而不是追求更宽的 Gemini 风格叙事。第三，本轮不应把 OCR / 多模态纳入正式边界，因为当前设计与快照都仍把 PDF 定义为 `pdf_text_only + parserLimited=True` 的保守路线；若同时引入 OCR/多模态，会把边界、解释性和回归责任一起搅乱。([GitHub][3])

### 输入给《V0.2→V0.3 差距裁决与反过拟合约束清单》

已基本确认的差距有四类：一是 parser-limited PDF 的 L0 gate 尚未完全前置；二是 applicability / evidence gap / unresolved facts 闭环未成熟；三是 ready pack coverage 与 resolved disciplineTags 之间仍有明显落差；四是 L3 grounded inference 还没有升级成稳定 workflow language。([GitHub][18])

只能部分确认的差距，主要是 Gemini 提出的更宽工程域问题，例如样本 A 的吊装站位地基承载力验算、样本 B 的深基坑 / 周边生命线 / 降水 / 顶管等域风险。这些适合进入 adjudication / experimental diagnostics，但不适合直接写进 official patch。([GitHub][11])

高过拟合风险方向也已明确：围绕冷轧厂/培花项目名、围绕固定数字、围绕章节标题、围绕附件名、围绕 Gemini wording 的修补，都应被禁止；更适合进入 experimental / diagnostics 的，是新 pack/policy 资产探索、更多 visibility-heavy 样本桶、以及 cross-model explanation 层诊断。([GitHub][7])

### 输入给《双样本人工复核裁决稿》

适合 future internal-reviewed 的条目包括：样本 A 的附件可视域缺口、专项方案挂接 partial 项、煤气区域控制/应急链路不完整、schedule-resource 冲突；样本 B 的核心章节 evidence gap、专项方案挂接 partial 项、应急预案针对性不足、临时用电/动火链路问题。它们都有结构化事实、规则命中或明确的 blocked/partial 语义可供复核。([GitHub][12])

更像 visibility truth 的条目包括：样本 A 的 `attachment-2=attachment_unparsed`；样本 B 的 `pdf_text_only + tables/figures not preserved + attachment_visibility_may_be_unknown`。这些条目适合先写成 visibility adjudication，而不是 issue adjudication。([GitHub][13])

当前证据不足、不能让 Gemini 直接决定的条目包括：样本 A 中 Gemini 额外提出的吊装站位地基承载力验算；样本 B 中 Gemini 长篇展开的深基坑周边环境与生命线约束细项。它们应进入 future adjudication candidate pool，而不是直接定 truth，更不能直接把 Gemini 的 severity 语言写回 expert truth。([GitHub][11])

## 9. 最终研究结论

V0.2 相对 V0.0，**构成了真实的代际升级，但升级发生在“结构化审查 contract + L0 诚实度 + L1/L2 问题语义 + pack 边界纪律”层，而不是发生在“更会写报告”层**。V0.0 已经不是纯空白；它已有 issue、rule-hit、matrix 雏形。V0.2 的实质进步，是把这些雏形推成了 canonical result object。([GitHub][9])

V0.2 与 Gemini 的核心差距，也不是文风。**真正的差距顺序是：L0 的 PDF 可视域与 parser 限制；ready pack / policy asset 覆盖面；L2 applicability/evidence gap 闭环；L3 workflow language。** 与此同时，V0.2 在 L0 诚实度、visibility truth 与 provenance discipline 上，往往反而比 Gemini 更稳。([GitHub][18])

当前系统最准确的定位，不是“成熟 formal reviewer”，也不是“只是高级辅助审查器”，而是：**review control plane 内部一条已经成形、但尚未成熟的 structured_review 早期正式结构化审查器**。它的核心价值，已经不在于会不会写长文，而在于能否把 visibility、facts、rule hits、issues、manual review 与 artifacts 稳定地对齐。([GitHub][3])

后续治理最该先抓的是：L0 parser/visibility/stage-gate 一致化；official scope 内的 L1 rule/evidence 核加固；L2 applicability / unresolvedFacts / evidence gap 闭环；L3 deterministic remediation bucket 与 workflow language。最不能犯的错误，则是把双样本表面词语、具体数字、具体附件名或 Gemini wording 直接写成规则建议。([GitHub][7])

anti-overfit 必须前置，不是因为“谨慎一点比较好”，而是因为仓库自己已经把它定义成硬约束：Gemini 不是 truth，双样本不是泛化能力，visibility truth 与 issue truth 不能混写，也不允许按项目名 / 文件名 / fixture id 分支。**如果这一前提不先立住，V0.2→V0.3 最容易走向的，不是更好的 formal review，而是更会在两份样本上得高分的解释器。** ([GitHub][4])
