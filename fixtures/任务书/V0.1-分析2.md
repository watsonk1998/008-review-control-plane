# 008-review-control-plane 产品架构设计书（当前态盘点 + 下一阶段演进版）

## A. 执行摘要（Executive Summary）

本设计书仅以当前仓库 `watsonk1998/008-review-control-plane` 的 README、docs、代码、前端类型与页面、tests、fixtures、evaluation 资产为代码事实源，不使用已删除旧仓库作为事实依据。当前仓库的顶层定位已经相当明确：它不是把自己定义成“单一 formal reviewer”，而是一个 **review control plane / orchestration 项目**，同时开始承载一条独立的 `structured_review` 正式结构化审查子域。README、architecture、formal-review 文档，以及前端首页与 runtime 行为，均在强调这条双轨边界：`review_assist` 是辅助总结，`structured_review` 才是正式结构化审查通道。([GitHub][1])

因此，当前 repo 不能被写成“空白系统”。`structured_review` 已经不是名义存在，也不只是 DTO：后端已有 `TaskType`、请求字段、`StructuredReviewTask/Result`、review 子域目录、`DocumentLoader`、parser / extractors / rules / evidence / report / evaluation 代码、artifact API、前端专门表单与详情渲染、以及端到端测试。这说明它已经具备一条真实的最小执行链路。与此同时，这条链路的成熟度并不均匀：真正“有实规则、能稳定产出 issue/matrix/report”的核心支持对象，目前主要还是 `construction_org` 与 `hazardous_special_scheme`；其他文档类型和多数 scenario packs 仍停留在 registry / placeholder 层。([GitHub][2])

综合判断，当前项目最准确的阶段描述不是“还没有 formal review”，也不是“已经是可稳定签发的 formal reviewer”，而是：**一个已建立双轨边界的 control plane，外加一个已达到最小可用路径的 `structured_review` P1 级原型**。它的强项已经从“会写更长的审查文本”转向“开始拥有稳定 contract、分层流水线、artifact、matrix、manual review 语义与评测入口”；它的短板则集中在 **输入对象仍偏 fixture 化、解析深度不均、规则与 packs 覆盖不足、visibility/manual review 语义未完全规范化、以及 golden pool / expected facts / rule hits 尚未真正接入自动评测闭环**。([GitHub][3])

下一阶段不应再重复提出“引入 structured_review”“创建 review 子域”“定义 issue schema”这类已经落地的概念工作；最值得做的是 **把已有骨架做实**。我建议优先顺序是：P0 先把当前两类正式支持文档的结构化审查链路、输入契约、visibility/L0 语义与 versioned golden cases wiring 做稳；P1 再扩展 packs / rules / evidence / UI 人工复核闭环与分层评测；P2 才进入复杂工程推理、多模态附件/图纸、批处理与 pack 运营化。这样做符合当前 repo 的真实成熟度，也能避免再次退回“靠 prompt、靠更大模型、靠单案例 patch”的伪改进路径。([GitHub][3])

## B. 当前仓库真实现状（As-Is）

### B.1 仓库顶层与文档定位

**已确认事实：** README 把 008 明确定位为 **review control plane**，强调它是统一入口、总控编排、能力路由与未来运行时底座；同时 README 明写 `review_assist` 与 `structured_review` 双轨能力，并把 `structured_review` 定义为 `parse -> facts -> rules -> evidence -> report` 的正式结构化审查流水线。`docs/architecture.md` 进一步把 runtime 定义为 orchestration/coordinator，而把正式审查判断下沉到 `apps/api/src/review/`。`docs/formal-review.md` 则把 `structured_review` 的显式输入参数、稳定结果字段、artifact API、manual review 语义和当前正式支持文档类型说得很具体。`docs/known-limitations.md` 再次强调 `review_assist` 不是正式审查结论。([GitHub][1])

**成熟度判断：** 这一层不是“只有 README 宣言”。文档边界与代码边界是对齐的：前端首页区分了双轨，runtime 对 `review_assist` 和 `structured_review` 走不同分支，`review_assist` 结果里带有“这是辅助审查结果，不等于正式审查结论”的 notice。也就是说，**产品定位与链路边界已经进入代码，不是口号层**。([GitHub][4])

**当前限制：** `DELIVERY_REPORT.md` 仍更像项目初次交付时的总体说明，强调 control plane、adapter、前后端入口和基础联通，但它对后续已经出现的 `structured_review` 细节并不完整，不能替代后续代码事实。这里存在明显的“交付报告早于后续实现增长”的情况。([GitHub][5])

### B.2 后端 API、Domain Models、Task Contract、Persistence

**已确认事实：** `apps/api/src/domain/models.py` 已把 `structured_review` 提升为正式 `TaskType`，并定义了 `ReviewDocumentType`、`AttachmentVisibility`、`ReviewLayer`、`FindingType`、`EvidenceSpan`、以及 `CreateTaskRequest` 上的 `documentType / disciplineTags / strictMode / policyPackIds`。`apps/api/src/review/schema.py` 又补足了 formal review 子域对象：`StructuredReviewTask`、`ResolvedReviewProfile`、`DocumentParseResult`、`ExtractedFacts`、`RuleHit`、`IssueCandidate`、`FinalIssue`、`StructuredReviewSummary/Matrices/Result`。`sqlite_store.py` 已把 review-specific 字段持久化到 SQLite；`routes/tasks.py` 已提供任务创建、结果、事件、artifact 列表与 artifact 下载接口。([GitHub][2])

**成熟度判断：** request/result DTO、artifact contract、持久化列和 API 路由都已存在，属于 **已实现**，不是 schema-only。尤其 `artifactIndex` + `/api/tasks/{taskId}/artifacts` 的配合，已经形成了“前端只信下载 URL，不直读本地路径”的契约闭环。([GitHub][3])

**当前限制：** 契约内部仍有重复和不完全贯通。典型例子是 `manualReviewNeeded` / `whetherManualReviewNeeded` 的双字段并存；`IssueCandidate` 有 `evidenceMissing` 与 `manualReviewReason`，但 `FinalIssue` 没把这些显式带出来。这意味着 **内部候选对象比外部结果 contract 更丰富**，存在结果层信息折损。([GitHub][6])

### B.3 Planner / Router / Runtime 的现状

**已确认事实：** `planner.py` 对 `structured_review` 单独生成 formal review 执行计划，执行阶段显式写成 `parse -> extract -> rules -> evidence -> explain -> report`，并明确 boundary：formal review pipeline 在 `apps/api/src/review`，runtime 仅负责 orchestration。`router.py` 已有 `choose_structured_review_profile` 和 `structured_review_executor` capability chain；而 `review_assist` 仍走 `fast -> deeptutor -> llm_gateway`，有 fixture 时会插入 `gpt_researcher`。`deepresearch_runtime.py` 则实例化 `StructuredReviewExecutor`，并在 `execute_task()` 中对 `review_assist` 与 `structured_review` 做独立分支处理。([GitHub][7])

**成熟度判断：** 这一层已经形成清晰的任务编排边界，属于 **已实现**。尤其 router 对 structured_review 的 capability chain 已不再把 GPT Researcher / FastGPT 当作 formal review 主裁判，这与 formal-review 文档里“LLM 只是解释层，非主裁判”的原则一致。([GitHub][8])

**当前限制：** `deepresearch_runtime.py` 目前仍硬性要求 `structured_review requires a fixtureId`，这说明 formal review 的输入模型仍偏 fixture 驱动，而不是平台化的“任意来源文档对象”。同时，router 与 review pipeline 都在做一部分 profile/documentType/pack 解析，存在职责重复。也就是说，**runtime 的边界方向是对的，但输入抽象与 profile 解析尚未完全收敛**。([GitHub][9])

### B.4 DocumentLoader、Parser、Visibility

**已确认事实：** `document_loader.py` 已支持 `.md/.txt/.docx/.pdf`。其中 `.docx` 走专门的 `parse_docx_document()`，`.pdf` 则通过 `pdfplumber` 抽取纯文本，再回落到 `_parse_text_document()`。`DocumentParseResult` 已包含 `sections / blocks / tables / attachments / figures / normalizedText / preview / visibilityReport / parseWarnings`。附件可视域判断由 `attachment_indexer.py` 完成：附件默认 `referenced_only`，若识别到附件标题则提升为 `attachment_unparsed`，若附件标题后存在实际内容块则提升为 `parsed`；只要有非 `parsed` 附件，就会把 `manualReviewNeeded` 置为 `true`。([GitHub][10])

**成熟度判断：** DOCX 解析与附件 visibility 语义已经进入真实运行时，属于 **初步可用**；它不只是“能抽文本”，而是已经产出结构化 section/block/table/attachment 结果，并被下游 rules/matrices/report 消费。([GitHub][10])

**当前限制：** PDF 路径明显更浅：当前只是文本抽取，再走文本 parser，并没有 DOCX 路径那种表格、图、附件结构深度。这意味着在 formal review 场景里，**“支持 PDF”目前更接近“能读到部分文本”，而不是“对 PDF 也完成同等级结构解析”**。此外，`AttachmentVisibility` 虽定义了 `missing` 与 `unknown`，但当前 attachment indexer 直接把这两类计数固定为 0，说明这两个状态更多是契约预留，不是已充分落地的推理结果。([GitHub][10])

### B.5 Review 子域：Parser / Extractors / Rules / Evidence / Report / Pipeline

**已确认事实：** `apps/api/src/review/` 已包含 `parser / extractors / rules / evidence / report / evaluation / pipeline.py / schema.py` 等子目录与文件。`StructuredReviewExecutor` 在 `pipeline.py` 中真实串起了 parse、facts、rule hits、candidates、final issues、matrices、summary、reportMarkdown 和 artifactIndex，并为 parse/facts/rule-hits/candidates/result/matrices/report 写出 artifacts。extractors 已覆盖 `project_facts`、`hazard_facts`、`schedule_resource_facts`；rule engine 已有 `construction_org` 与 `hazardous_special_scheme` 两套实规则；evidence builder 会把 rule hits 组装成 `IssueCandidate`；report 层则生成 matrices、summary 和 markdown 报告。([GitHub][11])

**成熟度判断：** 这是当前仓库里最重要的非空白证据。`structured_review` 已经形成了真实的六层流水线雏形，整体属于 **初步可用**，而不是 scaffold-only。([GitHub][12])

**当前限制：**

1. 真实规则当前集中在两类文档：`construction_org` 和 `hazardous_special_scheme`。`construction_scheme.base`、`supervision_plan.base`、`review_support_material.base` 虽然存在，但 `ruleIds=[]`；所有 scenario packs 也都是空 `ruleIds`。这意味着 registry 有了，但 coverage 远未做实。([GitHub][3])
2. facts 抽取已能提取危大类别、专项方案挂接、停机窗口、人力、应急预案标题、验算依据存在性等，但本质仍以 deterministic regex/section/table heuristics 为主，覆盖面偏窄。([GitHub][13])
3. `IssueCandidate` 比 `FinalIssue` richer，说明 report 层对证据语义有折损；这会影响 reviewer 对 “evidence 缺失 vs visibility 缺口” 的精确理解。([GitHub][6])

### B.6 Adapters：LLM / GPT Researcher / FastGPT / DeepTutor

**已确认事实：** `llm_gateway.py` 在 formal review 中的职责被刻意压窄：只做 issue title 清洗、recommendation 生成、候选 issue 整理，明确不允许增加不存在的文档/法规证据。`fastgpt_adapter.py` 主要是 chunk 检索；`gpt_researcher_adapter.py` 主要负责 local-doc research 或 source-grounded research；`known-limitations.md` 也说明 DeepTutor 是 bridge 形态，GPT Researcher 依赖较重，前端仍是 polling。([GitHub][14])

**成熟度判断：** adapter 层的工程化程度对 control plane 是 **已实现** 的；而对 formal review 来说，最关键的不是“接了多少模型”，而是它们已经被明确约束在工具层，不是主裁判。这个边界目前是健康的。([GitHub][15])

**当前限制：** `StructuredReviewExecutor` 的构造函数虽然接收 `fast_adapter`，但当前主执行链并没有真正使用外部检索阶段，formal review 实质上仍主要依赖本地文档解析与确定性规则。这本身不是坏事，但说明如果未来要把外部知识或标准解释引入 formal review，必须小心防止 adapter 越权成为裁判。([GitHub][12])

### B.7 前端：Task Entry、Structured Review Form、Task Detail

**已确认事实：** 前端类型 `apps/web/src/types/control-plane.ts` 已对齐后端 task types、review enums、evidence span、review issue、structured result。首页 `home-dashboard.tsx` 已把 `structured_review` 作为独立任务选项，并区分 control plane 与 formal review 边界。`structured-review-form.tsx` 已暴露 `documentType / disciplineTags / strictMode / policyPackIds`。`task-detail.tsx` 能识别 `StructuredReviewResult`，展示 summary、resolvedProfile、manual review 提示、reportMarkdown、issues、matrices、artifacts 与原始 JSON。([GitHub][16])

**成熟度判断：** 就“是否已有 structured_review 专门 UI 入口与消费 contract”而言，前端属于 **已实现**。当前 UI 不是把 structured_review 当普通文本结果塞进一个通用页面，而是已经有明确类型守卫和专门展示区域。([GitHub][17])

**当前限制：** UI 当前更像“工程调试/结果浏览页”，还不是“人工复核工作台”。它会显示 manual review summary、artifact list、attachment visibility matrix 和 raw JSON，但未见 reviewer decision 的写回入口、逐 issue 复核状态流转、附件原件预览或人工确认记录。与此同时，文档也明确前端目前仍采用 polling，而不是 SSE。([GitHub][17])

### B.8 Fixtures / Evaluation / Tests / Regression Assets

**已确认事实：** `fixtures/` 下已存在 `review_eval/`、`supervision/`、`golden pool/`、`任务书/` 等目录。`review_eval/README.md` 明确写出：当前版本化样本仍是 bootstrap data，不是最终专家确认 golden truth；legacy 的 `case_XXX/metadata.json` 数据集保留；版本语义规划为 `v0.1.0-gemini-seed`、`v0.2.0-internal-reviewed`、`v1.0.0-expert-golden`。目录结构显示 legacy evaluator case 分布在五类文档中，而 versioned seed 当前只有两个 `construction_org` 样本。([GitHub][18])

**已确认事实：** `evaluation/dataset.py` 目前只扫描 `metadata.json`，即仍以 legacy evaluator dataset 为主；`metrics.py` 统计的是 `issue_recall / l1_hit_rate / pack_selection_accuracy / policy_ref_accuracy / attachment_visibility_accuracy / severity_accuracy / manual_review_flag_accuracy`，没有 facts-level 或 rule-hit-level 指标；`harness.py` 已有 main/ablations/cross-pack/cross-model 模式。`docs/testing.md` 也与此对齐，提供 `make eval-review*` 命令，并把 structured_review 验收写到 issue/matrices/artifact API 级别。([GitHub][19])

**已确认事实：** `tests/test_structured_review.py`、`tests/test_runtime.py`、`tests/test_task_service_and_routes.py` 已对 structured_review 做了 parser、executor、runtime、route/artifact 的真实回归。测试断言不仅看“有结果”，还直接断言 duplicate sections、attachment visibility、special scheme gap、shutdown resource conflict、hazardous special scheme core sections / calculation evidence 等 issue 标题与 manual review 行为。([GitHub][20])

**成熟度判断：** regression 体系不是空白，属于 **初步可用**。但 versioned golden pool 及 expected facts / expected rule hits 目前更像“评测资产已准备一半”，尚未真正贯穿到自动评测主流程。也就是说，**当前 repo 同时拥有一个可跑的 legacy regression set，和一个尚未 fully wired 的 versioned golden pool 雏形**。([GitHub][21])

* **谁负责辅助总结：** `review_assist`。这是 control plane 的知识整合/辅助输出链路。([GitHub][1])

## F. 关键问题与真实风险

### F.1 输入对象定义风险

`StructuredReviewTask` 已经具备 `sourceDocumentPath` 字段，但对外 `CreateTaskRequest` 仍主要依赖 `fixtureId`，runtime 也直接对 `structured_review` 抛出 `requires a fixtureId`。这说明 formal review 的内部对象设计已经朝“通用文档对象”方向走了一步，但系统入口仍停在 fixture 驱动。这个割裂会限制后续上传、批处理、审查归档与 reviewer 二次复核。([GitHub][6])

### F.2 解析深度与输入降维风险

DOCX 路径已能形成 sections/blocks/tables/attachments，PDF 路径却主要是 `pdfplumber` 纯文本抽取再走 text parser。于是对 PDF 来说，formal review 很容易退化成“文本看到了多少”，而不是“文档结构看到了多少”。如果不把这种降维显式暴露给结果 contract，用户会误把 parser 能力不足理解为文档本身缺失。([GitHub][10])

### F.3 visibility 误判风险

当前 repo 已正确引入 `parsed / attachment_unparsed / referenced_only` 三类语义，并把“非 parsed 先人工复核”落到了 parser、rule、evidence、summary、notice 与 docs 中，这是明显进步。风险在于：`missing` 与 `unknown` 仍基本是预留状态，而不是已被充分推出的运行态；如果未来简单把“非 parsed”都上升为硬缺陷，仍可能把 visibility 问题误写成 document defect。([GitHub][32])

### F.4 facts 漏抽 / 错抽风险

当前 facts extraction 已开始覆盖危险源、专项方案挂接、验算依据、人力与停机窗口等关键点，但仍主要依赖标题、段落和表格中的有限 heuristics。`ExtractedFacts` 设计里已经有 `unresolvedFacts`，说明系统自己也承认有“抽不出来”的空间；如果没有对 facts 层建立独立评测，后续很容易在 rules 层背锅。([GitHub][6])

### F.5 rules / packs 过浅风险

当前实规则主要覆盖两个 base packs，其他三类文档和所有 scenario packs 都还是 placeholder。这样会带来一个认知风险：系统表面上“支持多个 doc type / discipline tags”，但其中相当一部分其实只存在 registry 覆盖，而非有实质审查能力。formal-review 文档对此已诚实降级，但 UI 和未来任务书若写法不谨慎，仍可能产生“名义支持被误读为稳定支持”的问题。([GitHub][3])

### F.6 evidence 链条不完整风险

`IssueCandidate` 明确包含 `docEvidence`、`policyEvidence`、`evidenceMissing`、`manualReviewReason`，但 `FinalIssue` 只保留了 evidence spans、recommendation、confidence 和 `whetherManualReviewNeeded`。也就是说，**候选层的“为什么证据不足 / 为什么要人工复核”在最终外部结果里被压缩了**。这会削弱 formal review 的可解释性，尤其在 reviewer 要区分“条文依据不足”“附件未解析”“正文已证实缺陷”时。([GitHub][6])

### F.7 LLM 越权判断风险

当前设计已经明确把 LLM 限定在解释层，这本身是优点而不是风险源。真正的风险在于未来扩展时若打破这条边界，把 issue 是否成立重新交给更大的模型或更长 prompt，就会直接破坏当前 repo 最健康的一条工程原则。因此，**LLM 越权不是当前已发生的问题，而是下一阶段最需要防止的回归风险**。([GitHub][3])

### F.8 report builder 过度摘要与 UI 工作台不足风险

当前 report builder 会输出总体结论、L1/L2/L3 issues、以及多张 JSON 矩阵；task detail 页面也会把这些内容直接展示出来。这种形式利于调试与追溯，但更接近“结构化开发输出”而非“审查员复核工作台”。特别是 attachment visibility、issue evidence、manual review 原因，目前更多是以 JSON / 文本块方式出现，缺少可操作的审查动作模型。([GitHub][33])

### F.9 regression 体系不足风险

当前确实有可跑的评测 harness，但它主要面向 legacy dataset，并以 issue/visibility/pack/policy/severity/manual-review 这些指标为主。versioned seed cases 中更精细的 `expected_facts.json`、`expected_rule_hits.json` 还未纳入评分；这会导致系统在“最终输出还看起来差不多”的情况下，事实层和规则层悄悄退化而不被 CI 及时发现。([GitHub][19])

### F.10 single-case overfitting 风险

当前 versioned seed 只有两个 construction_org 样本，且 README 明言其中一个 seed case 是基于原始施组与 Gemini deepresearch 结果构建的 bootstrap data，不是专家确认 gold。若团队下一阶段把规则、报告格式或 issue 组织过度围绕冷轧厂 / 培花两例优化，就会把“对两个样本变好”误当成“formal review 体系成熟”。([GitHub][21])

### F.11 legacy `review_assist` 持续干扰产品认知的风险

`review_assist` 已被很好地限制为辅助结论，但它仍与 `structured_review` 共享同一平台、同一首页和相近的任务体验。若没有持续强化“review_assist 是预审/辅助，structured_review 才是 formal review 输出”的边界，用户依然可能用“文风像审查报告”的标准去误判 `review_assist` 成熟度。([GitHub][1])

## G. 目标产品定位（To-Be Product Positioning）

### G.1 目标定义

下一阶段，这个产品不应被重新定义成“单一 formal reviewer”，而应被稳定定义为：

> **一个以任务编排、能力路由、结果契约、artifact、评测与可观察性为底座的 review control plane；其下承载一条正式结构化审查执行器家族，当前第一条家族成员即 `structured_review`。**

这个定义与当前 README / architecture 边界一致，也保留了 control plane 的长期扩展价值。([GitHub][1])

### G.2 双轨制的稳定划分

**`review_assist` 应继续归属于：**

* 快速预审与辅助判断
* 规范问答、知识解释、源文档研究
* 给人工审查员提供“从哪看、看什么”的线索
* 不能替代正式签发结论的场景

这与 current runtime、known-limitations 和首页定位完全一致，下一阶段不应再试图把它拔高成 formal review。([GitHub][9])

**`structured_review` 必须归属于：**

* 稳定输入 profile
* 结构化 parse/facts/rules/evidence
* 显式 issue / matrices / report / artifacts
* visibility / manual review 语义
* regression / golden pool 驱动的 formal review 演进

这也是当前 formal-review 文档、review 子域代码和 UI contract 已经开始承载的职责。([GitHub][3])

### G.3 为什么这种划分适合 control plane 的长期定位

因为 control plane 的长期价值不在于“自己写出多漂亮的报告”，而在于：

1. 统一任务入口与运行时；
2. 统一结果契约与 artifacts；
3. 统一可观察性与评测；
4. 允许多个 formal-review 执行器在同一平台下并存与演进。

当前 repo 已经在做第一条 formal-review 子域，即 `structured_review`。下一阶段应继续沿这条路线，而不是把所有能力重新揉回一个大 prompt。([GitHub][15])

## H. 目标架构设计（To-Be Architecture）

### H.1 保留的层

**应保留：**

* `planner / router / runtime` 作为平台级 orchestration 层
* `apps/api/src/review/` 作为 formal review 子域
* `artifactIndex + artifact API` 作为前端唯一可信下载入口
* `LLM Gateway` 的“解释层，不是主裁判”边界
* 前端对 `structured_review` 的独立类型与页面渲染

这些都是当前 repo 已经做对、且不该推倒重来的骨架。([GitHub][15])

### H.2 需要做实的层

#### 1. 输入对象层

把 formal review 的输入从“必须 fixtureId”收敛为统一的 `ReviewInputRef`：

* `type = fixture | upload | artifact`
* `path / artifactId / fixtureId`
* `sourceMimeType`
* `originalFilename`
* `ingestionWarnings`

这样既兼容当前 fixtures，又能迈向真实上传与批处理。当前 `StructuredReviewTask.sourceDocumentPath` 已经提示了这条方向，只是系统入口还没统一。([GitHub][6])

#### 2. Review profile 层

把 documentType / disciplineTags / policyPackIds 的**最终解析**集中到 review 子域内部的 `ReviewProfileResolver`，router 只负责粗粒度任务路由，不再承担 formal review 的最后裁决。理由很简单：最终 doc type / discipline inference 往往要依赖 parse/facts，而不是 query 文本。当前 pipeline 已有 `_resolve_profile()`，router 也有 `choose_structured_review_profile()`，这是典型的职责可收敛点。([GitHub][8])

#### 3. L0 可视域层

下一阶段必须把当前分散在 `AttachmentVisibility`、`visibilityReport`、`FindingType.visibility_gap`、`RuleHit.matchType='visibility_gap'`、summary/manual-review flags 中的语义，提升为显式的 **L0 VisibilityAssessment**。
这层只回答：

* 我看到了什么
* 我没看到什么
* 我为什么没看到
* 这是否阻断 L1/L2/L3 结论

这样可以从架构上防止“未解析 = 缺失”的误判。([GitHub][2])

#### 4. 分层评测层

把当前 versioned seed 中已有的 `expected_facts.json` / `expected_rule_hits.json` 真的接进 harness，并把 metrics 扩展为：

* parse/visibility 准确性
* facts 命中率
* rule-hit 命中率
* final issues / severity / manual review
* pack selection

否则当前 formal review 只能算“输出回归”，还算不上“流水线回归”。([GitHub][27])

### H.3 需要新增的层

#### 1. Reviewer Decision 层

新增 `ReviewDecision` / `IssueReviewState` / `AttachmentReviewState`：

* reviewer 是否已确认 visibility gap
* 哪个附件已人工复核
* 哪些 issue 被确认为有效 / 无效 / 待补件
* 是否形成“修改后重新报审”与原因

当前系统只会提示“需要人工复核”，但不会记住复核结果。下一阶段必须补这一层。([GitHub][17])

#### 2. Support Scope Manifest 层

新增 `support_scope.json` 或等价配置，明确：

* official supported doc types
* experimental doc types
* placeholder packs
* disabled scenario packs

这样 UI、docs、evaluation 和 runtime 都能共享同一口径，减少“命名支持 vs 实际支持”的歧义。当前 formal-review 文档虽然写了正式支持范围，但还没有看到这一层被系统化复用。([GitHub][3])

### H.4 需要解耦的层

#### runtime 继续保留的职责

* 任务生命周期
* 事件流
* artifact 写出与下载
* adapter 管理
* 通用 API 与状态存储

#### 必须下沉到 review 子域的职责

* profile 最终解析
* pack applicability
* parser/visibility 规则
* fact extraction 与 unresolved handling
* rule-hit 与 evidence 归档
* report semantics

#### 必须保持边界的职责

* GPT Researcher / FastGPT / DeepTutor：只能作为背景研究、检索、解释辅助，不直接裁决 formal review issue
* LLM Gateway：只能做结构整理与解释，不负责主命中判断

这与 current architecture 已经表达的方向是一致的，下一阶段是把它彻底收口。([GitHub][15])

### I.2 建议的结果契约收敛方向

下面给出一个**建议新增设计**，用于解决当前 `manualReviewNeeded` / `whetherManualReviewNeeded` / `visibility_gap` / `evidenceMissing` 分散的问题：

```ts
type ReviewStatus = "pass" | "needs_revision" | "ready_for_manual_review";

interface VisibilityAssessment {
  hasGap: boolean;
  reasons: Array<"attachment_unparsed" | "referenced_only" | "missing" | "unknown">;
  unresolvedAttachments: Array<{
    attachmentId: string;
    title: string;
    visibility: AttachmentVisibility;
    requiresManualReview: boolean;
  }>;
}

interface ManualReviewSummary {
  needed: boolean;
  reasons: Array<"visibility_gap" | "evidence_missing" | "referenced_only" | "attachment_unparsed">;
  reviewerDecisionRecorded: boolean;
}

interface FinalIssueV2 extends ReviewIssue {
  evidenceMissing: boolean;
  manualReviewReason?: string | null;
}

interface StructuredReviewResultV2 {
  summary: {
    overallConclusion: string;
    reviewStatus: ReviewStatus;
    documentType: ReviewDocumentType;
    selectedPacks: string[];
    manualReview: ManualReviewSummary;
  };
  resolvedProfile: ResolvedReviewProfile;
  visibility: VisibilityAssessment;
  issues: FinalIssueV2[];
  matrices: StructuredReviewMatrices;
  artifactIndex: TaskArtifact[];
  reportMarkdown: string;
}
```

这个建议不是从零发明新系统，而是把当前 repo 已存在的分散语义，收敛成更稳定的外部 contract。它的核心价值在于：**把 L0 visibility 与 issue 层、manual review 层拆清**，让 UI、artifact、eval 可以共享同一个结果口径。当前代码已经提供了这些概念的大部分素材。([GitHub][6])

### K.1 P0：做实已有骨架

**目标：**
把当前已经存在的 formal review 骨架做成“可持续迭代的最小可用产品”，而不是继续扩展概念面。P0 不是“新增 structured_review”，而是**让 `structured_review` 真正成为可验证、可回归、可解释的最小正式审查路径**。([GitHub][1])

**范围：**

1. **做实已有骨架**：DOCX/PDF parse + visibility + supported doc types 两条主链；
2. **解耦职责**：router/runtime 与 review profile 解析解耦；
3. **UI / contract 对齐**：manual review / visibility / evidenceMissing 外部 contract 收敛；
4. **评测补齐**：接入 versioned seed、facts/rule-hit 评分。

**交付产物：**

* `ReviewInputRef` 统一输入对象
* `VisibilityAssessment` / `ManualReviewSummary`
* versioned eval loader + stage-level metrics
* 两类主支持文档的更稳定 regression baseline

**风险：**

* 前后端 contract 变更
* legacy dataset 与 versioned dataset 并行期的治理复杂度
* “更稳”短期可能暴露更多 false positives/negatives

**验收标准：**

* `structured_review` 不再强依赖 fixtureId
* facts / rule hits / visibility / final issues 四层都有自动评测输出
* `attachment_unparsed` / `referenced_only` 不被系统写成硬性“缺失”
* 前端可明确区分 visibility gap、evidence missing、hard defect

### K.2 P1：扩深 packs、evidence、UI 复核与回归池

**目标：**
在 P0 已稳的基础上，把 formal review 从“两类主支持对象的原型”扩成“多类文档的可持续演进体系”。

**范围：**

* `construction_scheme` / `supervision_plan` / `review_support_material` 首批非空 base packs
* 2~3 个非空 scenario packs（如 `lifting_operations`、`temporary_power`、`hot_work`）
* reviewer decision API + UI
* 更多 versioned cases，开始 internal-reviewed 阶段

**交付产物：**

* 新 base packs / evidence packs
* reviewer decision artifacts
* 更清晰的 issue cards / visibility panel / evidence panel
* cross-pack 与 stage-level regression 报表

**验收标准：**

* 不少于 3 类非 placeholder packs
* manual review 结果能被记录并再次展示
* versioned eval 覆盖不再只有两个 construction_org seeds

### K.3 P2：复杂工程推理、多模态与运营化

**目标：**
在核心 formal review 主链稳住之后，再做复杂增强，而不是把复杂平台化工作提前。

**范围：**

* 图纸/图片/扫描件/复杂附件解析
* 多文档批处理
* 复杂整改编排与跨文档冲突推理
* pack version governance / release process / reviewer feedback 闭环

**验收标准：**

* 新能力不会破坏 P0/P1 的 regression baseline
* 多模态引入后仍保持“未解析 ≠ 缺失”的边界
* pack 版本升级可回溯、可回归、可比对

## M. 评测与 golden pool 演进方案（Evaluation & Golden Pool Roadmap）

### M.1 当前评测状态

当前 repo 的评测资产不是空白，而是**两套东西并存**：

1. **legacy regression dataset**：由 `case_XXX/metadata.json` 驱动，harness 当前实际扫描这套数据；它覆盖五类文档，构成今天真正可跑的回归基线。([GitHub][35])
2. **versioned golden pool 雏形**：目前只有两个 `construction_org` 的 `v0.1.0-gemini-seed` case，附带 `case_metadata.json`、`expected_facts.json`、`expected_rule_hits.json`、`ground_truth_visibility.json` 等更细粒度资产，但 README 明确说明它们只是 bootstrap seed，不是 expert-confirmed gold。([GitHub][21])

因此，当前状态既不是“稳定 golden pool”，也不只是“没有评测”。最准确的判断是：**legacy 内部回归集 + versioned mini golden seeds 并存；前者已运行，后者已起步但尚未 fully wired。** ([GitHub][21])

### M.2 当前主要问题

1. **当前自动评测仍偏最终结果层。** `metrics.py` 已评 issue recall、L1 hit rate、pack selection、policy ref、attachment visibility、severity、manual review flag，但没有 facts-level 或 rule-hit-level评分。([GitHub][36])
2. **versioned seed 没真正接入主流程。** `dataset.py` 只扫描 `metadata.json`，而 versioned cases 使用 `case_metadata.json`。([GitHub][19])
3. **module ablation 还不完整。** 目前有 normalizer / visibility / rule engine / llm explanation ablations，但还缺更清晰的 extractors / evidence / report-builder 级退化对照。([GitHub][28])
4. **跨模型 / 跨 pack 对照已有框架，但仍偏简化。** 这说明 harness 方向是对的，但要服务 formal review 演进，还需要更真实的 stage-level 对照。([GitHub][28])
5. **golden truth 风险仍存在。** golden pool 指导书和 seed README 都在提醒：Gemini 或当前 008 输出不能直接当 gold truth。这个原则必须内化到数据治理中。([GitHub][30])

### M.3 下一阶段演进方案

#### 1）端到端评测

保留当前 end-to-end issue 评测门，但不再把它当成唯一真相。它主要回答“最终 formal review 输出是否总体可接受”。([GitHub][37])

#### 2）模块消融

把现有 ablations 扩展成真正的分层消融：

* parser / visibility
* extractors
* rule engine
* evidence builder
* llm explanation
* report builder

当前 harness 已经有消融入口，因此这是“做实已有骨架”，不是从零发明。([GitHub][28])

#### 3）跨模型对照

严格限制为 **同 facts、同 rules、同 candidates，只替换 explanation layer**。
这样才能验证“模型不同是否仅影响表述，而不影响 formal review 事实与裁决”。这与当前 formal-review 文档的 LLM 边界一致。([GitHub][3])

#### 4）跨 pack 对照

当 non-placeholder packs 开始增多后，必须比较：

* auto pack selection
* forced expected packs
* base pack only
* base + scenario pack

避免“新增 pack 后只是输出更多问题，却不知道是否更准”。当前 harness 已有 cross-pack 框架，应继续沿用。([GitHub][28])

#### 5）分层评测

下一阶段必须把以下四层同时纳入评测：

* **facts**
* **rule hits**
* **visibility**
* **final issues**

其中 visibility 必须独立标注，不能让 visibility 错误污染 hard-defect truth。这个原则与 golden pool 指导书完全一致。([GitHub][30])

#### 6）regression baseline

保留 legacy 20-case 集合作为 baseline，不要在 versioned golden pool 尚未成熟前直接废弃它；但所有新演进必须优先接入 versioned format，以免继续把历史集合作为唯一数据模型。([GitHub][21])

#### 7）stable golden pool 版本机制

建议采用：

* `v0.1.x`：bootstrap seed
* `v0.2.x`：internal-reviewed
* `v1.0.x`：expert-golden

每个版本都要包含：

* case metadata
* source refs
* expected facts
* expected rule hits
* ground truth visibility
* ground truth issues
* evaluation notes
* release notes / changelog

这与当前 README 方向一致，只是要把它工程化。([GitHub][21])

### M.4 样本池覆盖要求

下一阶段的 versioned pool 至少要覆盖以下样本桶，每一桶都至少包含：

* 1 个“visibility 正常、问题较少”的正样本
* 1 个“visibility gap 明显”的样本
* 1 个“硬缺陷/高风险问题明显”的样本

必须覆盖：

* 施工组织设计
* 一般施工方案
* 危大专项方案
* 监理规划 / 审查辅助材料
* 机电安装类
* 土建类
* 钢结构类
* 临电类
* 起重吊装类

这里不是要求所有桶一次性做到 expert-golden，而是要求 **从 P1 起就按这些桶去建 versioned cases**，避免结构化审查永远围着两三个样本打转。当前 repo 的 legacy 集合已经覆盖五类文档类型，但 versioned seeds 仍只在 construction_org 上起步。([GitHub][35])

### M.5 防过拟合要求

必须明确禁止以下几种“假进步”：

1. **冷轧厂样本特征硬编码。**
2. **只围绕单一行业样本优化。**
3. **只围绕 Gemini 文风优化。** seed README 与 golden pool 指导都已经说明，Gemini 不是 gold truth。([GitHub][38])
4. **把工程增强建议伪装成强制性缺陷。**
5. **把 visibility 错误带入 golden truth。**

落实方式是：

* facts / visibility 独立标注
* hard defect 与 suggestion 分开标注
* 任何由 Gemini 或当前 008 结果派生出的 seed，都必须经过 internal-reviewed 或 expert-reviewed 提升，不能直接升 gold。([GitHub][30])

## N. 非目标与禁止事项（Non-goals & Anti-patterns）

1. **不为单案例硬编码。**
   当前 seed 规模仍小，更要避免围绕冷轧厂/培花样本写 patch。任何单案例“修复”都必须先进入 versioned regression 再判断是否泛化。([GitHub][21])

2. **不只靠 prompt。**
   formal review 不是一个大 prompt；当前 repo 已经有 parse/facts/rules/evidence/report 六层骨架，下一阶段必须继续做结构，不应回退。([GitHub][15])

3. **不只靠更强模型。**
   当前文档和 llm gateway 已明确把 LLM 放在解释层。换更大的模型不能替代 parser、facts、rules、eval。([GitHub][3])

4. **不把系统没读到当成文档缺失。**
   `attachment_unparsed`、`referenced_only`、`visibility_gap` 已经是当前 repo 的明确原则；下一阶段只能把它做实，不能推翻。([GitHub][32])

5. **不只模仿 Gemini 文风。**
   Gemini 与当前 008 结果文件可以作为对照材料，但 golden pool 指导与 seed README 都明确反对把 Gemini 输出当 truth。([GitHub][31])

6. **不把建议性工程增强伪装成强制性缺陷。**
   当前 `FindingType` 已区分 `hard_evidence`、`engineering_inference`、`visibility_gap`、`suggestion_enhancement`。下一阶段要继续利用这层区分，而不是把所有建议都抬成 defect。([GitHub][2])

7. **不先做一个庞大而空泛的多模态平台。**
   在 P0/P1 之前就上复杂多模态，会掩盖当前真正的主线问题：input contract、pack depth、visibility semantics、eval wiring。当前 architecture 文档把多模态列为可扩展方向，而非现阶段核心。([GitHub][15])

8. **不把 `review_assist` 强行包装成 formal review。**
   这是当前 repo 已写进 docs/runtime 的硬边界，下一阶段必须继续保持。([GitHub][39])

## O. Definition of Done

### O.1 功能 DoD

* `structured_review` 可接受 fixture 与上传文档两类输入。
* 对 `construction_org`、`hazardous_special_scheme` 的 formal review 主链稳定可跑。
* `construction_scheme` / `supervision_plan` / `review_support_material` 至少有首批非 placeholder 覆盖进入 P1。

### O.2 数据契约 DoD

* 前后端对 `manualReview`、`visibility`、`evidenceMissing` 字段语义一致。
* `artifactIndex` 仍是唯一可信下载入口。
* `FinalIssue` 不再丢失候选层关键的人工复核原因与证据缺口信息。

### O.3 结果可解释性 DoD

* 每个 final issue 都能回答：
  1）基于哪些 docEvidence；
  2）基于哪些 policyEvidence；
  3）为何命中；
  4）是否只是 visibility gap；
  5）是否需要人工复核。
* UI 不需要依赖 raw JSON 才能看懂这些信息。

### O.4 评测 DoD

* eval 同时覆盖 facts / rule hits / visibility / final issues。
* versioned cases 进入自动加载。
* legacy dataset 继续作为 baseline，但 versioned cases 成为新增能力的主验证格式。

### O.5 回归 DoD

* CI 至少跑稳定子集；本地能跑完整集。
* 有 ablation 报告与 cross-pack / cross-model 报告。
* 任一 parser/rule/contract 改动，都能定位是哪个层退化。

### O.6 UI / artifact DoD

* task detail 页面具备 reviewer 可用的 visibility panel、issue evidence panel、manual review status。
* artifact 不仅可下载，还可被 UI 结构化消费。
* reviewer decision 能被记录、再展示、再回归。

### O.7 文档 DoD

* README、formal-review、testing、known-limitations、architecture 的口径一致。
* supported / experimental / placeholder 的边界明确，不再靠读代码猜测。
* golden pool 版本与晋升规则写进仓库文档。

### O.8 风险边界 DoD

* 系统不会把 `attachment_unparsed` 直接写成“附件缺失”。
* LLM 不会越权生成不存在的法规/文档证据。
* review_assist 与 structured_review 的产品边界稳定可感知。

### O.9 总结性结论

1. **当前 repo 到底处于什么阶段？**
   当前 repo 处于 **“review control plane + `structured_review` 最小可用 formal review 子域”** 的阶段。它已经超过 DTO/scaffold，也明显还没有达到“可较稳定承担正式审查”的成熟度。对 `construction_org`、`hazardous_special_scheme` 可以定性为 **初步可用的 P1 原型**。([GitHub][1])

2. **当前 repo 最真实的优势是什么？**
   真实优势不是模型更大，而是：

   * 双轨边界清楚；
   * formal review 六层流水线已经落地；
   * LLM 被压在解释层；
   * result/artifact/UI/test/eval 骨架已经形成。([GitHub][15])

3. **当前 repo 最真实的短板是什么？**
   最大短板不是“没有 structured_review”，而是 **coverage 与闭环不足**：输入仍 fixture-first，parser 深度不均，packs/rules 多数还浅，visibility/manual-review 语义未完全规范化，golden pool/facts/rule-hits 还没 fully wired 到自动评测。([GitHub][9])

4. **下一阶段最值得投入的 3～5 件事是什么？**
   第一，把 `structured_review` 的输入 contract、L0 visibility、manual review 语义与前后端结果 contract 收敛。
   第二，把 versioned seeds、expected facts、expected rule hits 接进评测主流程。
   第三，把 `construction_org` / `hazardous_special_scheme` 两条主支持链做稳。
   第四，为 placeholder doc types 与 scenario packs 增加首批非空规则。
   第五，把 UI 从只读 viewer 升级为 reviewer cockpit。

5. **这 3～5 件事分别属于什么类型？**

   * 输入 contract / visibility / manual review 收敛：**做实已有骨架 + 解耦职责 + UI/contract 对齐**
   * versioned eval wiring：**评测补齐**
   * 两条主支持链做稳：**做实已有骨架**
   * 首批非空 packs：**新增能力**
   * reviewer cockpit：**UI / contract 对齐**

6. **如果只能优先做一个阶段，最建议先做哪个阶段，为什么？**
   我最建议先做 **P0**。因为当前 repo 的核心价值已经不在“有没有架构”，而在“现有架构是否足够稳、足够可测、足够可解释”。如果 P0 不先完成，P1/P2 只会在不稳的基础上继续堆文档类型、堆规则、堆模型，最后再次回到“看起来更复杂，但没有更可验证”的状态。([GitHub][12])

**一句话结论：**
当前 `008-review-control-plane` 不是空白系统，也不是成熟 formal reviewer；它已经是一个 **边界清晰、骨架真实、但仍需把主支持链路做实的 control plane + formal review P1 原型**。下一阶段最重要的不是再发明新概念，而是把已经落地的 `structured_review` 真正做成 **可解释、可评测、可回归、可人工复核** 的正式结构化审查能力。
