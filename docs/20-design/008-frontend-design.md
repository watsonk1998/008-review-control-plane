# 008 前端交互与页面架构设计

基于业务定位，前端架构采用**单页式审查工作台 (Single-page Dashboard)**，以“先选定审查坑位，全局配置审查边界，再聚合交付最终报告”为核心链路。

## 核心业务决策边界
1. **文件上传方式**：采用“先选择对象坑位，再结构化上传组件”模式，摒弃自由上传（大模型无需再额外识别文件大类）。
2. **Prompt（审查重点）机制**：作用域为**全局生效**。提供开放式多行文本输入，支持配置历史模板的“保存”与“引用”。
3. **输出交互形式**：当前阶段采用**静态交付**（呈现概览视图+风险明细列表并支持导出），远期扩展为“可双向联动高亮的交互式工作台”。

## 核心组件库与联动机理 (Component State Management)

### 1. 级联定位器模块 (Cascader & Slot Mapper)
- **交互**：用户完成 一级 -> 二级 -> 三级 的选择。
- **状态联动**：一旦选中末级节点（如“基坑工程”），触发防抖 API 请求，动态从后台拉取该类型工程关联的「预设规范库」与「预设重点项」以刷新下方的勾选列表。

### 2. 结构化文件上传槽 (Slot-based Uploader)
- **交互**：页面渲染明确的上传卡槽（主文件槽为必选大卡片，辅助依据为标签型次级槽）。
- **数据流承载**：上传接口带参，将文件类型强行与业务类型挂钩（e.g., `file_role=TARGET_DOC`, `file_role=AUX_CONTRACT`），免除大模型分类成本。

### 3. 提示词资产管理模块 (Prompt Template Manager)
- **交互**：带历史记录下拉和保存按键的多行文本框。
- **关联接口**：
  - `GET /api/v1/prompt-templates` (拉取租户下的常用审查侧重点)
  - `POST /api/v1/prompt-templates` (保存当前规则)

### 4. 静态产出与状态工作台 (Static Report Delivery)
- **进度展示**：针对大模型流式切片审查长耗时场景，配合 WebSocket/SSE 展示阶段性处理状态（“图纸解析中...” -> “标准比对中...”）。
- **结果呈现**：交付包含整体分析得分、高危拦截项汇总与清单表格的静态审查简报。
- **动作闭环**：提供标准化的【一键导出 PDF/Word】操作入口。

## 前后端交接契约补充

为了确保大模型长耗时审查任务的稳定分发与前后端解耦，审查流采用**前置文件上传 + 异步 SSE 进度订阅**的主方案。以下为前端提交时刻到结果获取的全生命周期数据契约。

### 1. Review Task 创建请求 (POST /api/v1/review-tasks)

> **注意**：文件需先通过单独的上传接口获取 `file_id`。创建审查任务时只传文件标识与业务元数据，不再重复传二进制文件流。

```json
{
  "task_name": "基坑工程审查任务-20260409",
  "target_category": {
    "level_1": "zxfa",
    "level_2": "wdfa",
    "level_3": "jk"
  },
  "documents": [
    {
      "file_id": "file_xxx",
      "file_name": "某商业中心基坑支护专项施工方案.pdf",
      "file_role": "TARGET_DOC"
    },
    {
      "file_id": "file_yyy",
      "file_name": "项目总承包合同.pdf",
      "file_role": "AUX_CONTRACT"
    },
    {
      "file_id": "file_zzz",
      "file_name": "基坑支护平面图.pdf",
      "file_role": "AUX_DRAWING"
    }
  ],
  "review_config": {
    "preset_standards": ["GB_50202_2018", "JGJ_120_2012"],
    "preset_focuses": ["FOCUS_CALC_CONSISTENCY", "FOCUS_EMERGENCY_PLAN"],
    "custom_prompt": "请重点核对是否满足严禁夜间土方外运的环保要求。"
  }
}
```

**`file_role` 枚举说明：**
- `TARGET_DOC`: 核心待审文件
- `AUX_CONTRACT`: 辅助依据-合同
- `AUX_DRAWING`: 辅助依据-设计文件/图纸
- `AUX_BID_DOC`: 辅助依据-招标文件
- `AUX_GEO_REPORT`: 辅助依据-地勘报告
- `AUX_OTHER`: 辅助依据-其他辅助材料

### 2. Review Task 创建响应

任务创建后将立即返回异步跟踪凭证：

```json
{
  "code": 200,
  "data": {
    "task_id": "task_1001",
    "status": "PENDING",
    "status_url": "/api/v1/review-tasks/task_1001",
    "stream_url": "/api/v1/review-tasks/task_1001/stream",
    "result_url": null
  }
}
```
**字段说明**：
- `result_url`：在任务未完成时为 `null`，完成后将指向静态报告结果。
- `status_url`：用于普通状态查询（如 SSE 断开时的 polling 降级轮询）。
- `stream_url`：用于前端建立 SSE 长连接以订阅实时进度。

### 3. SSE 事件流契约 (GET {stream_url})

前端通过 Server-Sent Events 获取实时审查进度。

> **前端消费约束**：前端应严格根据 `event` 和 `step` 的枚举状态驱动执行态 UI（如控制进度条或节点高亮），不应依赖对 `msg` 文本内容做任何字符串解析。

**1）进度事件示例**
```text
event: progress
data: {"step":"EXTRACT","progress":35,"msg":"正在提取审查依据并映射关键参数"}
```

**2）完成事件示例**
```text
event: done
data: {"step":"DONE","progress":100,"msg":"审查任务已完成","result_url":"/api/v1/reports/rep_1001"}
```

**3）失败事件示例**
```text
event: error
data: {"step":"FAILED","progress":100,"error_code":"DOC_PARSE_FAILED","error_msg":"主文件解析失败"}
```

### 4. 任务状态机

整个审查任务生命周期包含以下枚举状态：

- **`PENDING`**：任务已创建，等待进入大模型或引擎的执行队列。
- **`RUNNING`**：任务正在执行中（含解析、比对、生成报告等子环节）。
- **`SUCCEEDED`**：任务整体执行完成，结果可读取。
- **`FAILED`**：任务整体失败，遇到致命错误，无法生成有效结果。
- **`PARTIAL_FAILED`**：部分处理失败（如某份次要辅助依据无法解析），但主链路审查已完成，有可交付结果。
- **`CANCELLED`**：任务被用户手动或系统超时取消。

**状态流转路径**：
`PENDING -> RUNNING -> SUCCEEDED | FAILED | PARTIAL_FAILED | CANCELLED`

### 5. 结果读取契约 (GET {result_url})

当任务流转为终态时，前端可拉取最终审查结果，渲染静态交付页面。

**1）成功响应**
```json
{
  "task_id": "task_1001",
  "status": "SUCCEEDED",
  "summary": {
    "risk_level": "MEDIUM",
    "issue_count": 5
  },
  "issues": [
    {
      "type": "CALC_ERROR",
      "description": "计算书中嵌固深度参数与图纸不一致"
    }
  ],
  "report_download_url": "/api/v1/reports/rep_1001/download"
}
```

**2）失败响应**
```json
{
  "task_id": "task_1001",
  "status": "FAILED",
  "error_code": "TARGET_DOC_CORRUPTED",
  "error_msg": "核心待审文件无法解析或已损坏，审查终止"
}
```

**3）部分失败响应**
```json
{
  "task_id": "task_1001",
  "status": "PARTIAL_FAILED",
  "summary": {
    "risk_level": "HIGH",
    "issue_count": 12
  },
  "issues": [ ... ],
  "warnings": [
    {
      "code": "AUX_DOC_PARSE_SKIPPED",
      "message": "部分辅助依据（地勘报告）解析失败，审查结论未覆盖该材料"
    }
  ],
  "report_download_url": "/api/v1/reports/rep_1001/download"
}
```
