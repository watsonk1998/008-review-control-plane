《008 Validate PRD v1》

0. 文档信息

文档名称：008 Validate PRD v1
产品代号：008-review-control-plane
文档目标：定义 008 的产品身份、边界、首批 official 主战场、问题项合同、评测口径与治理晋升机制
适用范围：面向审查专家的正式结构化审查初稿生产系统
不包含：详细技术设计、PR 拆分、接口字段逐项说明、前端页面逐页原型

⸻

1. 一句话定义

008 是一个面向审查专家的正式结构化审查初稿生产系统。
它不替代专家作最终裁决，而是把规范检索、事实归并、问题分层、证据追溯与复核入口收敛为一条可治理、可复核、可晋升的审查生产线。这个定义与仓库当前将 008 定位为 review control plane、并将 structured_review 作为正式结构化审查子域的方向一致。 ￼

⸻

2. 背景与问题陈述

2.1 当前行业现实

传统施组/方案审查的核心痛点，不是“没人会写意见”，而是：
	•	前置材料整理成本高
	•	规范检索与适用匹配费时
	•	高风险问题容易漏
	•	证据链组织困难
	•	复核过程缺乏结构化承接
	•	不确定性容易被藏进报告措辞里

2.2 当前产品机会

008 的机会不是做一个“更像专家说话的报告器”，而是把专家最耗时、最难标准化、最难追溯的一层审查工作生产线化。

2.3 当前仓库事实约束

从仓库现状看，008 已经不是单纯的“审查器本体”，而是总控编排平台；正式审查主线已收敛到 structured_review。对外主协议已围绕 `summary / visibility / issues / matrices / artifactIndex / finalReportMarkdown / finalReportPacket / unresolvedFacts` 展开；008 内部结果仍可保留 `reportMarkdown` 作为内部报告字段。 ￼

⸻

3. 产品目标

3.1 主目标

形成 可复核、可采用、可追溯 的正式结构化审查初稿。

3.2 次级目标
	•	降低专家前处理成本
	•	稳定浮出高风险工艺问题
	•	提升正式问题项的证据闭合度
	•	降低 parser 受限场景下的伪确定性输出
	•	为 pack / scope 晋升提供治理底座

3.3 明确不追求
	•	自动签发正式审查结论
	•	通过文风相似度证明价值
	•	在首批 official scope 未做硬前扩成大而全工程 AI 平台

⸻

4. 目标用户与首要使用场景

4.1 首要用户

审查专家

4.2 次要用户
	•	审查团队负责人
	•	内控/质量治理角色
	•	产品治理与评测维护人员

4.3 首要使用场景

专家拿到一份施组/危大方案后，需要：
	•	快速明确适用规范与边界
	•	识别高风险问题
	•	形成结构化问题初稿
	•	查看证据与可视域状态
	•	对问题项做确认、降级/升级、补证据、补依据

4.4 不作为首批主场景
	•	面向施工单位自查的轻量检查器
	•	面向销售展示的“自动出报告”
	•	多模态图纸联审主流程
	•	多文档联合核对主流程

⸻

5. 产品身份与边界

5.1 产品身份

008 的身份是 formal-review 生产系统，不是：
	•	通用文档审阅器
	•	报告写作器
	•	大而全工程 AI 平台

5.2 与 review_assist 的关系
	•	review_assist：快速辅助总结，明确“不等于正式审查结论”
	•	structured_review：正式结构化审查主线，产出 issues、matrices、report、artifacts
这一区分已在仓库 README 中明确写出。 ￼

5.3 与 control plane 的关系

008 不是单一审查 pack，而是 review control plane 下的正式审查子域能力。

⸻

6. 核心价值主张

6.1 对外价值主张

相比传统审查方式，008 最不可替代的是：把高风险工艺问题的发现、规范依据、事实证据、严重度分层与复核入口压成一条可追溯的正式初稿生产线。

6.2 对内价值主张

008 最强的不是写得像专家，而是把专家最耗时的那一层审查生产线化。

⸻

7. 产品原则

7.1 诚实原则

看得见的才下结论；看不见的只能暴露不确定性。

7.2 正式性原则

只有满足问题项合同的问题，才允许进入正式初稿主清单。

7.3 分层原则

候选问题、正式问题、阻塞问题必须分层，不得混写。

7.4 治理原则

ready 不等于 official；scope 晋升属于产品治理，不属于默认扩展。

7.5 生产线原则

产品设计围绕“输入—判定—证据—复核—晋升”闭环，而不是围绕一次性出报告。

⸻

8. 双引擎审查模型

8.1 A 类：闭域可枚举审查项

规则主导，LLM 只辅助表达。
典型包括：
	•	程序触发
	•	编制依据完备性
	•	必备章节缺失
	•	硬性控制项缺失
	•	条文级硬缺陷

8.2 B 类：开域不可穷举审查项

LLM 参与推断，但只能先进入候选问题生成层。
典型包括：
	•	工艺流程逻辑
	•	工序衔接冲突
	•	资源与工期失配
	•	交叉作业叠加风险
	•	现场难落地问题

8.3 三条硬边界
	•	不允许模型凭经验补齐不可见事实
	•	不允许模型单独决定重大隐患成立
	•	不允许模型在证据不足时生成看似完整的正式结论

8.4 一个必要例外

对“工艺流程”等无法完全穷举的对象，允许模型参与“启发式候选发现”，但不得跳过证据与复核直接晋升为正式成立问题。

⸻

9. 输入、可视域与失败哲学

9.1 输入形式

当前 structured_review 输入主入口支持 fixtureId 或 sourceDocumentRef；结果中显式包含 visibility 等结构化字段。 ￼

9.2 可视域哲学

visibility 是正式结果的一等对象，而不是附属说明。
它必须显式承载：
	•	parseMode
	•	parseWarnings
	•	preflight
	•	manualReviewReason

9.3 parser 受限时的默认行为

只输出：
	•	候选问题
	•	阻塞问题
	•	复核提示
不输出：
	•	看似完整的正式结论
	•	伪装成确定事实的判断

9.4 产品口径

“当前不可见”与“文档缺失”不是一回事。

⸻

10. 问题项模型

10.1 正式问题项

已满足正式成立条件，可进入正式结构化审查初稿。

10.2 候选问题项

来自开域推断或部分证据触发，只能进入专家复核候选池。

10.3 阻塞问题项

规范可能适用，但关键事实未闭合、证据不可见或可视域受限，因此不得形成正式结论。

10.4 粒度

外显粒度以 审查意见级 issue 为主；
内在支撑保持条文级、事实级、证据级可追溯。

⸻

11. 问题项合同

11.1 正式问题项最小合同
	1.	事实证据
	2.	适用依据
	3.	问题判断
	4.	严重度/状态
	5.	复核或整改建议

前四项缺一不可。

11.2 候选问题项最小合同
	1.	候选观察
	2.	触发原因
	3.	待核事实
	4.	建议复核方向

11.3 阻塞问题项最小合同
	1.	可能适用的依据
	2.	当前缺失的关键事实
	3.	阻塞原因
	4.	建议补证路径

⸻

12. 严重度体系与重大隐患判定

12.1 严重度分层

建议至少分为：
	•	重大隐患
	•	高风险问题
	•	中风险问题
	•	一般问题
	•	候选高风险
	•	阻塞待核

12.2 重大隐患成立条件

必须满足“双命中”：
	•	规则命中
	•	证据可见且关键事实闭合

12.3 不满足双命中时

最多只能输出：
	•	疑似重大隐患
	•	高风险候选
	•	需人工复核

⸻

13. 规范引用有效性

13.1 错引规范的三类错误
	•	引用对象错
	•	适用落点错
	•	结论支撑错

13.2 评测重点

008 不只要“有引用”，还要“引用有效”。

13.3 产品要求

每个正式问题项都应能回答：
	•	为什么适用这条规范
	•	这条规范如何支持这个判断
	•	这个判断是否足以支撑当前严重度

⸻

14. 首批 official 主战场

14.1 主战场定义方式

首批主战场不以单一作业类型命名，而以 审查专家的高风险工艺审查工作单元 命名。

14.2 三个 official 工作单元

高风险工艺完整性审查
发现工艺链条、边界条件、必要控制项与正式要件缺口。

高风险工艺冲突审查
识别工序、空间、时间窗、资源与专业接口之间的显著冲突。

高风险工艺可实施性审查
识别“纸面可写但现场难落地”的问题；首批以候选问题生成为主。

14.3 首批代表性问题族
	•	程序触发与边界识别
	•	工艺链条缺口
	•	控制措施缺口
	•	资源与时间窗冲突
	•	空间与作业面冲突
	•	现场落地性不足

14.4 首批映射场景
	•	吊装
	•	动火
	•	煤气区域作业
	•	临时用电
	•	后续候选：特种设备、高处作业

14.5 为什么这样定义

这样能同时满足你已经定下的两句话：
	•	把高风险工艺问题稳定浮出来，并形成可复核初稿
	•	把专家最耗时的那一层审查生产线化

⸻

15. 成功指标

15.1 北极星指标

A：目标高风险行业问题下的硬性缺陷稳定命中率
B：结构化审查初稿的专家可采用率

15.2 配套质量指标
	•	错引规范率
	•	正式问题项证据闭合率
	•	候选问题转正式问题的人工复核转化质量
	•	parser-limited 场景下误作正式结论的拦截率
	•	阻塞问题识别准确率

15.3 配套效率指标
	•	专家初审耗时下降比例
	•	单份方案复核路径压缩率
	•	专家补证次数下降比例

⸻

16. 非目标

16.1 不是更像专家说话的报告生成器

价值不以“像 Gemini”或“像专家文风”为标准。

16.2 不是无限扩 scope 的工程 AI 平台

在 official 主链未做硬前，不以新增 doc type、更多 pack、OCR、多模态、图纸联审、多文档联审作为主成功标准。

16.3 不是 parser 不足时仍强出正式结论的伪自动化

当前 official 文档类型与 structured_review 主线已收敛，严格优先 formal spine，而不是盲扩覆盖面。 ￼

16.4 不是“ready = official”的能力包装机

仓库已明确区分 ready/official/experimental。 ￼

⸻

17. 专家工作流

17.1 核心动作
	•	确认/驳回问题项
	•	调整严重度与适用状态
	•	补证据 / 补事实 / 补依据

17.2 非核心动作
	•	润色报告措辞
	•	重写整篇初稿
	•	手工再做一遍规则匹配

17.3 产品设计方向

所有交互都应服务于“复核生产线”，而不是“自由写作”。

⸻

18. 治理与晋升机制

18.1 治理角色
	•	产品经理：提晋升申请、定边界、定对外口径
	•	技术负责人：定实现稳定性与可维护性
	•	审查专家负责人：定业务可用性与可采用性
	•	产品 owner / 评审委员会：定 official capability 最终裁决

18.2 晋升证据
	•	版本化测试通过
	•	代表性 case 评测通过
	•	facts schema / rule pack / issue 合同完整
	•	专家复核认可可采用

18.3 一条红线

ready pack ≠ official capability

⸻

19. 当前范围与未来范围

19.1 当前 official 范围

根据仓库 README，当前 structured_review 的 official documentType 仍限定为 construction_org 与 hazardous_special_scheme。 ￼

19.2 当前 experimental 范围

construction_scheme、supervision_plan、review_support_material 等具备 ready base pack，但不计入 official gate 成功标准。 ￼

19.3 未来扩展原则

未来扩展优先级不按“能不能做”，而按：
	•	是否符合首批主战场问题族
	•	是否能沉淀治理资产
	•	是否能在不牺牲正式性前提下形成正式初稿

⸻

20. 版本验收问题

PRD 验收时，至少要回答这 8 个问题：
	1.	这版是否更接近“正式初稿生产系统”而不是“报告器”？
	2.	重大隐患是否仍坚持双命中？
	3.	候选、正式、阻塞三类问题是否边界清晰？
	4.	规范引用有效性是否进入评测核心？
	5.	parser 受限时是否仍保持诚实输出？
	6.	专家工作流是否围绕确认、分层、补证展开？
	7.	主战场是否围绕高风险工艺完整性/冲突/可实施性三类单元？
	8.	ready 与 official 是否仍被严格区分？
