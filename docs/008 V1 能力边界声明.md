《008 V1 能力边界声明》

0. 文档目的

本文档用于明确 008 V1 的产品能力边界、输出边界、裁决边界与治理边界。
它的目标不是描述“系统可能还能做什么”，而是明确：
	•	008 V1 当前真正承诺什么
	•	008 V1 当前明确不承诺什么
	•	008 V1 在什么条件下必须降级输出
	•	008 V1 如何防止从“正式结构化审查初稿生产系统”滑向“高级报告生成器”或“失控扩张平台”

本文档是产品对内统一口径、对外能力陈述、评测验收与晋升治理的边界依据。

⸻

1. 一句话边界定义

008 V1 的边界，不是“尽量多审一点”，而是“在有限主战场内，稳定产出可复核的正式结构化审查初稿”。

008 V1 不是最终正式审查签发系统，不是无限扩展的工程 AI 平台，也不是以文风相似度取胜的报告生成器。
008 V1 的核心承诺只有一个：

在首批 official 主战场内，把高风险工艺问题的发现、依据、证据、分层与复核入口组织成正式结构化审查初稿。

⸻

2. 产品身份边界

2.1 008 V1 是什么

008 V1 是一个：
	•	面向审查专家
	•	以 formal-review 为中心
	•	以结构化问题项为核心产物
	•	以证据链和复核入口为基本约束
	•	以治理晋升为长期机制

的正式结构化审查初稿生产系统。

2.2 008 V1 不是什么

008 V1 不是：
	•	自动签发正式审查结论的系统
	•	单纯模仿专家文风的报告器
	•	以“大模型读完全文后直接给结论”为主的黑箱审查器
	•	先扩文档类型、再补正式能力的大平台
	•	把 ready 场景包装成 official 能力的能力陈列柜

⸻

3. 用户边界

3.1 首要用户边界

008 V1 的首要用户是审查专家。
产品设计围绕专家提效，而不是围绕施工单位自查、销售演示或泛工作流自动化。

3.2 用户价值边界

008 V1 解决的是专家审查中最耗时、最容易漏、最难追溯的一层工作，包括：
	•	前置材料整理
	•	规范检索与适用匹配
	•	高风险问题浮出
	•	结构化问题项初稿形成
	•	严重度分层
	•	复核追溯与 evidence 包装

008 V1 不承诺替代专家做最终裁决，也不承诺消灭人工复核。

⸻

4. 输出边界

4.1 008 V1 的标准输出

008 V1 的标准输出是：
	•	正式问题项
	•	候选问题项
	•	阻塞问题项
	•	可视域与受限说明
	•	复核入口与补证方向
	•	结构化初稿报告

4.2 008 V1 不输出的东西

在 V1 中，008 不承诺输出：
	•	无条件可签发的正式审查结论
	•	脱离证据链的权威式断言
	•	在事实缺口下仍完整成立的重大隐患结论
	•	对所有高风险作业类型的全面正式覆盖
	•	对图纸、多文档、OCR、多模态场景的稳定正式裁决

4.3 输出等级边界

V1 所有输出都必须明确属于以下之一：
	•	正式成立
	•	候选待核
	•	阻塞待补证

不允许把三者混成一类“看似完整”的意见列表。

⸻

5. 审查对象边界

5.1 A 类审查对象边界：闭域可枚举审查项

这类对象可以进入 V1 的正式主战场。
典型包括：
	•	程序触发与边界识别
	•	编制依据有效性与完备性
	•	必备章节与硬性要件缺失
	•	监测监控、应急、验收、资质等刚性项缺口
	•	条文级硬缺陷

对这类对象，V1 承诺追求：
	•	正式问题项生成
	•	规则命中
	•	证据闭合
	•	可复核初稿

5.2 B 类审查对象边界：开域不可穷举审查项

这类对象允许进入 V1，但其首要目标是候选问题生成，而不是大规模自动正式裁决。
典型包括：
	•	工艺流程逻辑问题
	•	工序前后衔接冲突
	•	资源与工期明显失配
	•	多专业交叉作业叠加风险
	•	纸面可写但现场难落地的问题

对这类对象，V1 的承诺边界是：
	•	可以稳定浮出高价值候选问题
	•	可以提供复核方向
	•	可以在证据补强后晋升
	•	不承诺无条件直接形成正式裁决

⸻

6. 模型边界

6.1 模型允许做什么

模型在 V1 中允许参与：
	•	开域工艺问题的启发式候选发现
	•	事实抽取与归并的辅助处理
	•	结构化结果的可读表达
	•	候选问题的去重与归并
	•	复核建议与整改建议的组织表达

6.2 模型不允许做什么

模型在 V1 中不允许：
	•	凭经验补齐不可见事实
	•	单独决定重大隐患成立
	•	在证据不足时生成看似完整的正式结论
	•	将 heuristic candidate 直接伪装成正式问题项
	•	用措辞自信度掩盖适用依据或事实闭合不足

6.3 一个必要边界说明

对于“工艺流程”等天然难以穷举的内容，V1 允许模型参与发现问题，
但模型在这里的角色是发现候选，不是单独裁决。

⸻

7. 事实边界

7.1 可见事实与不可见事实的边界

V1 必须严格区分：
	•	文档中已被系统有效读取的事实
	•	文档中可能存在但当前不可见的事实
	•	需要外部材料才能确认的事实

7.2 V1 的事实立场

只有已读取、已定位、已可追溯的事实，才允许进入正式问题项。
对未读取、未解析、未闭合事实，V1 只能：
	•	保持空缺
	•	标记阻塞
	•	提示补证
	•	生成候选待核

7.3 绝对禁止

V1 禁止把以下内容当成正式事实：
	•	模型补出来的事实
	•	从常识倒推的事实
	•	从类似工程经验类比得出的事实
	•	实际未解析到的附件或表格内容

⸻

8. 证据边界

8.1 正式问题项的证据要求

正式问题项必须能回答：
	•	事实来自哪里
	•	依据来自哪里
	•	为什么这个依据适用于这里
	•	为什么这个事实支持这个判断

8.2 候选问题项的证据要求

候选问题项不要求证据完全闭合，但必须至少说明：
	•	为什么怀疑这里有问题
	•	缺了什么关键事实
	•	下一步应补什么证据

8.3 V1 不接受的证据状态

以下状态不能进入正式问题项：
	•	事实来源不明
	•	条款来源不明
	•	证据链中断
	•	引用条文与结论之间没有桥接
	•	证据不可见但结论已成立

⸻

9. 规范边界

9.1 V1 承诺什么

V1 承诺的是：
在首批主战场内，尽可能做到规范引用有效，而不是仅仅“引用了规范”。

9.2 V1 不承诺什么

V1 不承诺：
	•	对全行业全部规范体系的一次性完整覆盖
	•	对所有规范条文场景都做到自动化无歧义适用
	•	仅凭模型语言能力解决全部条文适配问题

9.3 错引规范边界

以下三类错误都属于 V1 的边界红线：
	•	引用对象错
	•	适用落点错
	•	结论支撑错

一旦这些错误系统性出现，说明 V1 已经越过正式能力边界。

⸻

10. 严重度边界

10.1 V1 承诺的严重度能力

V1 承诺可以输出：
	•	问题严重度分层
	•	正式 / 候选 / 阻塞状态
	•	高风险问题提示
	•	重大隐患候选提示

10.2 重大隐患成立边界

重大隐患在 V1 中必须满足“双命中”：
	•	规则命中
	•	证据可见且关键事实闭合

缺任一项，都不得自动成立。

10.3 V1 的降级哲学

当无法满足重大隐患成立条件时，V1 必须降级为：
	•	高风险候选
	•	疑似重大隐患
	•	阻塞待核
	•	需人工复核

V1 不允许为追求“看起来更强”而越级裁决。

⸻

11. parser / 可视域边界

11.1 可视域边界是产品主边界

V1 的一个核心能力不是“尽量装作看懂”，而是诚实表达系统看到多少、没看到什么、这些缺口如何影响结论成立。

11.2 parser 受限场景下的边界行为

当存在以下情形时：
	•	PDF text-only
	•	附件未解析
	•	表格未保留
	•	图示不可见
	•	目录/结构信息失真
	•	关键字段未抽取

V1 只能输出：
	•	候选问题
	•	阻塞问题
	•	受限原因
	•	复核提示
	•	建议补证路径

11.3 受限场景下禁止事项

V1 禁止：
	•	把“当前不可见”写成“作者未提供”
	•	把“解析失败”写成“内容缺失”
	•	把“待核事项”写成“正式成立问题”
	•	把“受限场景”包装成“正式审查完成”

⸻

12. 首批主战场边界

12.1 首批主战场定义边界

V1 的首批 official 主战场，不以单一作业名称定义，而以审查专家的高风险工艺审查工作单元定义。

12.2 V1 承诺覆盖的工作单元

V1 承诺聚焦以下三类工作单元：
	1.	高风险工艺完整性审查
	2.	高风险工艺冲突审查
	3.	高风险工艺可实施性审查

其中第三类在 V1 中以候选问题与复核辅助为主，不承诺大规模自动正式裁决。

12.3 V1 首批代表性问题族边界

V1 首批承诺优先围绕以下问题族建设：
	•	程序触发与边界识别
	•	工艺链条缺口
	•	控制措施缺口
	•	资源与时间窗冲突
	•	空间与作业面冲突
	•	现场落地性不足

12.4 映射场景边界

V1 可用以下高风险场景做代表性映射，但这些场景不等于全部能力边界：
	•	吊装
	•	动火
	•	煤气区域作业
	•	临时用电

后续可纳入候选扩展：
	•	特种设备
	•	高处作业
	•	其他高风险作业

12.5 明确不承诺

V1 不承诺：
	•	对所有高风险作业形成同等成熟度的 official 能力
	•	因为某个 pack ready 就自动具备 official 能力
	•	因为某个 demo 表现不错就算场景完成

⸻

13. 能力成熟度边界

13.1 official 的边界

只有同时满足以下条件，能力才能被视为 official：
	•	产品目标对齐
	•	问题项合同成立
	•	规范引用有效
	•	重大隐患双命中
	•	parser 受限时行为诚实
	•	专家可采用率达标
	•	样本评测充分
	•	治理审批通过

13.2 ready 的边界

ready 只表示：
	•	方向正确
	•	基础实现可跑
	•	具备继续打磨价值

ready 不表示：
	•	可对外承诺
	•	可作为正式能力销售
	•	可直接进入 official 口径

13.3 experimental 的边界

experimental 表示：
	•	可试验
	•	可观察
	•	可积累样本与规则

但其输出不构成正式产品承诺。

⸻

14. 评测边界

14.1 V1 评测不是测“像不像专家”

V1 评测的核心不是文风，而是：
	•	问题是否成立
	•	引用是否有效
	•	证据是否闭合
	•	严重度是否可信
	•	可视域是否诚实
	•	复核入口是否明确

14.2 V1 评测不能替代边界声明

即使某些样本上表现好，也不能越过本文定义的边界去扩大战果口径。

14.3 一票否决边界

出现以下任一情况，视为越界：
	•	系统性错引规范
	•	系统性把不可见事实写成确定事实
	•	系统性把 parser 受限样本写成正式完整结论
	•	大量由模型单独决定重大隐患成立
	•	候选、正式、阻塞问题混写
	•	ready 被包装成 official

⸻

15. 组织治理边界

15.1 谁可以定义边界

能力边界属于产品治理，不属于研发局部自由发挥。

15.2 谁可以宣布 official

某项能力要进入 official，至少需要：
	•	产品经理确认边界与对外口径
	•	技术负责人确认实现与稳定性
	•	审查专家负责人确认业务可用性
	•	最终审批角色确认晋升结论

15.3 不允许的治理越界

不允许出现：
	•	研发以“已经实现”为由直接宣布 official
	•	销售以“demo 效果不错”为由提前对外承诺
	•	pack owner 以“ready”为由扩大产品口径
	•	评测样本不足时做正式晋升

⸻

16. 对外表述边界

16.1 允许的表述

允许对外表述为：
	•	面向审查专家的正式结构化审查初稿生产系统
	•	在首批 official 主战场内支持高风险工艺问题的可复核初稿生成
	•	支持候选问题、正式问题、阻塞问题的分层输出
	•	支持专家复核与证据追溯

16.2 禁止的表述

禁止对外表述为：
	•	自动完成正式审查
	•	全面替代专家
	•	支持全部高风险作业正式审查
	•	支持全类型工程文件完整自动审查
	•	已具备图纸 / 多模态 / 多文档联合正式审查能力
	•	所有 ready 场景均可对外正式承诺

⸻

17. V1 结束时应当成立的边界结论

V1 完成后，008 应能明确地说：

我们已经拥有一条受约束的 formal-review 主线。
这条主线可以在有限主战场内，稳定把高风险工艺问题组织成可复核的正式结构化审查初稿；
它知道自己看到了什么、没看到什么；
它能区分正式问题、候选问题、阻塞问题；
它不会为了显得更强而越权裁决；
它还没有扩成一个无限覆盖的一般工程 AI 平台。

如果做不到这几点，就说明 V1 还没有真正收住边界。

18. 本声明的使用方式

本文档应作为以下工作的边界依据：
	1.	PRD 与路线图写作
	2.	官方能力对外陈述
	3.	评测与验收判定
	4.	ready / official 晋升评审
	5.	新场景、新 pack、新能力扩展申请
	6.	销售、交付、研发、评测之间的统一口径
