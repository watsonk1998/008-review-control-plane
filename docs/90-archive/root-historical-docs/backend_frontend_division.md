
## 六、前后端工程分工与接口契约

为确保“008 + Hermes”双路协同架构顺利落地，前后端需明确工程分工与接口边界，隔离界面展示与底层执行复杂度。

### 14. 前端与后端研发职责边界

**【前端工程职责】（侧重交互与展现）**
- **页面交互层**：实现单页工作台的“三级级联选择”，并基于此动态渲染文件上传卡槽（区分待审文件与项目资料）。
- **文件处理层**：将文件上传动作与任务提交剥离。用户上传文件时，即刻调用上传接口换取 `file_id`。
- **状态防抖与长连接**：在提交任务后，妥善维护 SSE (Server-Sent Events) 长连接，将后端的实时进度正确映射到进度条与文案上，处理网络闪断重连。
- **结果与反馈层**：解析最终生成的静态报告并提供 PDF/Word 导出下载；实现 4 种轻反馈的界面操作动作收集。

**【后端工程职责】（侧重编译调度与双路编排）**
- **任务编译 (Task Compiler)**：实现统一收口网关，接收前端发来的配置与 `file_id` 集合，将其封装翻译成标准任务书 `Review Brief`。
- **双引擎并发调度**：维护后台任务队列，同时向 008 引擎和 Hermes 引擎下发任务，并监控两个子任务的执行状态。
- **SSE 状态流转换**：将底层的“解析中”、“底稿提取中”、“双路研判中”等底层执行进度，转化为对前端友好的进度事件推送。
- **融合与报告生成**：等待两路任务结束，触发 Hermes 融合逻辑，并在服务器端生成最终静态报告结构化数据及文件下载地址。

### 15. 核心 API 接口清单 (前后端契约)

基于上述职责，后端共计需提供以下 **6 个核心接口**：

1. **获取预设下拉选项** `GET /api/v1/review-configs`
   - **场景**：用户选中三级分类（如基坑工程）后拉取配置。
   - **返回**：该方案专属的预设审查依据、预设审查重点列表。
2. **通用文件上传** `POST /api/v1/files/upload`
   - **场景**：用户拖拽或选择文件时。
   - **返回**：文件唯一标识 `file_id`（任务提交时不再传文件实体，只传此 ID）。
3. **创建审查任务** `POST /api/v1/review-tasks`
   - **场景**：用户点击【提交审查任务】。
   - **核心入参**：`target_files` (含 file_id), `basis_files`, `focus_pack` (审查重点), `review_policy` (审查口径), `report_type` (报告类型) 等。
   - **返回**：全局唯一任务标识 `task_id`。
4. **SSE 进度订阅** `GET /api/v1/review-tasks/{task_id}/stream`
   - **场景**：创建任务成功后，前端立即建立长连接。
   - **推送机制**：分发 `progress` (进度百分比) 和 `message` (向用户展示的阶段文案)，任务结束时推送结果地址。
5. **获取最终报告** `GET /api/v1/reports/{task_id}`
   - **场景**：SSE 推送完成事件后，前端拉取结果。
   - **返回**：融合后的整体结果、风险明细清单、可信度说明以及文件下载 URL。
6. **提交轻反馈** `POST /api/v1/reports/{task_id}/feedback`
   - **场景**：用户在报告页点击反馈按钮。
   - **入参**：反馈类型枚举（有帮助/问题不准/有遗漏/存模板）及可选附加文本。
