《V0.3 边界声明》完整草案

0. 文档定位

本文不是 V0.3 实施任务书，不是 PR 拆解文档，不是产品愿景宣言，也不是对双样本结果的文本复述。本文的唯一职责，是在 V0.3 启动前，明确本轮的 scope、governance、anti-overfit 原则，以及哪些边界在本轮内被刻意保持不突破。其上位约束来自仓库事实源，直接上游研究输入来自《三角对比研究结果》，同时受《使用说明（给研究型 AI）》与《V0.3 前置工作流程指南——文档生成顺序与依赖关系》约束；其中《三角对比研究结果》是研究底稿，不是治理裁决文档。 ￼

1. 背景与现状

当前项目的真实定位仍是 review control plane，而不是单一 formal reviewer。仓库 README 已将 structured_review 明确为其中一条正式结构化审查子域，其主链为 parse -> facts -> rules -> evidence -> report；同时系统继续保留 review_assist，且 README 明确声明其“不等于正式审查结论”。这意味着 V0.3 的边界讨论必须建立在“控制平面 + 正式审查子链”这一双层现实上，而不能假设项目已经进入成熟全栈 formal reviewer 阶段。 ￼

从正式支持范围看，当前 official documentType 仍只有 construction_org 与 hazardous_special_scheme。construction_scheme、supervision_plan、review_support_material 虽已有 ready base pack，但在 README 与 docs/formal-review.md 中仍被明确放在 experimental 范围，不计入 official gate 主成功标准。与此同时，scenario packs 的状态也已经被清楚区分：lifting_operations.base、temporary_power.base、hot_work.base、gas_area_ops.base 为 ready，special_equipment.base、working_at_height.base 仍是 placeholder；并且仓库已反复强调 ready pack ≠ official documentType。 ￼

从解析与可视域边界看，当前 PDF 路径仍是 pdf_text_only + parserLimited=True 的受限路径。本轮仓库事实只允许在该路径上增加轻量结构提示，例如 appendix/table/figure caption candidates，而不引入 OCR 与多模态；并且明确规定这些 warning 只能表达“可视域受限 / 结构提示”，不能被解释为“正文或附件缺失”。visibility_gap、attachment_unparsed、referenced_only 以及 evidence 不足的情形，仍必须保留人工复核标记。strictMode 虽保留在 contract 中，但当前仍是 reserved / no-op，并不是可扩展为复杂语义系统的既成基础。 ￼

从评测与治理现状看，仓库已经存在 make eval-review、make eval-review-ablations、make eval-review-cross-pack、make eval-review-cross-model 等治理入口；评测池包含 versioned cases，其中只有一部分构成 official CI stage-gate。layeredMetrics 被要求按 L0 / L1 / L2 / L3 / CrossCutting 分组输出，且 L3 当前允许 diagnosticOnly=true。fixtures/research_inputs/latest-eval-summary.md 可以作为阶段性治理状态快照，但 research_inputs README 已明确其“非长期真相”属性：它不能替代 versioned stage gate，也不能因为某次 eval 通过就自动认定差距已关闭或边界可以放宽。 ￼

从 Gemini 的治理位置看，fixtures/review_eval/README.md 已把版本语义写清：v0.1.0-gemini-seed 是 seed labels，不是专家 gold truth；进入更高层级需要 internal review 或 expert confirmation。该 README 同时给出非协商原则：不得把 Gemini 结果直接当专家 truth，必须区分 issue truth 与 visibility truth，必须显式记录 provenance，不得把“未解析附件”直接标成“缺失”。《使用说明（给研究型 AI）》也重复这一边界：Gemini 只能用于候选问题发现、差距探针、seed baseline、bootstrap substitute，不得进入 gold labels、规则答案库或 prompt 风格模板。 ￼

从直接上游研究输入看，《三角对比研究结果》给出的结论是：V0.2 的代际变化主要不在“写得更像专家”，而在 canonical visibility、问题对象语义、pack 边界纪律，以及 rule-hit 到 issue 的闭环；其中 L0 已有明显 contract 升级但 gate 仍未完全前置，L2 是最实质的进步层，而样本 B 的主要短板首先来自 parser/visibility 与 ready pack coverage，而非纯模型“工程脑力”不足。该研究因此支持 V0.3 把重点放在 L0 与 L2，而不是把“追平 Gemini narrative”当成中心目标。 ￼

2. V0.3 边界总声明

V0.3 的任务不是扩张 official scope，不是追求更长、更像专家的报告文本，也不是把系统推向 OCR、多模态、图纸平台化或多文档批处理。V0.3 的任务，是在当前仓库已成立的 structured_review 正式主链之内，优先把 L0 的可视域诚实度、blocked-state 表达、manual-review 触发可信度，以及 L2 的 fact → rule hit → clause applicability → evidence gap 闭环做硬，把已有 official 两类 documentType 的可发布性、可解释性、可治理性拉高。 ￼

因此，V0.3 是一次“边界内加固”，不是一次“边界外扩”。其收边界的理由有三条：第一，仓库事实已经明确 official/experimental/skeleton、ready/placeholder、seed/internal-reviewed/gold 的分层，V0.3 不应违背这些现有治理语义；第二，双样本研究虽高价值，但不足以证明泛化，当前最需要的是机制增强而不是样本追分；第三，样本 B 已证明当 PDF text-only 与 pack coverage 受限时，系统最脆弱的不是文风，而是 visibility 与 grounded evidence 的上限，因此本轮必须优先保护可信度，而不是追求表面“更懂工程”。 ￼

3. V0.3 的范围边界

3.1 documentType 边界

V0.3 的 official 主战场只限于 construction_org 与 hazardous_special_scheme。这是 README 与 docs/formal-review.md 已明确写定的当前正式支持范围，也是现有 official gate 的边界。construction_scheme、supervision_plan、review_support_material 在 V0.3 期间继续保留 experimental / skeleton 路径，可用于诊断、验证、资产准备与治理试验，但不构成本轮 official 成功标准，也不得以“已有 ready base pack”为由被偷渡为 V0.3 主目标。 ￼

3.2 scenario packs 边界

V0.3 允许继续增强现有 ready scenario packs，尤其是 lifting_operations.base、temporary_power.base、hot_work.base、gas_area_ops.base。但这些增强默认只进入 experimental、diagnostics 与 internal-reviewed 准备区，不自动升格 official。pack promotion 仍受 tests / versioned cases / policy evidence / rule coverage 约束；placeholder packs 仍不得因样本表现或文本观感被提前“视同 ready”。V0.3 的 pack 策略是治理增强，不是状态偷跑。 ￼

3.3 能力层边界

V0.3 的主目标聚焦 L0 与 L2。L0 侧重 visibility taxonomy、parser warnings、blocked-state、manual-review 触发与 issue truth / visibility truth 的分离可信度；L2 侧重 facts、rule hits、clause applicability、partial / blocked / hit / pass 等语义与 evidence gap 的闭环表达。L3 可以做可控增强，但只作为从属目标，不以“全面追平 Gemini 的工程推理”作为 V0.3 的目标表述。现有评测规范已经允许 L3 diagnosticOnly=true，而《三角对比研究结果》也表明样本 B 中 L3 收缩首先是事实缺口与可视域受限所致，而不是简单的“不会推理”。 ￼

3.4 技术边界

V0.3 不纳入 OCR，不纳入多模态，不纳入图纸平台化，不纳入多文档批处理，也不把 strictMode 扩写为复杂语义系统。仓库事实已明确当前 structured_review 只支持最小单文件输入，PDF 仍是受限 text-only 路径，strictMode 当前仍是 reserved/no-op。V0.3 可以做的是在既有单文件、text-first、canonical visibility 的边界内加固结构提示、人工复核语义与 evidence discipline，而不是借 V0.3 名义重启一条新技术路线。 ￼

4. V0.3 的治理边界

4.1 promotion governance

任何能力、pack、documentType 或 case version 想进入更高治理层级，必须遵守现有 promotion governance。对 packs 而言，tests / versioned cases / policy evidence / rule coverage 是明确写在 support-scope 与 formal-review 文档中的 promotionCriteria。对 case versions 而言，v0.1.0-gemini-seed 与 v0.1.0-bootstrap-seed 仅是 seed；进入 v0.2.0-internal-reviewed 需要内部人工复核，进入 v1.0.0-expert-golden 需要专家确认。V0.3 不接受“因为 ready 了所以默认 official”“因为文本像了所以默认 internal-reviewed”这类越级判断。 ￼

4.2 eval governance

V0.3 的主治理方式仍是 versioned cases + layeredMetrics + reviewer gate，而不是两份样本报告的文本观感。docs/testing.md 已明确 official CI stage-gate 的 versioned case 数量与 layeredMetrics 的分层要求，也明确 experimental versioned cases 只进入 diagnostics，不提升 skeleton documentType 为 official gate。详情页与 reviewer 页展示的重点也被仓库固定为 resolvedProfile / visibility / unresolvedFacts / artifactIndex / reviewerDecision 以及 attachmentVisibility、ruleHits、conflicts、sectionStructure 等结构化对象，而不是优先看报告 prose。latest-eval-summary.md 只能作为阶段性快照参考，不能替代 versioned stage gate。 ￼

4.3 Gemini seed governance

Gemini 在 V0.3 中只能被用于四类用途：候选问题发现、差距探针、seed baseline、在缺少人工裁决材料时的 bootstrap substitute。Gemini 不能被当作 gold truth，不能被直接转写为规则答案库，不能被当作 prompt 风格模板，也不能因为在某个样本上“更像专家”就被提升为真相裁决器。其原因不是否认 Gemini 的参考价值，而是当前仓库已经将 visibility truth、issue truth 与 provenance 记录纳入正式治理，而 Gemini 结果并不天然满足这些 canonical 约束。 ￼

5. 反过拟合声明

V0.3 将“防止双样本过拟合”视为硬约束，而非附带提醒。冷轧厂样本与培花样本都具有高研究价值，但它们只能证明系统在这两个样本上的现状与差距结构，不能证明泛化能力，更不能直接生成产品边界。V0.3 不以把系统做成“冷轧厂 / 培花双样本高分机”为目标，不围绕项目名、固定数字、特定章节标题、特定附件名做 patch，也不围绕 Gemini 输出做答案蒸馏式优化。 ￼

高风险过拟合方向已经在上游研究中被点明：围绕“10400 立方米”“DN3600 原水管”“高科西路”“微顶管”“承压水”等表面词写专用规则，或者把 Gemini 叙事中的空白位置反向编码为 prompt 模板，都会把系统推向 case-specific patch，而不会提升 formal reviewer 的稳定性。对样本 A 也同理，围绕 50t 汽车吊、7 天停机、37 人、特定重复标题或特定附件名做专用分支，只会产生样本专用补丁，而不是通用机制。 ￼

因此，V0.3 允许的新增能力，必须优先体现为通用机制增强：visibility taxonomy 加固、fact taxonomy 加固、blocked-state 与 manual-review 语义统一、clause applicability 与 evidence gap 闭环加固、reviewer gate 与 promotion governance 加固。V0.3 优先保护的是“在更多样本上更可信”，而不是“在两份样本上更像 Gemini”；优先保护的是已有 official 两类 documentType 的硬度，而不是扩更多 doc types；优先保护的是 visibility / applicability / evidence gap 的可靠性，而不是报告篇幅与叙事饱满度。 ￼

6. V0.3 成功标准

V0.3 的成功，不等于更像 Gemini，不等于扩更多 documentTypes，不等于做更长的报告。V0.3 的成功，等于在既有 official 边界内，让结构化审查结果更可信、更可治理、更可复核、更可发布。 ￼

在 L0 上，成功应表现为：visibility 对象、parse warnings、blocked-state、manual-review 标记与 issue truth / visibility truth 的区分更加稳定；系统在“当前不可视 / 未解析 / 仅被引用”情形下更少误写成“缺失”；PDF text-only 条件下的保守表达更一致。上游研究已经说明，V0.2 在这一层有明显提升，但 gate 尚未彻底前置，因此 V0.3 的成功标准应是把这层从“半兑现”推进到“可作为稳定 stage gate 语言”。 ￼

在 L2 上，成功应表现为：facts、rule hits、issue semantics、partial / blocked / hit / pass、clause applicability 与 evidence gap 的链路更完整，能够更稳定地区分“正文真的缺失”“证据暂不足”“仅能部分适用”“需要人工确认”这些不同状态，而不是统一输出强结论。样本 A 已显示这是 V0.2 最实的进步层，因此 V0.3 应把 L2 从“已有闭环雏形”推进到“更强的可复核条文适用闭环”。 ￼

在 governance 与 eval 上，成功应表现为：official / experimental / diagnostics / seed / internal-reviewed / gold 的边界更难被混淆；versioned cases 与 layeredMetrics 真正成为主裁决机制；reviewer gate 对升级路径形成实约束；experimental 增强不再轻易冒充 official 成果。latest-eval-summary.md 仍可作为快照，但不会替代正式 gate。 ￼

在 anti-overfit 上，成功应表现为：新增改动可以被描述成机制提升，而不是样本词命中；双样本带来的洞察被转译为治理约束与通用资产，而不是被转译为硬编码 patch；面对证据不足时，系统更敢于保守表达、显式标记 manual review，而不是为了追分写出缺乏 provenance 的强判断。 ￼

7. 升级条件与越界条件

未来要突破当前边界，必须满足明确前置条件。若要把 experimental documentType 或 scenario pack 升为 official，至少需要完成：稳定 tests、足够的 versioned cases、明确的 policy evidence、可解释的 rule coverage、通过 reviewer gate 的人工复核，以及与现有 official stage gate 相兼容的 layeredMetrics 表现。在这些条件缺失前，任何 experimental 结果都只能留在 diagnostics 或 internal-reviewed 准备区。 ￼

未来要把 seed 进一步升级为 internal-reviewed 或更高版本，也必须先补人工复核与 provenance 记录。Gemini-seed 可以作为起点，但不能直接升格为 internal-reviewed，更不能越过 internal-reviewed 进入 expert gold。只有在 issue truth、visibility truth、证据来源与裁决理由都被显式记录并经人工复核后，seed 才可能进入更高层级。 ￼

相反，以下情形不允许突破当前边界：仅因双样本文本结果更好看；仅因 Gemini 在某一案例上更像专家；仅因 ready pack 数量增加；仅因某次本地 eval 通过；仅因 prompt 调整后报告更长、更强势。这些都不足以构成 official scope 扩张、pack promotion、或 seed 升级的充分条件。V0.3 明确拒绝这类越界逻辑。 ￼

同样，OCR、多模态、图纸平台化、多文档批处理等方向，只有在另立范围、另建治理、另补评测与 reviewer gate 的前提下，才可能进入未来版本议程；它们不应通过“增强 L0”的名义被悄然塞入 V0.3。strictMode 也不得在缺乏独立治理设计时被扩写成复杂语义系统。 ￼

8. 最终声明

V0.3 的边界定义如下：在不扩 official documentType、不引入 OCR/多模态/多文档平台化、不把 Gemini 伪装为 gold truth、不把双样本观察翻译成专用补丁的前提下，集中加固 construction_org 与 hazardous_special_scheme 两类 official 主战场中的 L0 可视域诚实度与 L2 条文适用闭环，以 versioned cases、layeredMetrics、reviewer gate 与 promotion governance 作为唯一主治理机制；一切增强优先服务于可信度、可复核性与可发布性，而非样本追分、文风模仿或边界失控。 ￼
