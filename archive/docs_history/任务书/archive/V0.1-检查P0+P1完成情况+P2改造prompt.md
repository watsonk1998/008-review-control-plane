## B. P0+P1 完成度检查

### B1. 检查口径说明

本任务书采用以下口径：

* **P0**：双轨边界已经实装；`structured_review` 对当前主支持对象可独立运行；输入解析、visibility、facts/rules/evidence/report/artifacts/API/UI/test 至少形成最小闭环。
* **P1**：packs/readiness、契约收敛、facts/rule-hit 级评测、artifact/UI 消费、文档与 runbook 达到可持续演进状态。
* **本轮并入的 P3**：不是远期平台想象，而是那些已经开始影响 P0/P1 收口、若不并入本轮会导致系统继续停留在“可跑原型但难稳定演进”的项。([GitHub][4])

成熟度定义如下：

| 状态           | 定义                                                        |
| ------------ | --------------------------------------------------------- |
| 已实现          | GitHub 仓库已有真实代码路径，并已进入 route/runtime/UI/test 中至少两层以上的真实消费 |
| 初步可用         | 已有真实执行链路，能在当前支持对象上跑通，但覆盖面、深度或契约稳定性不足                      |

### B2. P0 完成情况

#### B2.1 双轨边界

**已确认事实（GitHub 仓库证据）**

`review_assist` 仍保留；`structured_review` 已经是独立 `TaskType`，并且请求模型里有 `documentType`、`disciplineTags`、`strictMode`、`policyPackIds`、`sourceDocumentRef` 等正式字段。`planner` 为 `structured_review` 生成单独的执行阶段 `parse -> extract -> rules -> evidence -> explain -> report -> finalize`；`router` 为其选择专门 capability chain；`runtime` 也已经对 `review_assist` 与 `structured_review` 走独立分支。前端首页把两者作为不同能力处理，`structured_review` 有独立表单和结果渲染；测试也分别覆盖了 `review_assist` 与 `structured_review` 的关键路径。([GitHub][2])

**未完成什么**

双轨边界已经成立，但并未完全收口。当前 `router` 仍在做 `choose_structured_review_profile()` 和 pack 选择，`pipeline` 内也仍会再做一次 profile 解析，这意味着 formal review 的最终 profile 决策还没有完全下沉为 review 子域的单一事实源。另一个尾项是 API/TS 兼容层里仍保留 `whetherManualReviewNeeded` 这类 alias，说明对外 contract 仍有历史兼容痕迹。([GitHub][5])

**成熟度判断**

* `review_assist` 与 `structured_review` 的**产品边界**：**已实现**
* runtime/router/planner 的**独立执行路径**：**已实现**
* formal review 领域决策是否**完全下沉**：**初步可用，未完全收口**

**当前最薄弱点**

最薄弱的不是“有没有双轨”，而是**final profile / pack resolution 仍不是单点决策**。这会直接影响后续 contract、评测与规则演进。([GitHub][5])

#### B2.2 输入解析 / 可视域

**已确认事实（GitHub 仓库证据）**

当前仓库已经不再是纯 fixture-first。`CreateTaskRequest` 对 `structured_review` 校验 `fixtureId` 与 `sourceDocumentRef` 二选一；`TaskService` 会把 fixture 自动转换为 `SourceDocumentRef`；仓库还存在 `POST /api/uploads/documents` 上传接口。解析结果 `DocumentParseResult` 已包含 `sections`、`blocks`、`tables`、`attachments`、`figures`、`parseMode`、`parserLimited`、`parseWarnings`，以及更明确的 `VisibilityAssessment`/`visibilityReport` 语义；DOCX 走专门结构化 parser，附件可视域可区分 `parsed`、`attachment_unparsed`、`referenced_only`、`missing`、`unknown`。相关测试已覆盖 upload source、attachment visibility、manual review 语义。([GitHub][2])

**未完成什么**

当前解析深度仍不均衡。`.docx` 已有专门 parser，但 `.pdf` 仍主要是文本抽取后进入 `_parse_text_document()`，深度低于 DOCX。其次，parse contract 里同时存在 `visibility` 与 `visibilityReport` 两套表达，虽然语义接近，但仍有漂移风险。更重要的是，`pipeline` 里仍保留 `disable_visibility_check` 的执行选项；这条旁路对 ablation 有价值，但如果不收口边界，就会给“绕过 visibility 问题”留下错误路径。最终结果页能展示 `visibilitySummary`、矩阵和 parse warnings，但当前 `StructuredReviewResult` 仍缺一个与 parse artifact 同级的 top-level canonical `visibility` 对象。([GitHub][6])

**成熟度判断**

* 输入来源（fixture + upload）：**已实现**
* DOCX 结构化解析与 attachment visibility：**初步可用**
* PDF 解析深度：**初步可用，但明显弱于 DOCX**
* visibility 语义在 runtime/report/UI/artifact 间的一致性：**初步可用，未完全做实**

**当前最薄弱点**

最薄弱点不是“没有 visibility”，而是**visibility 已存在但仍未成为全链路唯一 canonical 语义**：存在 `visibility`/`visibilityReport` 双表达、结果对象缺 top-level visibility、旁路开关仍存在。([GitHub][7])

#### B2.3 最小规则核 / 结构化闭环

**已确认事实（GitHub 仓库证据）**

review 子域已经真实存在，并且 `pipeline` 已经串起 parse、facts、rule hits、candidates、final issues、matrices、report 与 artifacts。当前官方支持范围由 `support_scope` 明确收敛到 `construction_org` 与 `hazardous_special_scheme` 两类文档。规则引擎也不是空壳：`construction_org` 路径已覆盖结构完整性、重复章节、附件可视域、专项方案挂接缺口、应急预案针对性、停机窗口/人力资源冲突；`hazardous_special_scheme` 路径已覆盖危大专项方案核心章节、附件可视域、验算依据/计算书、应急预案针对性、措施联动等。`EvidenceBuilder` 会生成 doc/policy evidence 与 manual review reason；结果页能展示 issue、matrices、reportMarkdown、artifacts；测试也覆盖了这些关键问题链路。([GitHub][8])

**未完成什么**

当前“最小规则核”是成立的，但只对**两类官方支持对象**成立。`construction_scheme.base`、`supervision_plan.base`、`review_support_material.base` 仍是 placeholder；全部 scenario packs 也是 placeholder，`ruleIds=[]`。这意味着系统已经有最小结构化闭环，但并未进入“可持续 pack 扩展状态”。此外，虽然 `FinalIssue` 现在已经保留了 `manualReviewNeeded`、`evidenceMissing`、`manualReviewReason` 等关键信息，但 packReadiness、L0 visibility 与 unresolved facts 仍没有在最终结果上形成更清晰的一等对象。([GitHub][9])

**成熟度判断**

* facts → rules → evidence → final issues → matrices → artifacts → UI：**已实现**
* 最小规则核：**初步可用**
* pack 可扩展性：**有骨架，未做实**
* 对非官方支持对象的正式审查能力：**有骨架，不应宣称已完成**

**当前最薄弱点**

最薄弱点不是“没有规则核”，而是**规则核只覆盖主支持对象，pack 体系远未到可持续演进状态**。([GitHub][8])

#### B3.2 artifacts / matrices / API / UI 消费

`artifactIndex`、`/api/tasks/{taskId}/artifacts`、artifact download API、前端 artifact list、result route serialization 都已经存在；task detail 也能展示 summary、resolved profile、issues、matrices、report、artifact list、raw JSON，并通过 SSE + fallback polling 跟踪任务状态。([GitHub][10])

#### B3.3 eval / harness / seed-golden-versioned assets

**成熟度判断**

* 可执行 unit/integration/eval harness：**已实现**

## H. P3 的范围界定与并入原则

### H1. 当前仓库语境下，什么算 P3

在当前仓库语境下，P3 不是“未来平台化大图景”，而是**超出 P1 原型收口、但已经开始影响下一轮可用性的增强项**。具体包括：

1. 最小 reviewer decision / write-back 语义。
2. 首批非 placeholder scenario packs。
3. versioned case 治理与 stage-gate 口径统一。
4. parser provenance/PDF 受限能力的明确抽象。

这些项已经开始直接影响 P0/P1 收口，所以不能再完全后置。([GitHub][9])

### H2. 为什么这些是 P3，而不是仍属于 P1 的尾项

* **reviewer decision**：当前系统已经有 manual review 提示；再往前一步是存储最小复核结论，这超出“只读消费”，因此属于 P3，但又必须尽早并入，否则 manual review 永远停留在提示层。
* **scenario packs**：当前 pack registry 已有骨架；从 placeholder 走到首批真实规则，是超出 P1 最小闭环的一步，但不做又会让 pack 体系长期停在名义层。
* **versioned governance**：评测框架已有；把 bootstrap seed 提升为更可信的 stage-gate，是 P3 的演进，但不并入就无法稳定推进 P1。

### H3. 本轮必须并入的 P3

1. 最小 reviewer decision state。
2. 2~3 个首批非 placeholder scenario packs。
3. versioned case 与 docs/harness 口径统一。

### H4. 本轮不应并入的 P3

1. 完整 reviewer cockpit / 分配 / 审批流。
2. 图纸、多模态、OCR、扫描件大平台。
3. 全行业 / 全文档类型 / 全场景覆盖。
4. rule authoring DSL / 平台化运维后台。

### H5. 并入原则

* 只并入那些**直接降低当前 P0/P1 风险**的项。
* 不并入会导致 contract、pack、manual review、eval 长期停在“半成品”的项。
* 所有并入项都必须能绑定到当前 repo 的真实文件、真实 tests、真实 eval。

**执行顺序原则**：
先收敛 contract 和 profile，再做 parser/visibility，再扩 packs/rules/evidence，再做 report/UI，再做 eval/docs，最后才并入 reviewer decision 这种轻量 P3。这样能避免“先做交互，再倒逼补后端”的错误顺序。([GitHub][4])

## K. 测试、评测与回归方案

### K1. 现有命令：继续沿用，但要补强

当前仓库已有以下命令，应继续沿用：

* `make test`
* `make test-review-unit`
* `make test-review-integration`
* `make eval-review`
* `make eval-review-ablations`
* `make eval-review-cross-pack`
* `make eval-review-cross-model`
* `make smoke`
* `make verify-connectivity` ([GitHub][12])

### K2. 本轮必须新增或做实的断言

1. **visibility 语义**

   * `StructuredReviewResult.visibility` 与 parse artifact `visibility` 一致。
   * `summary.visibilitySummary` 仅为 display summary，不再是第二事实源。
   * `disable_visibility_check` 不能进入正常任务路径。

2. **facts 抽取**

   * versioned cases 中 `expected_facts.json` 必须进入主报告。

3. **rule hits**

   * rule-hit 数量、ruleId、packId、readiness 至少对 official cases 可比对。

4. **issue 结果**

   * `FinalIssue` 必须保留 `manualReviewNeeded`、`evidenceMissing`、`manualReviewReason`。

5. **severity / layer**

   * `high_severity_issue_recall` 与 `severity_accuracy` 继续作为硬门槛之一。

6. **manual review flag**

   * issue-level 与 summary-level 一致；route serializer alias 不得成为主字段。

7. **artifact index**

   * `result.artifactIndex` 与 `/artifacts` route 一致。

8. **UI contract**

   * TS 类型不再依赖 alias；task detail 能在不看 raw JSON 的情况下呈现核心结果。

9. **API contract**

   * `fixtureId XOR sourceDocumentRef`、upload metadata、support-scope readiness、result serializer 全部有回归测试。

10. **reviewer decision（本轮并入的 P3）**

* issue-level decision 可写可读。

### K3. 本轮必须更完整的指标

基于当前 `metrics.py` 已有基础，本轮要把以下指标真正纳入稳定门槛：

* 问题召回率（issue recall）
* 重大问题命中率（high severity recall）
* 危大识别命中率（hazard identification accuracy）
* 硬证据准确率（hard evidence accuracy）
* 附件可视域正确率（attachment visibility accuracy）
* 严重度校准准确性（severity accuracy）
* `manualReviewNeeded` 准确性
* facts 层最小可测性（facts accuracy）
* rule-hit 层最小可测性（rule hit accuracy）
* pack 选择准确率 / ready-vs-placeholder 覆盖成熟度 ([GitHub][16])

### K4. 端到端 / 集成 / 单元的推荐分工

* **单元测试**：parser、attachment indexer、profile resolver、rules、evidence builder。
* **集成测试**：pipeline、task service、routes、artifact route、upload route。
* **端到端测试**：fixture structured_review、upload structured_review、review_assist 不回归。
* **评测**：legacy baseline 继续保留；versioned cases 负责 facts/rule-hit/visibility stage-gate。

### K5. ablation / cross-pack / cross-model 的保留策略

这三类接口已经存在，不应该废弃；但本轮要明确：

* `ablation` 只能用于定位退化，不得作为“功能通路”。
* `cross-pack` 仅比较 ready packs 与首批 scenario packs，不拿 placeholder packs 伪造覆盖。
* `cross-model` 只能替换 explanation layer，不允许模型改写 facts/rules 裁判。([GitHub][17])

## L. Definition of Done（本轮）

本轮完成必须同时满足以下条件：

1. **P0 未完成项明确收口**

   * `structured_review` final profile 决策只有一处。
   * `StructuredReviewResult` 有 canonical visibility。
   * route/UI 内部不再把 alias 当主字段。
   * runtime 默认路径不再允许 visibility bypass。

2. **P1 未完成项明确收口**

   * packs readiness 与 support scope 一致；至少 2~3 个 scenario packs 非 placeholder。
   * facts/rule-hit/visibility/final issues 都进入稳定评测输出。
   * docs/testing/architecture 对评测池与命令口径一致。

3. **本轮并入的 P3 达到明确可验收状态**

   * reviewer decision 能保存、读取、展示。
   * 不是完整工作流，只是最小闭环。

4. **主支持对象更稳定**

   * `structured_review` 在当前 official support 对象上更稳运行。
   * `review_assist` 不回归损坏。

5. **契约更完整**

   * parser / visibility / facts / rules / evidence / report 契约贯通。
   * `FinalIssue[]` / matrices / artifactIndex / UI contract 稳定。

6. **测试 / eval / docs 同步更新**

   * 相关 tests 全部通过。
   * `make eval-review*` 有稳定输出。
   * 文档不再与代码口径冲突。

7. **验收方式**

   * 不允许通过“把 prompt 写更长”过验收。
   * 所有验收都必须能绑定到结构化中间产物、规则、UI 与评测结果。

## M. 风险、边界与开放问题

### M1. 本轮必须解决的

1. profile 决策重复。
2. visibility canonical contract 缺失。
3. `disable_visibility_check` 旁路未收口。
4. docs 与 eval 口径不一致。
5. placeholder packs 长期停在 registry-only。
6. reviewer decision 仍为零。

### M2. 本轮暂不解决、但必须显式挂起的

1. 把 `construction_scheme`、`supervision_plan`、`review_support_material` 全部升为 official。
2. 全量 scenario packs 覆盖。
3. 图纸/扫描件/OCR 深层能力。
4. reviewer 工作流平台化。

### M3. 再往后阶段才处理的

1. 多模态/图纸平台。
2. 完整 reviewer assignment / audit / write-back 平台。
3. 全行业 pack 运营体系。
4. 通用 rule authoring DSL。

## N. 禁止事项 / 反模式

1. 不得把已存在的 `structured_review` 骨架误写成“从零开始”。
2. 不得把已存在的 artifact API、review schema、upload/sourceDocumentRef、tests/eval 命令再写成“新增概念”。
3. 不得为了单一冷轧厂样本写正则/关键词 patch。
4. 不得把 `review_assist` 包装成 formal review。
5. 不得让 LLM 重新成为 formal review 的第一性裁判。
6. 不得通过“更长 prompt / 更大模型”掩盖 parser、facts、rules、evidence 的 contract 问题。
7. 不得把“未解析附件 / 系统不可视”直接输出成“文档缺失”。
8. 不得在 UI 层写业务判定规则去补后端 contract 漏洞。
9. 不得把未来平台化能力无边界塞进本轮任务。
10. 不得把 placeholder packs 的存在误报成“已支持该文档/场景正式审查”。
