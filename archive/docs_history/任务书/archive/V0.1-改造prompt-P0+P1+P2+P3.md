独立检查后的结论很明确：当前仓库绝不是“还没有 structured_review / 还没有 review 子域 / formal review 仍完全靠大 prompt 顶着跑”的空白状态。相反，仓库已经明确建立了 `review_assist` 与 `structured_review` 的双轨边界，后者已有独立的 review 子域、稳定的 `sourceDocumentRef` 输入、`parse -> facts -> rules -> evidence -> report` 主链、artifact API、top-level canonical `visibility`、`manualReviewNeeded` 语义、reviewer decision API/UI、SSE 任务流，以及针对 structured_review 的单元/集成/评测回归。也就是说，《V0.1-分析2》中相当一部分“P0 应建设”的内容，今天已经是仓库现状，不应再重复发明。([GitHub][1])

但这并不意味着《V0.1-分析2》已经被完全做完。当前剩余问题主要集中在四类：一是主支持范围仍窄，官方支持的 `documentType` 只有 `construction_org` 与 `hazardous_special_scheme`，其余文档类型仍是 skeleton；二是 parser/visibility 虽已做实，但 PDF 仍明确处于 `pdf_text_only + parserLimited=True` 的受限路径，表格/图片/附件可视域仍需更稳的降级表达；三是 packs/rules 虽不再是纯 registry，但仍有三类 base pack 与三类 scenario pack 处于 placeholder；四是 versioned cases、facts/rule-hit 评测虽然已经接入主流程，但官方 stage-gate 池仍偏小，尚不足以支撑更大范围的“正式支持”扩容。([GitHub][2])

因此，本任务书的范围非常克制：不新增一个“新的 structured_review”，不重写 review 子域，不重复设计 artifact API，不把 `review_assist` 包装成 formal review，也不靠更大模型或更长 prompt 去掩盖 contract 问题。它只针对已经核实存在的剩余缺口给出文件级施工清单，并明确说明：实际改造执行者是具有本地文件权限的 Codex；Codex 将在本地工作区 `/Users/lucas/repos/review/008-review-control-plane` 内落地修改；本任务书的作用是帮助 Codex 在该目录中做“查漏补缺、收敛、贯通、补齐、验收”，而不是从零发明一个新系统。([GitHub][3])

### D1. 本次要做什么

本次只做《V0.1-分析2》中**仍未完成或仅部分完成**的部分，具体包括：

2. **做强 parser/L0 可视域路径**：保留当前 top-level `visibility` 为唯一事实源，同时增强 PDF 受限路径的 parseWarnings、标题/附件/表格提示与 UI 展示，不把“系统不可视”伪装成“文档缺失”。([GitHub][6])

4. **扩 versioned stage-gate 与细粒度评测**：不是重写 harness，而是在现有 facts/rule-hit/visibility wiring 上扩大 case pool 和支持范围。([GitHub][7])
5. **把最小 reviewer workflow 做成 contract-first cockpit**：前端继续消费 `visibility`、`issues`、`matrices`、`artifactIndex`、`reviewerDecision`，减少对 summary prose 或 raw JSON 的依赖。([GitHub][14])

### D2. 本次不做什么

本次明确不做以下内容：

* 不做“再新增一个 structured_review”。
* 不做与当前 GitHub repo 脱节的整套“理想系统重写”。
* 不做全行业 / 全场景一次性覆盖。
* 不做 case-specific patch，不允许看到“煤气区域”“汽车吊”“某冷轧厂固定表述”就写死规则。

这些边界与当前仓库文档已经确立的原则一致：LLM 只做解释层，不做第一性裁判；系统不得把未解析附件直接写成缺失；support-scope 必须诚实表达官方支持范围。([GitHub][2])

## E. 目标状态（To-Be for remaining scope）

本轮完成后，系统应达到的状态不是“支持所有文档类型”，而是“把已经存在但未收口的 formal review 主链补完”。具体目标如下：

第一，`construction_org` 与 `hazardous_special_scheme` 继续保持当前官方支持地位，但它们的 parser/visibility/manual review / report / eval 契约将更稳；PDF 受限路径的降级事实会被更清楚地表达出来，避免误判。([GitHub][2])

第二，`construction_scheme.base`、`supervision_plan.base`、`review_support_material.base` 不再只是空 registry；它们至少要进入“可执行但仍非 official”的 experimental 层级。这里的原则不是立刻把 documentType 升为 official，而是先把 empty placeholder 变成可测、可回归、可明确标识 readiness 的最小 pack。([GitHub][19])

第三，`visibility`、`manualReviewNeeded`、`unresolvedFacts`、`artifactIndex`、`reviewerDecision` 会继续作为主 contract，前端和 API 不再把 summary text 或 legacy alias 当作第二真相源；`whetherManualReviewNeeded` 仅允许留在历史任务回放兼容层。([GitHub][1])

第四，评测闭环会从“已经接线”提升到“覆盖更完整”：versioned cases 的 doc type/pack 覆盖更广，facts/rule-hit/visibility/manual-review 的 stage-gate 不再只盯住极少数 case。当前 harness 保留，但 gate pool 扩容。([GitHub][7])

第五，reviewer workflow 不追求一步做成完整运营平台，但至少要达到“reviewer 看 canonical panels 即可理解结果并写回最小复核决策”的程度，减少依赖 raw JSON。([GitHub][14])
