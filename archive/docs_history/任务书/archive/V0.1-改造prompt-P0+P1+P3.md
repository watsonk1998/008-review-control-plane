### G2. 为什么这些是 P2，而不是仍属于 P1/P3 尾项

* **P1/P3 尾项** 的核心是“把已有 canonical behavior 收口”
* **P2** 的核心是“在不扩平台边界的前提下，增加一层质量深度或演进能力”

例如：

* 去掉 alias/backfill、统一 support-scope、补 reviewerDecision route/test，这些是 **P0/P1/P3 尾项**
* 让 evaluator 直接评 `ExtractedFacts` / `RuleHit[]` / `unresolvedFacts`，以及让 L2/L3 在 official scope 上更深一点，这才是 **P2**

### G3. 本轮必须并入的 P2 项

1. **direct facts / rule-hit / unresolvedFacts evaluation**
2. **official scope + ready scenario packs 的有限 L2/L3 深化**
3. **reviewer decision 从 JSON 持久化提升到稳定最小状态流**
4. **parse provenance seam，为未来 PDF/OCR/多模态深化留口**

### G4. 不应并入本轮的 P2 项

* 全 doc type / 全行业 pack 覆盖
* 多文档批处理平台
* 图纸/图像/OCR/多模态完整平台
* 多 reviewer 权限、审计、分配、运营化平台
* rule authoring 平台
* human labeling / feedback ops 全家桶

### G5. 并入原则

1. **必须是当前 GitHub 结构已能承载的增强，不是重写架构**
2. **必须直接解决当前 repo 的演进阻塞点**
3. **必须能在 tests / eval / docs 中被验收**
4. **必须避免把任务书膨胀成未来平台 roadmap**
5. **必须坚持 official scope first，而不是盲目扩面**

## K. 测试、评测与回归方案

### K1. 现有命令的沿用与补强

当前仓库已经有结构化审查相关的测试/评测命令，本轮不应重新发明命令体系，而应在现有命令上补强断言与门槛。([GitHub][8])

| 命令                             | 当前状态    | 本轮动作                                                                                                  |
| ------------------------------ | ------- | ----------------------------------------------------------------------------------------------------- |
| `make test`                    | 继续作为总入口 | 保留；确保 structured_review 与 review_assist 都不回归                                                          |
| `make test-review-unit`        | 已存在     | 增 canonical visibility、no-alias-in-executor、PDF downgrade、manualReviewReason/evidenceMissing 断言       |
| `make test-review-integration` | 已存在     | 增 sourceDocumentRef upload、reviewer decision、artifactIndex authoritative、resolvedProfile ownership 断言 |
| `make eval-review`             | 已存在     | 继续跑 legacy stable subset + versioned stage-gate；补 direct facts/rule-hit/unresolvedFacts 输出            |
| `make eval-review-ablations`   | 已存在     | 保留 parser/visibility/rule/LLM explanation ablations，但明确 internal-only，不可对外暴露                          |
| `make eval-review-cross-pack`  | 已存在     | 只在 ready packs 上比较 auto vs forced expected packs                                                      |
| `make eval-review-cross-model` | 已存在     | 只替换 explanation model，保持 facts/rules 不变                                                               |
| `make smoke`                   | 已存在     | 用于 task create → runtime → artifacts → detail page 基本链路                                               |
| `make verify-connectivity`     | 已存在     | 原样保留，用于环境/adapter 连通性，不承担 review 正确性验收                                                                |

### K2. 本轮至少要新增或做实的断言

1. **visibility 语义**

   * 新结果必须始终带 top-level `visibility`
   * public API 不接受 `disable_visibility_check`
   * internal ablation only 的路径必须被测试隔离
   * `attachment_unparsed / referenced_only / missing / unknown` 不得被 UI/报告偷换成“附件缺失”

2. **facts 抽取**

   * official scope 的关键 facts 需要 direct regression
   * `unresolvedFacts` 需要有 case-level 回归
   * facts artifact 与 evaluator 输入需一致

3. **rule hits**

   * `RuleHit[]` 的 packId/readiness/status/matchType/layer/severity 需稳定
   * forced pack 与 auto pack 的差异需可测

4. **issue 结果**

   * `manualReviewNeeded`、`evidenceMissing`、`manualReviewReason` 必须在 API/UI 一致
   * `whetherManualReviewNeeded` 只允许兼容层，不允许新 UI 依赖

5. **severity / layer**

   * high severity recall、L1 hit rate、hazard identification 需要继续保持 stage gate

6. **artifact index**

   * `artifactIndex` 与 `/api/tasks/{id}/artifacts` 必须一致
   * task service fallback 只用于老任务兼容

7. **reviewer decision**

   * route 写回、store 持久化、detail 页回显、task/issue/attachment 三层状态一致
   * 至少有一组 route + UI + persistence 的回归

### K3. 本轮必须更完整的评测指标

当前仓库已经有这些指标，本轮要做的是让它们更“直接”和更可解释，而不是重新起名。([GitHub][24])

* 问题召回率（`issue_recall`）
* 重大问题命中率 / 高严重度问题召回
* 危大识别命中率（`hazard_identification_accuracy`）
* 硬证据准确率（`hard_evidence_accuracy`）
* 附件可视域正确率（`attachment_visibility_accuracy`）
* 严重度校准准确率（`severity_accuracy`）
* `manualReviewNeeded` 准确性
* facts / rule-hit 层级的最小可测性
* `unresolvedFacts` 的最小可测性
* pack 选择准确率 / readiness 一致性
* reviewer decision 的最小可测性

### K4. 版本化样本与 regression baseline 策略

* **继续保留 legacy stable subset**，不在 versioned pool 成熟前贸然废弃
* **versioned cases 作为本轮新增能力主验证格式**
* 先修复 README / testing.md / dataset inventory 的口径差异
* 新增样本优先覆盖：

  * official scope 的 upload path
  * PDF text-only downgrade
  * visibility gap
  * unresolved facts
  * reviewer decision 最小闭环
* 不允许把 Gemini seed 直接当 gold truth；versioned labels 必须保持 bootstrap / internal-reviewed / expert-golden 分层。([GitHub][18])

## M. 风险、边界与开放问题

### M1. 本轮必须解决的

* route/schema 的 alias/backfill 债务
* internal-only visibility override 与 public API 的边界
* support-scope / docs/testing / fixtures inventory 口径不一致
* official scope 主链的 parser/visibility/contract 收口
* reviewer decision 的最小稳定性与回归
* eval 对 facts/rule-hit/unresolvedFacts 的直接性不足

### M2. 本轮暂不解决、但必须显式挂起的

* 多文档批处理
* artifact / object storage 作为新 source type
* OCR/图纸/复杂多模态完整平台
* 多 reviewer 协作/权限/审计全流程
* 全 doc type official coverage
* 全 scenario packs ready coverage

### M3. 再往后阶段处理的

* pack authoring 平台
* human labeling / feedback ops 平台
* 大规模运营化 dashboard
* 自动化 reviewer assignment / SLA / queue
* 跨项目知识图谱式审查平台

### M4. 仍需显式跟踪的开放问题

1. `whetherManualReviewNeeded` 的兼容窗口保留多久
2. `summary.visibilitySummary` 是否长期保留为摘要对象，还是进一步瘦身
3. `SourceDocumentRef.sourceType` 未来是否扩为 `artifact` / `object_store`
4. versioned dataset inventory 以 docs/testing 还是 fixtures README 为权威展示口径
5. `strictMode` 何时从 reserved/no-op 变为真实语义
