前端也已经把两者分开：首页将 structured_review 标注为“正式审查”，并显式提示当前仍是 fixture-first、P0 正式支持范围仅 construction_org / hazardous_special_scheme；review_assist 侧则明确“不提供正式结论”。结果页对 StructuredReviewResult 有专门类型守卫与专门渲染，不是把 structured_review 当普通文本结果展示。 ￼

测试层也覆盖了双轨关键路径：apps/api/tests/test_runtime.py 同时校验 review_assist 和 structured_review；apps/api/tests/test_task_service_and_routes.py 覆盖结构化任务元数据与 artifact/stream 路由；apps/api/tests/test_structured_review.py 则覆盖 parser、executor、最终 issues 与 matrices。 ￼

当前成熟度判断
P0-1 不是未开始，而是“已实现、但边界未完全收口”。 双轨产品边界、API 路径、前端入口、runtime 分支、测试回归都已经存在，因此不能再把“新增 structured_review / 新增 review 子域 / 新增 artifact API”写成本轮新增项。 ￼

未完成项
当前双轨边界最薄弱的地方有四个。

第一，structured_review 仍然是 fixture-first：runtime 的 _run_structured_review() 明确要求 fixtureId；首页表单也要求 structured_review 提交时必须有 fixtureId。这说明 task type 独立了，但输入边界仍未从“评测/样本驱动”升级为“真实文档输入对象驱动”。 ￼

第二，review profile 的决策权仍分散在 router / planner / pipeline 三层：router 会先猜 document type / discipline tags / packs；planner 把这些写进计划；pipeline 又根据 parse facts 做最终 resolvedProfile。这不是错误，但当前存在“前置猜测”和“最终裁决”两套并行逻辑，trace 清晰度不够。 ￼

第三，artifact contract 存在“两套真相”：pipeline 会写出 parse / facts / rule-hits / candidates / result / matrices / markdown report 等多类 artifacts，但 artifactIndex 只收录结果/矩阵/报告子集；而 TaskService 的 /artifacts 列表则扫描整个任务目录。前端结果页又会优先信任 artifactIndex。这会导致“结果内看到的 artifact 集”与“目录里实际存在的 artifact 集”不一致。 ￼

第四，strictMode 已经在请求、持久化、planner、resolvedProfile、UI 中全链路透传，但在已检查的 rule engine / evidence / report 代码里没有看到它对命中逻辑或结论阈值产生明确分支作用。合理推断：它当前更像 contract 字段，而不是已经做实的策略开关。 ￼

当前边界最薄弱点
P0-1 当前最薄弱的点不是“有没有双轨”，而是：structured_review 的独立性已经有了，但其输入对象与最终 profile 裁决还没有完全摆脱 control-plane 层的前置猜测和 fixture 依赖。这决定了本轮不能重建双轨，而应收口边界。 ￼

B3. P0-2 输入解析 / 可视域完成情况

已确认事实（GitHub 仓库证据）
输入解析层并非“只抽一坨纯文本”。DocumentParseResult 已包含 sections / blocks / tables / attachments / figures / normalizedText / preview / visibilityReport / parseWarnings；DOCX 解析器会保留段落、表格、图片/figure 占位、附件标题与附件可视域，并在顶层重复章节标题时记录 duplicateSectionTitles。 ￼

可视域语义也已经不是空白。AttachmentVisibility 在 domain model 中有 parsed / attachment_unparsed / referenced_only / missing / unknown 五态；attachment_indexer.py 会依据“引用但无正文”“明确缺失标识”“标题已识别但正文不可见”“parser_limited 情况下仅能判 unknown”等规则生成状态，并把 manualReviewNeeded、reasonCounts、各状态计数写入 visibilityReport。因此，GitHub 当前仓库已经明确承认“系统没看到”和“文档真的缺失”是不同语义。 ￼

这些语义也已经被消费：pipeline 会把 visibilityReport 写入 summary / matrices / reportMarkdown；UI 会展示 visibility summary、attachment visibility matrix、manual review 提示；测试直接断言 attachment_unparsed、duplicate section title、manualReviewNeeded 等行为。 ￼

当前成熟度判断
P0-2 不是“还没有 visibility 语义”，而是“visibility 已经成型，但仍未收敛成一等契约”。 当前 DOCX 路径已经足够证明 parser / visibility 不是口号；真正未做实的是 PDF 深度、L0 对象化、以及绕过路径治理。 ￼

未完成项
第一，PDF 仍然是文本抽取级别。document_loader.py 对 PDF 只做 pdfplumber 文本提取，再回落到 _parse_text_document()；当前明确给出 pdf_text_extraction_only、pdf_tables_not_preserved、pdf_attachment_visibility_may_be_unknown、pdf_figures_images_not_parsed 等 warning。也就是说，GitHub 仓库并没有把 PDF 做到与 DOCX 等级相同的结构解析。 ￼

第二，L0 可视域虽然事实存在，但还不是一等对象。当前 visibility 主要散落在 DocumentParseResult.visibilityReport、FindingType.visibility_gap、RuleHit.matchType、summary.visibilitySummary、manualReviewNeeded 与 matrix/report 中；缺少一个稳定、单独、被 API / UI / eval 共享的 VisibilityAssessment 顶层 contract。 ￼

第三，代码内部确实存在“关闭 visibility check”的路径：StructuredReviewExecutor._apply_execution_options() 支持 disable_visibility_check，会把全部附件强制改成 parsed 并清空 manual review；评测 harness 也把它作为 ablation 变体。已确认事实是这个开关存在于代码和评测；尚无 GitHub 证据表明 routes/tasks 直接把它暴露给终端用户，因此当前更像内部评测/调试开关，但它仍属于必须防守的反模式入口。 ￼

第四，manualReviewNeeded 语义已是 canonical，但 alias 仍并存。README 和 formal-review 文档都说明 manualReviewNeeded 是 canonical，whetherManualReviewNeeded 仅为兼容保留；schema 与 TS 类型里也仍保留 alias。这不是缺功能，而是存在语义漂移风险。 ￼

当前 visibility / manual review / parser contract 的主要问题
当前最需要纠正的不是“把 visibility 再发明一遍”，而是把现有语义从“多个位置都提到”收敛为“一个稳定对象 + 一套稳定展示 + 一套稳定回归”。否则同一事实会同时存在于 parse_result、summary、issue、matrix、report、UI 文案里，后续很容易重新漂移。 ￼

B4. P0-3 最小规则核 / 结构化闭环完成情况
