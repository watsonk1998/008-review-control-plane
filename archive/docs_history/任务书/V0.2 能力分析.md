## 0. 执行摘要

**结论先行：V0.2 相对 V0.0 已经构成一次真实的系统层代际升级，但不是“从零到一发明结构化审查”的那种跨代。**
原因是：V0.0 样本本身已经有 L1/L2/L3、规则命中、冲突矩阵、附件可视域矩阵和 evidence index，所以它并非纯粹的摘要器；V0.2 的真正跨代点在于，把原本较粗的 structured review，升级成了**带有明确可视域语义、issue type、applicability state、unresolved facts、hard defect / visibility gap / evidence gap / enhancement 分层**的 typed review object。也就是说，V0.2 的核心进步首先是**审查对象模型和审查工作流表达**，其次才是问题发现粒度的提升。([GitHub][2])

**V0.2 相对 Gemini 的核心差距，不是文风，而是 L2/L3：**
Gemini 更强的是把事实升级成法定程序与红线判断、把“条文—事实—结论”串成连续论证、再把工程约束转成整改优先级；V0.2 更强的是对 `pdf_text_only`、`parser limited`、`unresolved facts`、`blocked_by_visibility` 这类正式审查必需的“不确定性语义”表达得更诚实、更适合进入控制平面与人工复核流程。换句话说，Gemini 更像**强推理的审查研究员**，V0.2 更像**可回归、可复核、可接工作流的结构化审查底座**。([GitHub][3])

**当前系统最准确的定位**是：
一个已经形成 `structured_review` 主链、并且在 official scope 上开始具备正式审查骨架的 **review control plane**；它已经不是“审查辅助总结器”，但也还远未达到“可替代人工签发的成熟 formal reviewer”。仓库文档明确把 official document type 收敛在 `construction_org` 和 `hazardous_special_scheme`，其他类型仍在 experimental / skeleton 侧，这本身也说明系统当前的成熟边界是**刻意收缩**的。([GitHub][1])

**下一步最该做的不是“更像 Gemini”，而是三件 P0：**
第一，L0 parser / visibility / blocked-state 语义继续做硬；第二，把 L2 的 fact → rule hit → clause applicability → evidence gap 做成真正闭环；第三，把 layered eval 和 reviewer gate 变成真实发布门，而不是只存在于文档。V0.2 设计文档本身也是这样定义目标、边界和 stage gate 的。([GitHub][4]) 

**最大风险**不是“V0.2 不够像专家”，而是：
为了在冷轧厂 / 培花这两例上追分，走向项目名、术语、数字、附件名驱动的 patch；或者把 Gemini 输出偷偷当答案库。那样会让系统在两例上更像“会答题”，在新样本上更像“会乱报”。设计文档和 `review_eval` 说明都明确反对这一路线。([GitHub][5])

## 3. V0.0 vs V0.2：能力升级研究

### 3.1 哪些能力确实升级了

**第一，L0 从“有可视域描述”升级成了“有可视域状态语义”。**
V0.0 已经会写 visibility 与附件矩阵，但 V0.2 把 `parseMode`、`parserLimited`、`parseWarnings`、`manual_review_reason`、`unresolvedFacts` 和 `blocked_by_*` 语义更明确地搬进结果主干。培花 V0.2 对 PDF 的自我诊断尤其关键：它直接承认 `pdf_text_only`、提取页数与总页数差异、表格图示未保留，并把多个关键事实列入 unresolved facts。这是正式审查器应有的“知道自己没看全”的能力，而不是单纯写得更长。([GitHub][3])

**第二，issue 从“发现列表”升级成了“typed findings”。**
V0.0 的 issue 已经有 `severity`、`finding_type`、`manual_review_needed` 和 recommendation，但 V0.2 明显进一步引入了 `issue_kind`、`applicability_state`、`manual_review_reason`、`evidence_missing`，并在顶层按 Hard Defects / Visibility Gaps / Evidence Gaps / Enhancements 重新组织。这使它更接近审查对象，而不只是报告正文。([GitHub][2])

**第三，V0.2 明显削弱了 V0.0 的“辅助总结器”气质。**
V0.0 虽然不是纯摘要，但它更像“把一些发现写成一份结构化报告”；V0.2 则更像“把审查结果建模成可复核对象”，因为它已经把可视域缺口、证据缺口、硬缺陷、建议增强拆分，并与 official scope、pack、eval 逻辑对齐。仓库文档对这种定位也写得很清楚。([GitHub][1])

### 3.2 哪些升级主要是表面升级

这里必须克制：**不是所有“看起来更强”的东西都等于能力跨代。**
V0.0 样本里已经有 L1/L2/L3、hazard matrix、rule-hit matrix、conflict matrix、attachment visibility matrix、section structure、evidence index，所以不能把 V0.2 说成“第一次有结构化审查”。V0.2 有一部分提升，确实属于**报告层包装和工作流语义显化**：分类标题更多、字段更齐、结果更像审查工单。这些很重要，但它们本身不等于问题发现 recall 的同幅度跃升。([GitHub][2])

更具体地说，冷轧厂样本里，V0.0 已经能抓到专项方案/控制措施/资源冲突等主轴；V0.2 的增量更多体现在把同类风险拆成 hard defect、evidence gap、L3 conflict 和 visibility gap，并给出 applicability/blocked 语义。**这是结构化成熟度升级，不完全等于“从无到有新发现”。**([GitHub][2])

### 3.3 V0.2 离“正式结构化审查器”还有多远

我的判断是：**V0.2 已具备“正式结构化审查器的最小骨架”，但还没有达到“成熟 formal reviewer”阶段。**
理由有三点。第一，official scope 仍很窄，仓库明确只把 `construction_org` 与 `hazardous_special_scheme` 作为 official。第二，L2 的条文适用链虽然开始出现，但从样本看仍不够稳定、不够闭环。第三，L3 工程推理虽然开始显式化，但还没有形成足够强的整改编排与 reviewer workflow 语言。([GitHub][1])

### 3.4 V0.0 到 V0.2 变化最本质的部分是什么

最本质的变化不是“报告更像专家”，而是：

> **审查结果从“粗粒度结构化报告”升级成了“带状态语义、证据缺口语义和可视域语义的审查对象”。**

这使 V0.2 能更自然地接入 support-scope、artifact、reviewer decision、versioned eval 与 stage gate，而这正是设计文档希望建立的主链。([GitHub][4])

## 4. V0.2 vs Gemini：差距研究

### 4.1 Gemini 的领先项

**领先项 1：L1 风险升级更果断。**
在冷轧厂样本中，Gemini 明确把危大工程识别、专项方案和法定程序问题上升到“致命盲区 / 一票否决”级别；这不是简单换个说法，而是把事实放进了更强的法律与程序框架。V0.2 也能抓缺口，但风险升级通常更克制。([GitHub][7])

**领先项 2：L2 条文适用链更完整。**
Gemini 更像在回答“为什么这条适用、缺口卡在哪里、因此结论为何升级”；V0.2 虽然已经有 `applicability_state` 和 `evidence_missing`，但从样本看，更多还是“开始具备这个语法”，还没有在所有关键问题上形成足够强的条文—事实—结论闭环。([GitHub][8])

**领先项 3：L3 工程推理更强。**
Gemini 在冷轧厂样本里更积极地推到吊装站位/空间/承载、危险环境管控、人员资质与作业组织；在培花样本里则更充分地展开了基坑、地下水、周边环境与管线等工程上下文。V0.2 在这层已经起步，但更像“给出一些 grounded inference”，还不像“把工程推理编排成整改路线图”。([GitHub][7])

### 4.2 V0.2 的相对优势项

**优势项 1：对可视域边界更诚实。**
培花 V0.2 明确写出 `pdf_text_only`、`parserLimited=yes`、表格/图示未保留、未决事实列表；在提供的 Gemini 样本里，我没有看到与之对等的 visibility boundary 表达。这一点对于正式审查系统很重要，因为“没看到”与“没有”必须分开。([GitHub][3])

**优势项 2：对 unresolved facts / evidence gap 的流程价值更高。**
Gemini 更像一个强审查员；V0.2 更像一个强控制平面对象。V0.2 把“项目名缺失、工程代号缺失、关键计算量缺失、附件标题待确认”等未决事实直接结构化，这对后续 reviewer decision 和复核工作流更有价值。([GitHub][3])

**优势项 3：更适合作为可回归系统而非一次性报告。**
仓库正式把 `structured_review` 定义为带 official scope、artifact、support-scope、versioned eval 的正式子域；Gemini 样本则没有这种控制平面语义。它可以启发问题发现，但不天然等于“可发布系统能力”。([GitHub][1])

### 4.3 Gemini 的领先主要体现在哪一层

我的判断是：**Gemini 的主领先在 L2 / L3，次领先在 L1 风险升级。**
L0 上，Gemini 并不比 V0.2 更强，至少在“显式可视域表达”这一点上不是；L1 上它更善于把问题升级为法律/程序级硬缺陷；L2 上它的适用链更连贯；L3 上它最强。([GitHub][3])

### 4.4 这些领先更像模型优势，还是方法/结构优势

**以方法/结构差距为主，模型长链推理为辅。**
从结果反推，V0.2 漏掉或弱化的很多点，并不是“不会写”，而是没有稳定的事实入口、条文适用链、可视域补偿和 reviewer workflow 表达；这说明 parser / facts / rules / evidence / report/workflow 才是主因。只有在更自由的工程推断和长链整改编排上，模型能力差距才更显著。([GitHub][3]) 

### 4.5 Gemini 内容中，哪些属于什么类型

从样本看，Gemini 内容可分四类：
**硬缺陷识别**：危大工程识别缺口、专项方案/法定程序缺口；
**规范硬约束**：把问题映射到危大管理、临电/消防/专项方案等规范或法规约束；
**工程推断**：吊装空间与承载、停机窗口与资源、基坑与周边环境等；
**建议性增强**：更细的应急、监测、组织编排建议。
这四类混在一起时看起来“很强”，但系统实现上必须拆开，否则就会把建议性增强误当硬缺陷。([GitHub][7])

### 4.6 哪些 Gemini 结论不能直接模仿

不能直接照搬的有三类。
第一，**样本词驱动的结论**：围绕某个环境词、某个吨位、某个停机天数、某个附件名写规则，都会直接把系统引向双样本专用化。第二，**忽略 visibility 的高自信结论**：如果系统自己处于 `pdf_text_only` 或附件不可见状态，却硬模仿 Gemini 的完整论证，会放大误报。第三，**文风模仿**：把结果写得更像深度研究，并不会自动带来 parser、facts、rules、evidence 的提升。([GitHub][3])

## 5. V0.2 结果 vs V0.2 设计文档：兑现度研究

### 5.1 已兑现项

**已兑现 1：`structured_review` 主链与正式边界。**
设计文档把 V0.2 定义为围绕 `structured_review` 主链做深、做稳、做可评测；仓库现状也确实把它写成正式子域，并且把 official scope、support-scope、pack、artifact、reviewer workflow 和 eval 组织到一起。([GitHub][1]) 

**已兑现 2：visibility gap 与缺失分离。**
设计文档特别强调“未解析 ≠ 缺失”；V0.2 培花样本对 `pdf_text_only`、parser limit、unresolved facts 的处理，说明这套语义已经进入产物层，而不只停留在文档里。([GitHub][3]) 

**已兑现 3：硬缺陷与建议增强分离。**
V0.2 样本已经显式按 Hard Defects / Visibility Gaps / Evidence Gaps / Enhancements 组织；这与设计文档要求的“不要把建议增强与硬缺陷混写”是一致的。([GitHub][8]) 

### 5.2 半兑现项

**半兑现 1：L2 规则命中与证据链。**
V0.2 结果已经能看到 `applicability_state`、`evidence_missing` 和部分 manual review reason，说明系统开始把“条文适用链”显式化；但从样本看，这仍不足以和 Gemini 一样形成稳定的“条文—事实—结论”闭环。它更像是**语法已落地、能力还在成长**。([GitHub][8])

**半兑现 2：L3 工程推理。**
设计文档把 L3 定义为工程推理、资源冲突、整改编排与建议增强区分；V0.2 冷轧厂样本里已经能看到停机窗口/资源冲突这类 L3 问题，但整体仍偏“起步态”，还没有形成很强的 remediation buckets 和 reviewer workflow 语言。([GitHub][8]) 

### 5.3 未兑现或较弱项

**较弱 1：L0/L1/L2/L3 虽存在，但成熟度非常不均。**
L0/L1 已经是真实能力层；L2 是半落地；L3 仍明显偏弱。当前输出已经不再只是报告层伪分层，但也还没有做到四层都同等成熟。([GitHub][8])

**较弱 2：layered eval 的“制度化兑现”还不够可见。**
仓库文档已经明确了 test / eval / versioned cases / cross-pack / cross-model 路径，设计文档也把 stage gate 作为主线；但从当前两份真实样本结果来看，更多看到的是“能力表达已出现”，还没有看到足够多的实战样本证明 gate 已经真正压住了 L2/L3 质量波动。([GitHub][9]) 

### 5.4 根因更可能在哪一层

如果说“没有完全兑现”的根因在哪，排序大致是：
**第一是 parser / visibility；第二是 facts extraction 与 clause applicability；第三是 evidence / unresolved-facts 工作流；第四才是 report 表达。**
也就是说，当前未兑现不是因为“报告模板还不够好看”，而是底层事实入口和适用链还没完全长起来。([GitHub][3])

## 6. 分层归因：L0 / L1 / L2 / L3

### L0：可视域 / parser / 附件 / manual review

V0.2 相比 V0.0 的最大进步在 L0：它不只是说“可见/不可见”，而是把 parse mode、parser limit、warnings、unresolved facts 与 blocked 语义往前推；培花 PDF 路径就是最明显例子。相对 Gemini，V0.2 的弱点不是“不会承认看不见”，而是**看不见时能补出的有效事实仍然太少**。主因是 parser 与 visibility contract 仍偏弱，尤其在 PDF、表格、图示、附件引用上。通用改进方向应是统一 visibility taxonomy、把 `referenced_only / attachment_unparsed / missing` 彻底拆开、把 blocked 状态进入 gate；过拟合风险则是为某个附件名、某个章节名、某种 PDF 样式写特判。([GitHub][3])

### L1：硬规则 / 红线 / 法定程序 / 结构完整性

V0.2 相比 V0.0 已经更接近正式审查：hard defect 更显式，official scope 更清楚，风险类别与 issue kind 也更清楚。相对 Gemini，V0.2 的差距在于**不够善于把“重要问题”升级为“法定程序/红线问题”**。主因更像 rule pack maturity、severity mapping、promotion policy，而不是报告长度。通用改进方向应该是加固 official 两类文档的红线规则与 promotion criteria，而不是为某个样本里的危险环境词写专项规则。([GitHub][8])

### L2：条文适用 / policy evidence / unresolved facts

V0.2 的 L2 比 V0.0 有实质提升，因为 `applicability_state`、`evidence_missing`、`manual_review_reason` 已经出现；但相对 Gemini，它仍明显偏弱，尤其在“为什么适用、卡在哪里、当前不能下结论是因为 visibility 还是 missing fact”这条链上。主因是 facts taxonomy、rule-to-clause applicability、evidence gap typing 仍不完整。通用改进方向是把 fact → rule hit → clause → applicability → evidence gap → reviewer action 串成统一对象；过拟合风险则是把 Gemini 的个案结论直接搬成 rules，或者把某个样本中的关键词当作 clause 入口。([GitHub][8])

### L3：工程推理 / 资源冲突 / 可操作整改

V0.2 相比 V0.0 已经开始把工程冲突写成独立问题，而不是只停留在建议段；冷轧厂样本中的停机窗口/资源冲突就是信号。相对 Gemini，V0.2 的差距最明显：Gemini 更善于把工程事实组织成“为什么这会导致现场不可执行、应该先整改什么”的论证。主因是 grounded facts 还不够、L3 remediation buckets 还不成熟、reviewer workflow 还未充分吸收这类结论。通用改进方向应是 deterministic remediation buckets 与 reviewer confirmation 回流；过拟合风险则是直接为“某个天数、某个吨位、某个人数”写规则。([GitHub][8])

## 7. 反过拟合审计

这是本报告最关键的部分。两份样本都很有价值，但**样本层结论不等于系统层结论**。真正应该保留的是“通用机制缺口”，而不是“这两份文档里恰好出现了什么”。设计文档也把这一点列成硬约束。

| 建议方向                                               | 是否有过拟合风险 | 风险类型                | 是否推荐 | 推荐级别               | 理由                                       |
| -------------------------------------------------- | -------- | ------------------- | ---- | ------------------ | ---------------------------------------- |
| 统一 L0 visibility taxonomy 与 blocked-by-* 语义        | 低        | 实现复杂度               | 是    | P0                 | 通用于 DOCX/PDF/附件，不依赖项目语义                  |
| 把“附件未解析 / 引用但不可见 / 真缺失”彻底拆分                        | 低        | schema 漂移           | 是    | P0                 | 直接降低误判，且高度可泛化                            |
| 扩通用 hazard/resource/control facts taxonomy         | 中        | 场景过扩张               | 是    | P1                 | 只在“风险类别”层扩，不在样本词层扩                       |
| 把 fact→rule→clause→applicability→evidence gap 串成闭环 | 低        | 对象模型变复杂             | 是    | P0                 | 这是系统能力增强，不是 case patch                   |
| 把 unresolved facts 更强地接入 reviewer workflow         | 低        | 流程复杂度               | 是    | P1                 | 提升正式审查可信度，且不依赖样本细节                       |
| 构建 deterministic remediation buckets               | 中        | L3 误触发              | 是    | P1（先 experimental） | 价值高，但应先让 reviewer 校准                     |
| 为“煤气区域”单独写规则                                       | 高        | case token overfit  | 否    | 否决                 | 应抽象成“危险气体/危险环境作业 taxonomy”               |
| 为“50t 吊车 / 7 天停机 / 37 人”写特判                        | 高        | case number overfit | 否    | 否决                 | 应抽象成“吊装能力/工期资源冲突”机制                      |
| 为某个附件名或某个章节标题写例外逻辑                                 | 高        | 文件名/模板 overfit      | 否    | 否决                 | 会把系统锁死在样本模板上                             |
| 直接把 Gemini 结果并入 gold labels                        | 高        | label leakage       | 否    | 否决                 | 仓库和设计文档都明确禁止                             |
| 以“更像 Gemini 的文风”为目标重写报告                            | 高        | style overfit       | 否    | 否决                 | 不能提升 parser/facts/rules/evidence         |
| 立即扩 official scope 到更多 doc types                   | 中高       | 范围膨胀                | 暂不推荐 | P2                 | 现有样本不足以支撑升格，应先走 experimental/diagnostics |
| 增加更多样本桶：低问题密度 / 高 visibility gap / 非当前 doc type    | 低        | 数据治理成本              | 是    | P1                 | 这是对抗双样本偏见的最有效手段之一                        |

这张表的核心含义是：**“推荐”与“看起来立刻见效”不是一回事。**
真正值得做的是机制增强、对象模型增强、评测与工作流增强；真正该否决的是项目词、数字、附件名、模板章节驱动的 patch。([GitHub][5]) 

不推荐项的原因都一样：它们会把系统从“通用审查能力”拖回“针对冷轧厂 / 培花答题”。推荐项也不是无条件直接进 official：像 L3 remediation buckets、扩展场景 facts taxonomy 这类东西，更适合先进入 **experimental / diagnostics / internal-reviewed cases**，让 reviewer 先校准噪声，再考虑升格。([GitHub][4])

## 8. 下一步优化优先级

### 8.1 P0：必须优先做

| 优先级  | 方向                                             | 主要层 | 工作类型                            | 过拟合风险 | 建议落点                   | 说明                      |
| ---- | ---------------------------------------------- | --- | ------------------------------- | ----- | ---------------------- | ----------------------- |
| P0-1 | L0 parser / visibility / blocked-state 加固      | L0  | parser + report + eval          | 低     | official + diagnostics | 先解决“没看全时如何不乱判”          |
| P0-2 | fact→rule→clause→applicability→evidence gap 闭环 | L2  | facts + rules + evidence + eval | 低     | official               | 这是 V0.2 到 Gemini 真差距的主轴 |
| P0-3 | layered eval + reviewer gate 制度化               | 横切  | eval + workflow                 | 低     | official               | 把 L0/L1/L2/L3 质量真正变成发布门 |

这三项是**系统机制增强**，不是样本补丁，也最符合设计文档对 V0.2 的原始定位。([GitHub][9]) 

### 8.2 P1：重要但可次阶段做

| 优先级  | 方向                                                        | 主要层      | 工作类型              | 过拟合风险 | 建议落点                             | 说明            |
| ---- | --------------------------------------------------------- | -------- | ----------------- | ----- | -------------------------------- | ------------- |
| P1-1 | 扩通用高风险场景 facts taxonomy 与 pack promotion discipline       | L1/L2/L3 | facts + rules     | 中     | experimental 先行                  | 只做类别，不做样本词    |
| P1-2 | reviewer cockpit + unresolved facts / remediation buckets | L3       | workflow + report | 中     | experimental / internal-reviewed | 让 L3 真正进入人工复核 |
| P1-3 | 新增样本桶以对冲双样本偏见                                             | 横切       | eval + data       | 低     | diagnostics                      | 比追两例高分更重要     |

### 8.3 P2：不急，甚至应延后

| 优先级  | 方向                   | 主要层 | 工作类型             | 过拟合风险 | 建议落点  | 说明               |
| ---- | -------------------- | --- | ---------------- | ----- | ----- | ---------------- |
| P2-1 | 扩 official doc types | 横切  | scope governance | 中高    | 延后    | 先把现有 official 做硬 |
| P2-2 | 多模态/OCR/图纸平台化        | L0  | platform         | 中     | 延后    | 价值高，但不是 V0.2 主线  |
| P2-3 | Gemini 风格长报告或“专家腔”优化 | 表达层 | report style     | 高     | 不建议优先 | 极易掩盖真实能力缺口       |

### 8.4 后续优化最值得投入的 5 个方向

按价值排序，我给出的五个方向是：

1. **L0 可视域与 blocked-state 统一化**
2. **L2 适用链与 evidence gap 闭环**
3. **layered eval / stage gate 制度化**
4. **通用高风险场景 facts taxonomy + pack promotion discipline**
5. **reviewer cockpit + remediation buckets（先 experimental）**

其中前 3 个是 P0，后 2 个是 P1。前 3 个都是明确的通用机制增强；第 4、5 个如果做得太激进，会有中等过拟合风险，所以应先放在 experimental / diagnostics / internal-reviewed，不宜直接升 official。([GitHub][4])

## 9. 最终结论

**1. V0.2 相对 V0.0 是否已经构成真正代际升级？**
是，但要精确定义：它构成了**系统骨架与审查对象模型层面的代际升级**，不构成“已经完成 formal reviewer 深度跃迁”的全面跨代。V0.0 已经不是纯总结器；V0.2 的跨代点在于 typed review object、visibility semantics、issue kind、applicability state、unresolved facts 和 workflow/eval 融合。([GitHub][2])

**2. 当前 V0.2 在行业审查系统里的最准确定位是什么？**
它最准确的定位是：**一个已具备正式结构化审查最小骨架、适合做控制平面与复核闭环底座的 structured reviewer spine**；不是摘要器，也不是成熟到可替代人工签发的 formal reviewer。official scope 很窄这一点，恰恰说明仓库对自身成熟边界是清醒的。([GitHub][1])

**3. 与 Gemini 的差距，本质上主要是什么？**
本质上是**L2/L3 的结构性差距**：fact coverage、clause applicability、evidence gap typing、工程推理与整改编排仍弱于 Gemini；不是“写得不够像专家”。Gemini 更像强推理审查员，V0.2 更像诚实的控制平面对象。([GitHub][7])

**4. 下一步优化的第一优先级是什么？**
第一优先级不是继续扩问题点，而是**先让系统更可信**：把 L0 visibility / parser / blocked-state 做硬，再把 L2 fact→rule→clause→evidence gap 做闭环，并用 layered eval 把它们变成真实 gate。([GitHub][9]) 

**5. 最不能做错的一件事是什么？**
最不能做错的是：**把系统优化成“冷轧厂 + 培花 + Gemini 文风”的专用答题器。**
这会在短期上看起来很强，长期上破坏泛化、破坏可信度、破坏评测。([GitHub][5]) 

**6. 为什么“防过拟合”在当前阶段比“追求两例表现更强”更重要？**
因为当前真正缺的不是 case trick，而是机制稳定性。两份样本足够暴露问题，却远远不足以证明泛化；如果此时按样本词写规则，只会把系统从“正在形成的正式结构化审查器”拉回“高分样本机”。相反，只有坚持 visibility truth、issue truth、policy applicability、reviewer workflow 和 versioned eval 的统一，V0.3 才可能长成真正可扩展的审查系统。([GitHub][4]) 

这份报告的最终判断可以压缩成一句话：

> **V0.2 已经不是 V0.0 的包装版，而是一次“结构化审查对象模型 + 可视域语义 + 复核工作流”层面的真实升级；但它与 Gemini 的核心差距仍在 L2/L3 的通用机制，而不是文风。下一步最该做的是机制加固与评测加固，而不是双样本追分。**
