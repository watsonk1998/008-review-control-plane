# 《V0.2→V0.3 差距裁决与反过拟合约束清单》

## 0. 文档定位

本文是 V0.3 启动前的 **gap adjudication + anti-overfit gate document**。本文只做一件事：在既定仓库事实与 V0.3 已拍板边界内，裁决哪些差距进入本轮，哪些被降级、延后、隔离或否决。

本文**不是**：

1. 不是 V0.3 实施任务书；
2. 不是 PR 分解计划；
3. 不是产品愿景文档；
4. 不是对双样本结果或 Gemini 报告的改写版总结。

本文默认并反复坚持以下前提：

- `structured_review` 是 `008-review-control-plane` 内的一条正式结构化审查主链，不等于整个项目本体；
- official scope 当前仍仅限 `construction_org` 与 `hazardous_special_scheme`；
- 当前并不存在可直接依赖的 `v0.2.0-internal-reviewed` 或 `v1.0.0-expert-golden` 实际材料；
- anti-overfit 不是提醒项，而是本轮治理主轴之一。

## 2. 差距裁决总声明

V0.2→V0.3 本轮正式确认的主线差距，不是“报告还不够像 Gemini”，也不是“official scope 还不够大”，而是以下五类：

1. **L0 parser-limited PDF 的 gate 仍未完全前置**；
2. **L0 attachment state taxonomy 与 manual-review 语义仍需继续做硬**；
3. **L2 的 fact → rule hit → applicability → evidence gap → unresolvedFacts 闭环仍未成熟**；
4. **truth layering / reviewer gate / promotion governance 仍缺少已复核材料承接层**；
5. **official 两类文档内的 rule/evidence 核仍应继续加固，但不应借机扩 doc type。**

本轮同时明确降级三类方向：

- ready scenario packs 的增强与 promotion governance，默认只进入 `experimental / diagnostics / internal-reviewed preparation`；
- L3 deterministic remediation bucket 与 workflow language，只能作为诊断增强或受限 P1，不能覆盖本轮主线；
- placeholder 或仅在 seed 中出现的扩域 pack，只保留观察，不进入 official 主线。

本轮明确不纳入以下方向：

- 扩 official scope 到 `construction_scheme / supervision_plan / review_support_material`；
- OCR / 多模态 / 图纸平台化 / 多文档批处理；
- 以“更像 Gemini”为目标的报告 prose、篇幅、语气或叙事丰满度；
- 围绕项目名、固定数字、章节标题、附件名、双样本特征写 patch；
- 把 Gemini seed 当作 gold、裁决 authority、答案库或 prompt 风格模板。

之所以如此裁决，原因只有三条：

- **仓库事实约束**：official scope、pack registry、truth 语义、PDF 边界、manual-review 语义都已在仓库主干写定；
- **结构化证据约束**：样本 A/B 与 eval 快照显示最实的缺口在 L0 与 L2，而不是 prose；
- **治理现实约束**：当前没有 internal-reviewed / expert-golden 承接层，任何越界扩张都只会放大 seed 污染与双样本过拟合。

本轮总判词如下：

> **V0.3 应做“边界内加固”，不做“边界外扩张”；应优先把 L0 可视域诚实度与 L2 条文适用闭环做硬，不应把双样本追分或 Gemini 模仿伪装成正式进展。**

## 4. 进入 V0.3 主线的已确认差距

### 4.1 P0 confirmed gaps

#### 4.1.1 parser-limited PDF 的 L0 gate 未完全前置

**为什么确认**

- 仓库事实已经把 `visibility` 定义为 canonical L0 对象，并要求 PDF 保持 `pdf_text_only + parserLimited=True` 的保守语义；
- 样本 B 的 `fixtures/research_inputs/sample-b-puhua-rainwater/structured-review-l0-visibility.json` 明确显示：`parseMode=pdf_text_only`、`parserLimited=true`、`visibility.manualReviewNeeded=false`；
- 但同样本的 `structured-review-result.json` 又显示：`summary.manualReviewNeeded=true`，且多个 issue 仍以 `partial` / `blocked_by_missing_fact` / `manualReviewReason` 方式承接人工复核语义；
- `fixtures/research_inputs/latest-eval-summary.md` 还显示总体 `attachment_visibility_accuracy=0.5833`，说明 L0 仍是治理软点。

**为什么进入本轮**

这是官方主链的可信度问题，不是解释层问题。L0 gate 不彻底前置，后续 L1/L2/L3 都会继承 parser-limited 状态下的歧义。

**为什么不是双样本 patch**

该差距指向的是所有 parser-limited PDF 路径共通的 gate 设计问题，而不是“培花样本专有问题”。任何围绕项目名、页数、工程域做补丁都无效；必须做的是 canonical visibility 与 task-level/manual-review 语义的一致化。

**对应能力层与系统层**

- 能力层：L0
- 系统层：parser

#### 4.1.2 attachment state taxonomy 与 manual-review 语义需继续加固

**为什么确认**

- 仓库事实明确要求区分 `attachment_unparsed / referenced_only / missing / unknown`，并明示“未解析 ≠ 缺失”；
- 样本 A 的 `fixtures/research_inputs/sample-a-cold-rolling/structured-review-l0-visibility.json` 直接给出：`attachment-2` 为 `attachment_unparsed`，`missing=0`，`manualReviewReason=title_detected_without_attachment_body`；
- 样本 B 的附件矩阵为空，但未被错误写成“附件缺失”；
- 这说明 taxonomy 已有基础，但其 reviewer gate 与 task-level 语言仍需继续收口。

**为什么进入本轮**

L0 taxonomy 是 official 主线的可信度底板。只要 taxonomy 和 manual-review 语义不够硬，系统就会在 evidence 与 report 层回退成强结论冲动。

**为什么不是双样本 patch**

这里应保留的是 `attachment state taxonomy` 机制，而不是样本 A 的附件名、标题、编号。任何围绕“附件2：施工总平面布置图”写特判都属于被否决的样本补丁。

**对应能力层与系统层**

- 能力层：L0
- 系统层：evidence

#### 4.1.3 fact → rule hit → applicability → evidence gap → unresolvedFacts 闭环未成熟

**为什么确认**

- 仓库 contract 已把 `issues / applicabilityState / evidenceMissing / unresolvedFacts` 固化进结果对象；
- 样本 B 的 `structured-review-result.json` 显示 `unresolvedFactCount=6`，且 issue 出现 `blocked_by_missing_fact` 与 `partial`；
- 样本 A/B 的 `structured-review-rule-hits.json` 也显示同一规则族会落在 `hit / pass / manual_review_needed` 等不同状态，但当前还未把这些状态完全提升为稳定的 clause applicability / evidence gap 闭环；
- 本地 `artifacts/research-pack/eval/eval-review.json` 的 `versionedDiagnostics` 还显示 `facts_accuracy=0.85`、`rule_hit_accuracy=0.84`，说明 diagnostics 层仍未闭合。

**为什么进入本轮**

V0.3 已明确主攻 L2。本轮不是要新增更长的条文解释，而是要把“为什么命中、为什么部分适用、为什么被 visibility 或 missing fact 卡住”做成更稳定的治理语言。

**为什么不是双样本 patch**

这条差距本质上是链路治理问题：facts、rule hits、applicability、evidence gap、unresolvedFacts 之间的派生关系还不够硬。它不是由“某个数字没抽到”或“某个章节标题没识别”单独造成的，因此不能用样本词表修补。

**对应能力层与系统层**

- 能力层：L2
- 系统层：evidence

#### 4.1.4 truth layering / reviewer gate / promotion governance 仍缺“已复核材料承接层”

**为什么确认**

- `fixtures/review_eval/README.md` 已明确定义 `v0.2.0-internal-reviewed` 与 `v1.0.0-expert-golden`；
- 但当前 `fixtures/review_eval/` 实际目录中并不存在上述版本目录；
- 仓库虽已有 reviewer decision API 与 stage gate 机制，但缺少“从 seed / bootstrap 走向 reviewed truth”的实体承接层；
- 在这一现实下，任何把 seed 直接冒充 reviewed truth 的做法都没有治理载体。

**为什么进入本轮**

这是 V0.3 能否做“正式裁决”而不是“继续研究”的前置条件。没有 reviewed truth 承接层，official mainline 很容易被 seed 污染。

**为什么不是双样本 patch**

该差距是数据与治理层结构缺失，与样本具体内容无关。即使换样本，这个承接层缺口仍然存在。

**对应能力层与系统层**

- 能力层：CrossCutting
- 系统层：workflow

### 4.2 P1 confirmed gaps

#### 4.2.1 official 两类文档内的 rule/evidence 核应继续做硬，但不扩 doc type

**为什么确认**

- 仓库事实已经把 official 主战场限定为 `construction_org` 与 `hazardous_special_scheme`；
- 样本 A/B 已经证明当前系统在 official `construction_org` 内具备真实 rule/evidence 能力，但也暴露出 L0 与 L2 的薄弱点；
- `fixtures/research_inputs/latest-eval-summary.md` 显示 narrow official stage gate 通过，但 overall / diagnostics 仍有软点；
- 因此，本轮最合理的 P1 不是扩 doc type，而是继续加固 official 两类文档内的 rule/evidence 核。

**为什么进入本轮**

这条工作与已拍板边界一致：先把现有 official 两类做硬，再谈 promotion governance。

**为什么不是双样本 patch**

这里要求的是 official 主线下的规则核、证据核、pack discipline 做硬，而不是围绕 A/B 写专用 extractor 或规则分支。

**对应能力层与系统层**

- 能力层：L1 / L2
- 系统层：rules

## 5. 被降级、延后或仅保留观察的差距

### 5.1 ready scenario packs 的增强与 promotion 治理 —— Partially Confirmed

该方向确实存在真实需求，但本轮只能降级进入 `experimental / internal-reviewed preparation`，不能进入 official mainline。

**原因如下：**

- README 与 `docs/formal-review.md` 已明示 `ready pack ≠ official support`；
- `docs/testing.md` 已把 experimental versioned cases 放入 diagnostics，而非 blocking official gate；
- 本地 `artifacts/research-pack/eval/eval-review.json` 的 `versionedDiagnostics` 显示 diagnostics 层仍存在 `facts_accuracy=0.85`、`rule_hit_accuracy=0.84` 之类的不足；
- 因此，ready scenario packs 可以继续增强，但增强结果默认只能进入 `experimental / diagnostics / internal-reviewed preparation`，不能自动升 official。

本条之所以是 **Partially Confirmed**，是因为它的方向成立，但其升格条件尚未满足 reviewer-backed truth、更多 versioned cases、promotion criteria 全量满足等前置条件。

### 5.2 L3 deterministic remediation bucket / workflow language —— Partially Confirmed

该方向有真实价值，但本轮只能保持 diagnostics 或受限 P1，不应覆盖主线。

**原因如下：**

- 三角研究已经说明 L3 当前仍是半落地；
- 样本 A 的 `conflict-matrix.json` 显示 grounded conflict inference 已出现；
- 样本 B 则因 `shutdownWindowDays`、`laborTotal` 等关键事实缺失，导致 L3 明显收缩；
- `docs/testing.md` 与 `docs/formal-review.md` 也明确：L3 当前仍是 `diagnosticOnly=true`。

本条之所以是 **Partially Confirmed**，是因为 L3 的增强方向存在，但在缺乏稳定 facts 与 reviewed truth 的前提下，不适合升格为本轮 official 主线。

### 5.3 placeholder 或 seed 才出现的扩域 pack —— Deferred

以下方向本轮统一 **Deferred**：

- `special_equipment.base`
- `working_at_height.base`
- `deep_excavation.base`
- `dewatering.base`
- `pipe_jacking.base`
- `electromech_installation.base`

**原因如下：**

- 当前 README / `docs/formal-review.md` 中，`special_equipment.base` 与 `working_at_height.base` 仍是 placeholder；
- `deep_excavation.base / dewatering.base / pipe_jacking.base / electromech_installation.base` 并不在当前 repo-truth pack registry 中；
- 这些扩域 pack 只在 Gemini-seed `case_metadata.json` 或 `expected_rule_hits.json` 中作为 seed/candidate 出现；
- 既无 official repo-truth 支撑，也无 internal-reviewed 承接层，贸然纳入只会把 seed signal 误写成 supported scope。

因此，本条只能进入 `internal-reviewed preparation / out of scope`，仅保留观察，不进入本轮主线。

## 6. 明确否决项与禁止事项

以下条目在 V0.3 范围内统一视为 **Rejected**。这不是建议，而是政策性禁止。

### 6.1 明确否决的优化方向

1. **禁止扩 official scope 到 `construction_scheme / supervision_plan / review_support_material`**。
   - 理由：与 README / `docs/formal-review.md` 现行 official scope 冲突。

2. **禁止以 OCR / 多模态 / 图纸平台化 / 多文档批处理作为本轮能力入口**。
   - 理由：与当前 PDF 边界、单文件 contract 与《V0.3 边界声明》冲突。

3. **禁止把“更像 Gemini”的 prose、篇幅、语气或 narrative completeness 当成本轮目标**。
   - 理由：这会把 report-only 目标伪装成 formal-review 进展。

### 6.2 明确禁止的 patch 类型

以下 patch 类型一律禁止：

- 围绕项目名写 patch；
- 围绕固定数字写特判；
- 围绕章节标题写样本模板逻辑；
- 围绕附件名写例外判断；
- 为双样本单独写 `rules / extractors / report` 分支；
- 围绕 V0.0 / V0.2 / Gemini 的单样本文案对照做 prompt 贴合。

判定原则如下：

> **任何不能被描述为“taxonomy / contract / rule/evidence/governance 机制增强”的修改，默认视为高风险样本补丁。**

### 6.3 明确禁止的 Gemini 使用方式

以下用法统一禁止：

- 把 Gemini seed 当作 gold truth；
- 把 Gemini 结果当作最终裁决 authority；
- 把 Gemini 结果蒸馏成规则答案库；
- 把 Gemini wording 当作 prompt hidden target；
- 把 Gemini narrative 当作 report 风格模板；
- 用 Gemini 输出覆盖 canonical visibility、issue truth 或 evidence truth。

### 6.4 明确禁止的样本特征驱动路线

以下路线一律不得进入 V0.3：

- 因为冷轧厂样本里出现“附件2”就为“附件2”写规则；
- 因为培花样本里出现 PDF 表格或某些工程域名词就假定 future pack 已 ready；
- 因为 Gemini-seed 列出了某些 expected packs，就把这些 pack 视为当前已支持；
- 因为某次本地 eval 快照通过，就假定边界可以放宽。

## 7. Gemini seed 的临时裁决地位

### 7.1 当前允许 Gemini 扮演的角色

当前阶段，Gemini seed 只允许扮演以下角色：

| 允许角色 | 允许方式 |
| --- | --- |
| seed baseline | 作为对照基线，帮助识别当前系统常漏的 candidate issue 类型 |
| bootstrap substitute | 在没有 reviewed truth 时，临时支撑最小 versioned seed case 的初始化 |
| candidate issue inventory | 作为候选问题盘点来源，进入后续 reviewer / adjudication 候选池 |
| gap probe | 用于探测当前 repo-truth rule/policy asset 尚未覆盖的域 |

### 7.2 当前禁止 Gemini 扮演的角色

Gemini seed **不允许**扮演以下角色：

| 禁止角色 | 禁止原因 |
| --- | --- |
| gold truth | 与 `fixtures/review_eval/README.md` 非协商原则冲突 |
| final adjudication authority | 当前不存在 reviewer / expert confirmed 承接层 |
| answer bank | 会把 seed signal 污染为规则答案库 |
| prompt hidden target | 会把 formal-review 优化退化成 prompt 模仿 |
| style target | 会把 prose chase 伪装成能力升级 |

### 7.3 在没有人工裁决材料时，Gemini 如何“临时顶一顶”

在缺乏完整人工裁决材料的前提下，Gemini 只能以 **受限、可追溯、不可越级** 的方式临时发挥作用：

1. 允许它帮助建立最小 seed case；
2. 允许它提示可能的 candidate issue inventory；
3. 允许它暴露 current pack coverage 之外的候选域；
4. 必须显式保留 `seeded_unreviewed / bootstrap / not_final` 等 provenance；
5. 必须与 repo-truth、structured evidence、reviewer gate 分层治理，不得越级。

### 7.4 哪些事情绝不能由 Gemini seed 直接决定

以下事项不得由 Gemini seed 直接决定：

- official mainline admission；
- issue truth 的最终确认；
- visibility truth 的最终确认；
- pack promotion；
- reviewer gate 通过与否；
- seed 向 internal-reviewed / expert-golden 的升级。

### 7.5 从 seed 到 internal-reviewed / expert-golden 还缺什么

当前要从 seed 走向更高 truth 层，还至少缺以下材料：

- 成体系的逐 issue adjudication notes；
- reviewer confirmed truth set；
- internal-reviewed 版本目录与样本资产；
- 专家确认后的 gold labels；
- 与这些 labels 对应的 provenance、裁决理由与 reviewer gate 记录。

在这些材料缺失前，Gemini seed 只能是 seed，不能被包装成“临时 gold”。

## 8. 反过拟合约束清单

anti-overfit 在本清单中是主干制度，不是附录提醒。以下约束分为三组执行。

### 8.1 结构性约束

| 约束内容 | 约束目的 | 违反风险 |
| --- | --- | --- |
| 仓库事实优先于边界声明，边界声明优先于研究观察 | 防止研究底稿倒置成为治理真相源 | 范围失控、与主干事实冲突 |
| official scope 仅限 `construction_org / hazardous_special_scheme` | 防止 scope creep 伪装成“自然延伸” | official gate 被 experimental/seed 污染 |
| 必须区分 issue truth / visibility truth / evidence truth / suggestion enhancement / seed signal / reviewed truth / gold truth | 防止多层语义混写 | truth 污染、误判 severity、误导 reviewer |
| 优先做硬机制，不优先追双样本分数或 prose 观感 | 防止“追分”挤掉正式治理 | 形成两样本高分解释器而非 formal reviewer |

### 8.2 数据与评测约束

| 约束内容 | 约束目的 | 违反风险 |
| --- | --- | --- |
| 差距裁决优先依据 JSON / matrices，不优先依据 Markdown 报告 | 保持结构化证据优先级 | 把 prose 偏好误判成能力差距 |
| `latest-eval-summary.md` 只作阶段性快照，不作长期真相 | 防止单次本地评测被过度解读 | 因快照波动错误放宽边界 |
| official gate、experimental diagnostics、seed case 必须分层治理 | 防止实验结果偷渡主线 | promotion governance 失效 |
| 未出现 internal-reviewed / expert-golden 之前，不得假装有人类 gold | 保持 truth 语义诚实 | reviewer gate 被 seed 替代 |
| 不能因为双样本结果“更像 Gemini”就判定差距已关闭 | 防止 narrative-driven optimism | 忽视泛化不足与真实治理缺口 |

### 8.3 实现与规则约束

| 约束内容 | 约束目的 | 违反风险 |
| --- | --- | --- |
| 禁止在 `rules/*`、`extractors/*`、`report/*` 中按项目名 / 文件名 / 样本特征分支 | 防止 case-specific patch | 双样本过拟合、后续样本失效 |
| 禁止把 `attachment_unparsed / referenced_only / unknown` 直接写成 `missing` | 保护 canonical visibility truth | 误报缺失、错误升级 hard defect |
| 禁止把 Gemini 结果蒸馏成规则答案库或 prompt 模板 | 防止 seed 污染主链 | 规则失真、explanation overfit |
| 禁止 placeholder pack 或 seed-only pack 越级冒充 current support | 保护 pack registry 与 support-scope 真相 | official/experimental 边界塌陷 |
| 禁止为双样本单独添加规则、extractor 或报告分支 | 保持机制级增强导向 | 系统退化为样本特征解释器 |

## 9. 进入 V0.3 的准入条件与退出条件

### 9.1 允许进入 official mainline 的准入条件

一个差距或增强方向，只有同时满足以下条件，才允许进入 V0.3 official mainline：

1. 与当前 repo-truth 一致，不越 official scope；
2. 至少有 1 个仓库事实锚点 + 1 个 structured evidence 锚点；
3. 能被描述为 taxonomy / contract / parser / facts / rules / evidence / workflow / eval 机制增强；
4. 过拟合风险为 `low`；
5. 不依赖 OCR / 多模态 / 多文档批处理；
6. 不把 Gemini seed 当 authority；
7. 能自然纳入 versioned cases + layeredMetrics + reviewer gate 的现有治理框架。

### 9.2 只能待在 experimental / diagnostics / internal-reviewed preparation 的条件

以下方向即使存在，也只能留在非主线层级：

- ready scenario pack 的增强，但 promotion 条件未齐；
- L3 workflow language，需要更多 reviewer / internal-reviewed 支撑；
- seed-only 或 placeholder 域的 future pack 候选；
- 只能由双样本现象支撑、缺乏更广泛治理承接的方向。

### 9.3 应立即退出本轮范围的条件

以下任一条件出现，即应立即退出 V0.3 本轮范围：

- 需要扩 official scope；
- 需要 OCR / 多模态 / 图纸平台化 / 多文档批处理；
- 需要把 Gemini 当 authority、gold、答案库或风格目标；
- 依赖项目名、固定数字、章节标题、附件名等样本补丁；
- 不能在现有 reviewer / internal-reviewed 治理框架内落位；
- 会破坏 canonical visibility truth 与“未解析 ≠ 缺失”的边界。

### 9.4 必须等待 internal-reviewed / reviewer gate / 更多样本支持的条件

以下方向在缺乏 reviewed 承接层前，不得进入 official mainline：

- 任何 pack promotion；
- 任何 seed → reviewed → gold 的 truth 升级；
- 任何超出当前 registry 的 future domain pack；
- 任何需要 L3 作为正式阻断依据的工作流增强。

## 10. 最终裁决结论

V0.2→V0.3 这一轮真正要解决的，不是“报告再写得像 Gemini 一点”，也不是“再把几个 documentType 加进 official”；真正要解决的是：

1. **把 L0 可视域诚实度、parser-limited 状态与 manual-review 触发做硬**；
2. **把 L2 的 fact → rule hit → applicability → evidence gap → unresolvedFacts 闭环做硬**；
3. **把 truth layering、reviewer gate 与 promotion governance 补上已复核承接层**；
4. **在不扩 scope 的前提下，把 official 两类文档的 rule/evidence 核继续做硬。**

本轮明确不解决的，是：

- official scope 扩张；
- OCR / 多模态 / 图纸平台化 / 多文档批处理；
- 任何以 Gemini prose 为目标的 narrative chase；
- 任何围绕双样本表面特征的补丁路线。

本轮最应优先保护的，是：

- canonical visibility truth；
- issue truth / visibility truth / evidence truth 的分层治理；
- official / experimental / diagnostics / internal-reviewed preparation / out of scope 的边界纪律；
- “在更多样本上更可信”而非“在两份样本上更像 Gemini”的方向选择。

本轮最不能犯的错误，是：

- 把 Gemini seed 伪装成 gold；
- 把 ready pack 伪装成 official support；
- 把双样本现象直接翻译成通用规则；
- 把 prose 追逐伪装成 formal-review 进展；
- 在缺乏 internal-reviewed 承接层的前提下提前宣布“差距已关闭”。

因此，本清单的最终裁决如下：

> **V0.3 必须先完成“差距裁决 + anti-overfit gate”，再进入规模化实施。否则，系统最容易得到的不是更可信的 formal review，而是一个更会在两份样本上得分、更会模仿 Gemini 叙事、却缺乏真实治理硬度的解释器。**
